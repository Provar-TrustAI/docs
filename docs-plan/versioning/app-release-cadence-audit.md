# Trust AI App — Release Cadence Audit

**Spike:** DEV-1764 (docs.provar.dev versioning strategy)
**Date:** 2026-05-20
**Scope:** How does `trust-ai-app` version today, what artifacts does each release produce, and what is machine-readable enough to flow into `docs.provar.dev`?

---

## TL;DR

Trust AI **adopted CalVer on 2026-04-20** (commit `08135576`, tag `v2026.04.20`, PR #214). The legacy SemVer tags (`v0.1.0` through `v0.1.4`) are frozen relics of the V0 Railway-bootstrap era; every release after the rename publishes as `vYYYY.MM.DD[.N]`. There is one release artifact pipeline (`.github/workflows/publish-ghcr.yml`) and one canonical version-strategy doc (`docs/release/image-tags.md`). The deployable unit is **the four-service bundle** (`api`, `agent-gateway`, `web`, `worker`) sharing one tag — not per-service versions.

**Machine-readable artifacts that already exist at HEAD and can feed docs:**

1. `services/api/openapi.json` (659 KB, OpenAPI 3.1.0, `info.version: "0.1.0"`) — full `/v1` API surface generated by FastAPI.
2. `docs/RELEASE_NOTES_v0.md` + `docs/RELEASE_NOTES_paddington_v1.md` — long-form narrative release notes, hand-written, milestone-grouped, ticket-cited.
3. Git tags themselves — `git tag --sort=-creatordate` is a parsable release feed.
4. `package.json` (`version: "0.1.0"`) and `services/api/pyproject.toml` (`version = "0.1.0"`) — both still pinned at the pre-CalVer value. Not actively maintained as truth; the **git tag is the truth** post-CalVer.
5. The GHCR registry itself — `gh api /orgs/Provar-TrustAI/packages/container/trustai-app%2Fapi/versions` lists every released image tag.

**Not machine-readable today:** there is no `CHANGELOG.md` at the repo root, no per-release JSON manifest, no auto-generated TypeScript-types tarball published as a release asset, no machine-readable concept/schema dump.

## Versioning scheme

### Pre-2026-04-20: SemVer (5 tags, frozen)

| Tag | Date | Subject |
|---|---|---|
| `v0.1.0` | 2026-04 | docs: Tower improvement backlog from V0 sprint lessons (#193) |
| `v0.1.1` | 2026-04 | feat: Railway deploy pipeline — nginx proxy, standup/teardown scripts (#209) |
| `v0.1.2` | 2026-04 | fix: Railway deploy fixes — IPv6 networking, migrations, auth text (#210) |
| `v0.1.3` | 2026-04 | ci: publish GHCR images on tags only, fix lowercase image name (#213) |
| `v0.1.4` | 2026-04 | fix: Railway standup hardening — dynamic auth URLs, web proxy vars (#211) |

These five tags represent the V0.5 deployment-pipeline shakedown. Brady chose to **abandon SemVer** at the V0.5 → CalVer rollover.

### Post-2026-04-20: CalVer (16 tags so far, active)

Canonical doc: `/Users/brady.hunt/Developer/trust-ai-app/docs/release/image-tags.md`.

| Pattern | Example | Meaning |
|---|---|---|
| `vYYYY.MM.DD` | `v2026.04.20` | Scheduled release. Cut on `main` HEAD by tagging. Also updates `latest`. |
| `vYYYY.MM.DD.N` | `v2026.05.05.4` | Nth hotfix on top of that day's release. Cut on a `hotfix/vYYYY.MM.DD.N` branch off the release tag, PR'd back to `main`, then tagged on the merge commit. |
| `latest` | `latest` | Floating GHCR tag pointing at the newest published image. **No git equivalent** — only an OCI tag. |

**Quoted rationale** from the CalVer commit (`08135576`, PR #214):

> Adopt CalVer. Release tags move from SemVer (`v0.1.0`) to CalVer (`vYYYY.MM.DD`) with `.N` hotfix suffix (`v2026.04.20.1`). Unified version applies to all four container images so deployed envs are always consistent across services.

### Recent CalVer cadence (most recent 10 tags)

```
v2026.05.19  (1 release)
v2026.05.11  + v2026.05.11.1
v2026.05.06
v2026.05.05  + v2026.05.05.1 + .2 + .3 + .4  (4 hotfixes on one cut)
v2026.05.04  + v2026.05.04.1 + .2 + .3       (3 hotfixes)
v2026.04.20  + v2026.04.20.1 + .2            (2 hotfixes)
```

Observation: cadence is **bursty, not weekly**. Cuts happen when something is ready; hotfixes pile on the same day's tag rather than waiting for the next cut. May 4–6 was a 3-day stretch with 8 tags. May 11 was 2. May 19 was 1. No fixed schedule.

### Internal milestone naming (separate axis from tags)

In parallel with CalVer tags, the app has a **product-milestone vocabulary** that does NOT map 1:1 to tags:

- **V0** — April 2026, completed. Documented in `docs/RELEASE_NOTES_v0.md`. Covers the agent-evaluation-loop MVP (Sessions → Annotation Queue → Datasets → Evaluators → Evaluations → Dashboard). Closed by M0–M10.
- **V0.5** — April 2026, completed. The Railway deploy + GHCR pipeline work, not a customer-facing release. Documented across `docs/planning/v0-5-tickets/`. Mostly the `v0.1.0`–`v0.1.4` SemVer tags.
- **Paddington v1** — April 2026, closed by DEV-472 / PAD-M4-07. Documented in `docs/RELEASE_NOTES_paddington_v1.md`. Trust Agent conversational layer, OOTB skills, approval gate, floating chat. **Crucially**: this "v1" is the Paddington feature's v1, not the platform's v1. It shipped under CalVer tag(s), not a `v1.x` SemVer.
- **Current** — internal milestone tickets are now `M5-NN`, `M4-NN` etc. Visible in the recent commit log (`DEV-1748: M4-07 — Strands version pin`).

This is the spike's biggest semantic landmine: **"v0" and "v1" mean two different things** depending on which axis you read.

- Marketing / product axis: V0 (April 2026 launch), Paddington v1 (April 2026 conversational layer), future V1, V2 etc.
- Engineering / tag axis: `v0.1.x` (legacy SemVer) → `v2026.MM.DD[.N]` (current CalVer). No "v1" tag exists or will exist.

The docs site must speak the product axis. Aligning docs to engineering tag names would be incomprehensible to a customer.

## Release artifacts produced today

### CI / CD workflows

- **`.github/workflows/ci.yml`** (33 KB) — Runs on every push and PR. Lint, typecheck, build, test, alembic, E2E. No version logic; does not publish artifacts.
- **`.github/workflows/publish-ghcr.yml`** (5.3 KB) — **Tag-triggered only** (`on: push: tags: ["v*"]` plus `workflow_dispatch`). Builds four multi-arch images, tags `<git-tag>` and `latest`, opens a follow-up PR to bump `values/<service>/values-dev.yaml` `image.tag`.

**Critical detail**: no workflow today publishes anything other than container images. Nothing in CI exports the OpenAPI JSON as a tag-associated release asset, generates a TypeScript-types tarball, or produces a changelog JSON. The only release "output" is the four GHCR images plus the git tag itself.

### Per-release artifacts (today)

| Artifact | Produced where | Machine-readable? | Used by docs? |
|---|---|---|---|
| Git tag (`vYYYY.MM.DD[.N]`) | Manual `git tag` + push | Yes (git API, gh CLI) | Not yet |
| GHCR images × 4 services | `publish-ghcr.yml` | Yes (registry API) | No — docs don't pull images |
| Helm values bump PR | `publish-ghcr.yml` follow-up | Yes (YAML diff) | No |
| `services/api/openapi.json` | Committed at HEAD, **not regenerated per tag** | Yes (OpenAPI 3.1.0) | **Yes — this is the obvious feed** |
| `RELEASE_NOTES_*.md` | Hand-written, per major product milestone | Partially (Markdown — parseable, but not a per-tag manifest) | **Yes — direct content port** |
| `CHANGELOG.md` | **Does not exist** | n/a | n/a |
| GitHub Release (with body, assets) | **Not used** today — tags are pushed without creating Releases | n/a | n/a |

### Where `openapi.json` comes from

FastAPI auto-generates the OpenAPI schema from route decorators + Pydantic models in `services/api/src/`. The committed `services/api/openapi.json` (659 KB at `info.version: "0.1.0"`) is a **snapshot**, not a build artifact — there is no scripted regeneration step in `services/api/scripts/`. To refresh it, someone runs the app and curls `/openapi.json`, or imports `app` and calls `app.openapi()`.

This is the **biggest mechanization opportunity** the docs-side workflow can exploit: a 10-line script in `services/api/scripts/dump_openapi.py` wired into a tag-triggered workflow would publish a per-tag OpenAPI artifact that the docs repo can consume.

### Tower orchestrator (release-relevant behavior)

`tools/tower/` is the Claude-Code autonomous-dispatch orchestrator. Inspection of `tools/tower/src/` shows it owns:

- agent runs (`agents/`, `prompts/`)
- PR creation + merge gating (`github/`, `merge/`)
- preview environments (`preview/`)
- worktree isolation (`worktree/`)
- E2E (`scripts/`)

Tower does **not** produce releases. Releases are still manual `git tag && git push --tags`. Tower could grow a `tower release` command (M5+ idea), but today the human is in the loop. Important for the docs spike because it means the docs-side workflow must **react to a tag push event** — there's no upstream pipeline to plug into.

## What's currently machine-readable enough to feed docs

**Tier 1 — already exists, ready to consume:**

1. **`services/api/openapi.json`** — canonical `/v1` API surface. Includes paths, schemas, security, descriptions from FastAPI route decorators. This is what `reference/api.mdx` should be generated from, full stop.
2. **Git tags** — `git tag -l --sort=-creatordate` is the release feed. `gh api repos/.../tags` is the same data over HTTPS.
3. **`docs/RELEASE_NOTES_v0.md` + `docs/RELEASE_NOTES_paddington_v1.md`** — direct content port to `docs.provar.dev/changelog/v0` and `/changelog/paddington-v1`. No regeneration needed; they're already milestone-organized Markdown.

**Tier 2 — exists in the repo but not yet wired as a release artifact:**

4. **SQLAlchemy models** (`services/api/src/domain/*/models.py`) — could be reflected into a `schemas/` glossary, but Pydantic schemas (already in `openapi.json`) are the customer-facing version. Models = internal storage shape, schemas = wire shape. Docs want wire shape.
5. **Pydantic schemas** (`services/api/src/domain/*/schemas.py`) — already flow into `openapi.json`. No separate export needed.
6. **Frontend TypeScript types** generated via `openapi-typescript` (visible in `package.json` devDependencies and `pnpm generate-types`). These are derived from `openapi.json` — docs should re-derive from the same source, not from the FE-generated `.d.ts`.

**Tier 3 — would need a new emitter to exist:**

7. Per-tag JSON release manifest (`{ "tag": "...", "openapi_version": "...", "linear_tickets": [...], "summary": "..." }`).
8. Concept/schema glossary as JSON (the seven nouns: Projects, Sessions, Annotations, Datasets, Evaluators, Evaluations, Agent Versions).
9. Skill registry JSON dump (for Paddington Trust Agent — which OOTB skills exist at this tag).

Tier 3 is **out of scope** for the spike's MVP recommendation. Tier 1 is enough.

## Decision-gate answers (from this audit)

> **Is Trust AI versioning by semver, calendar, or rolling?**

**CalVer (`vYYYY.MM.DD[.N]`)** since 2026-04-20. The product axis is separately milestone-named (V0, Paddington v1, future V1). The docs site should speak the **product axis** — customers care about "V1" and "Paddington v1", not `v2026.05.19`.

> **How many simultaneous versions to maintain?**

For the next 12–18 months: **one current version**. V0 launched April 2026; V1 is the next product cut and there is no public V0 self-host. With CalVer's rolling-release shape and no customer self-host yet, customers consume `latest`. Docs follow latest. A "previous-version" preservation question becomes real only once a V1 cut happens AND there are V0 customers who haven't migrated. Until then, the changelog narrates history.

> **Is OpenAPI the canonical machine-readable interface, or are there other artifacts to consume?**

**OpenAPI 3.1.0 is canonical**. FastAPI generates it, the FE consumes it via `openapi-typescript`, and `info.version` is the one field that flips per release. Other potentially-consumable artifacts (SQLAlchemy models, raw Pydantic schemas, Helm values, GHCR tag list) are either internal-shape (wrong audience) or derived from the OpenAPI schema. There is one canonical interface.

> **Do customers self-host on older versions?**

**Not yet, and not on the V0/V1 timeline.** The current deploy target is Railway running TrustAI-owned envs (QA, dev, prod). The README and deploy docs assume Provar runs the stack. There is no self-host distribution, no customer-facing helm chart, and no published image-pull instructions outside the internal `docs/release/image-tags.md`. **Implication**: old-version docs do not need to stay live indefinitely. The changelog is sufficient archival history; we do not need to keep `/v0/api` live for self-hosters who lag behind. Re-evaluate when (and if) a self-host SKU ships.

> **Does Mintlify versioning require a paid tier?**

Answered in `mintlify-versioning-options.md`. Short answer: the `versions` field exists in the published `docs.json` schema with no tier gate visible. Pricing page does not list "versioning" as a paid feature. **Provisional verdict: free tier supports it**, pending confirmation by toggling the feature on a non-prod Mintlify project before committing.

## Implications for the docs-side architecture

1. **Docs version names must match the product axis (V0, V1) not the tag axis (v2026.MM.DD).** The tag is too noisy and too internal to surface to customers.
2. **There is exactly one upstream signal that matters**: `services/api/openapi.json` at a given tag. Everything else (release notes, changelog entries, concept-doc updates) is human-authored Markdown that lives in the docs repo and is updated by the docs-side workflow when the human says "ship the docs for V1."
3. **The app-side has no obligation to emit anything new for V1**. The docs-side workflow can `gh api` the tag list, `wget` the OpenAPI JSON from the tagged commit on GitHub raw, and regenerate `reference/api.mdx` without any change to the app repo. (Optional improvement: add a `dump_openapi.py` script and a tag-triggered "publish OpenAPI as release asset" job — but not blocking.)
4. **Rolling CalVer + no self-host = single-version-docs is the right default.** A version dropdown is a future-need, not a v1-need.
