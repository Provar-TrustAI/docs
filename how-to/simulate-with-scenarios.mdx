This how-to walks you through driving a connected agent against **personas** and **goals** end to end. It follows the loop the **Scenarios** surface is built around: define a scenario cheaply, run it to generate real sessions, set the expected output from a session you like, then run and evaluate. After you've worked through it:

- You can create a scenario — name, goal, personas — with the Trust Agent or the **New scenario** modal
- You can generate real, scenario-linked sessions by running a scenario, and capture a **Playground** conversation with **Save as a scenario**
- You can author a scenario's **Expected Output** from a good session and read graded, per-persona results

This page is a task recipe. For what scenarios, personas, goals, and the Playground *are*, read the concept pages it links to — [Scenarios](/concepts/scenarios) and [Playground](/concepts/playground) — rather than re-deriving them here.

## Prerequisites

- A **Trust AI account** with access to at least one [Project](/concepts/projects)
- At least one **connected agent** on the Project (every run in this recipe drives a real agent)
- **Personas** are optional but recommended — a scenario simulates one session per attached persona, so persona coverage is where the fan-out comes from

## Create a scenario

The fastest way to persona-and-goal coverage is to let the [Trust Agent](/concepts/trust-agent) draft it for you; or create scenarios yourself from the **Scenarios** list.

<Note>
  *Ask the Trust Agent:* "Create scenarios covering my churn-risk and new-trial personas for the goals 'check order status' and 'request a refund.'" It proposes persona-and-goal test cases — no transcripts, just the definitions — and adds the ones you confirm to your Scenarios list.
</Note>

<Steps>
  <Step title="Open the Scenarios list">
    *Ask the Trust Agent:* "Show me my scenarios," or open the **Scenarios** nav item on your Project yourself — it lands at `/projects/:id/scenarios` (old dataset links redirect here). Every scenario is one entry in a flat table with a stable **ID** (like `SCN-000123`) plus **Scenario**, **User Goal**, **Personas**, **Expected routes**, **Expected actions**, **Evaluators**, **Sessions**, and **Tags** columns. Clicking a scenario opens its detail fly-in.
    {/* CAPTURE-PENDING: scenarios-list screenshot — the flat Scenarios table with ID / Personas / Expected routes columns and the header actions */}

    **You should see:** the Scenarios table render with SCN-prefixed ids, and clicking any scenario open a fly-in panel.
  </Step>

  <Step title="Create one with New scenario">
    Click **New scenario**. The modal collects only three things — a **Name** (required), a **Goal** (required — what the simulated user is trying to achieve), and **Personas** (optional; the label reminds you it's one session per persona on each run). It also tells you expected output comes later, usually from a session you like — that deferral is the point of the loop. Click **Create scenario**.

    Two shortcuts worth knowing: **Add recommended scenarios** in the page header adopts curated starter scenarios in one move, and the **Name**, **Goal**, **Personas**, **Evaluators**, and **Tags** cells are all editable inline on the table — no modal round-trip to adjust a scenario later.

    **You should see:** your scenario appear in the table with its own ID, personas attached, and no expected output yet.
  </Step>

  <Step title="Manage your personas (once)">
    *Ask the Trust Agent:* "List my personas and tell me which one is the default." To review them yourself, open **Manage personas** from the Scenarios page header (the same modal also opens from any **Personas** cell's popover footer). Each persona shows a **usage count**, at most one persona is the **project default** at a time, and personas are referenced by **name** everywhere — chips, columns, and pickers.

    **You should see:** a persona list with usage counts and exactly one persona marked default.
  </Step>
</Steps>

## Generate sessions by running the scenario

A scenario generates sessions by being **run**: each run simulates one fresh persona-versus-agent conversation per attached persona, and every conversation persists as a real [Session](/concepts/sessions) linked to the scenario. To review conversations before you care about grades, run without evaluating.

<Note>
  *Ask the Trust Agent:* "Run my refund-request scenario against my support agent." It launches the persona fan-out for you. To generate sessions without creating a graded run, use **Run only** by hand — the steps below.
</Note>

<Steps>
  <Step title="Start a run">
    Select your scenario and use the **Run** split-button. The primary **Run** action simulates *and* grades (the menu spells it out: **Run & evaluate** — "Simulate + grade → creates an evaluation"); the chevron also offers **Run only** — "Simulate without grading." The same two actions live on the scenario fly-in's **More** menu, and the fly-in's empty **Sessions** tab leads with them: run the scenario to simulate one session per persona, review them, then set expected output from a good one.

    **You should see:** a **Run** split-button whose chevron offers **Run only** and **Run & evaluate**.
  </Step>

  <Step title="Confirm the fan-out and launch">
    Choosing **Run only** opens the evaluation launcher titled **Run scenario**. Pick the **connected agent** to drive, and check the **This will run** preview — it counts one simulated conversation per attached persona. A goal-only scenario with no personas still runs one conversation, with the simulated user enacting the goal. The confirm button reads **Run N conversations**.
    {/* CAPTURE-PENDING: evaluation-launcher screenshot — the Run scenario launcher with the agent picker and the "This will run" fan-out preview */}

    **You should see:** the launcher preview one conversation per persona and the button count match it.
  </Step>

  <Step title="Watch the sessions land">
    You stay on the Scenarios list while the run is in flight — the scenario shows a live **Generating…** badge that settles when it finishes. The finished conversations appear on the scenario's **Sessions** tab (the tab shows a live count), each carrying the **Scenario** source chip. A **Run only** run creates no entry in your Evaluations list — it exists to produce sessions, not grades.

    **You should see:** a **Generating…** badge during the run, then new sessions under the scenario's **Sessions** tab with Source **Scenario**.
  </Step>
</Steps>

## Save a Playground conversation as a scenario

The **Playground** is the hands-on counterpart: drive a connected agent yourself (**Chat**) or auto-drive one persona against it (**Simulate**), then capture a conversation you like as a new scenario. This is a hands-on surface, so you work in the UI directly — the [Trust Agent](/concepts/trust-agent) helps around the edges (opening it, finding past runs).

<Steps>
  <Step title="Drive a conversation">
    Open the Playground at `/projects/:id/playground` and pick a connected agent — the picker groups agents by source system. A segmented **Chat / Simulate** toggle controls the mode. On **Chat**, you type each message yourself and the connected agent replies live. On **Simulate**, a config modal takes exactly **one persona**, **one goal**, and a run mode — **Full auto** ("Simulates the whole conversation end-to-end") or **Stay in the loop** ("Suggests each user turn as you go") — and the persona-versus-agent conversation auto-drives. The **History** menu lists your real prior sessions so you can reopen an earlier run.
    {/* CAPTURE-PENDING: playground-simulate-modal screenshot — the Simulate config modal (one persona, one goal, Full auto / Stay in the loop) */}

    **You should see:** the **Chat / Simulate** toggle, and a Simulate run auto-drive the conversation without you typing turns.
  </Step>

  <Step title="Save it as a scenario">
    When a conversation is worth keeping, choose **Save as a scenario** (available once the conversation has at least one exchange). The form is the same lightweight intake as **New scenario** — **Name**, **Goal**, **Personas** — and saving creates a new scenario with the conversation you just ran attached as its session, Source **Scenario**. Expected output is deliberately deferred: the fresh scenario reads as missing expected output until you author it.

    **You should see:** a new scenario on the Scenarios list whose **Sessions** tab already holds the captured conversation.
  </Step>
</Steps>

## Set the expected output from a good session

Expected Output is the scenario's grading contract, and the loop is designed so you author it *after* you've seen real sessions — from vocabulary the product learned from them, not from guesswork.

<Steps>
  <Step title="Review the sessions">
    Open the scenario's fly-in. Its tabs follow the workflow: **Personas & Goals**, **Expected Output**, then **Sessions**. On the **Sessions** tab, open any session into the nested detail panel to read its full transcript and its evaluations. Find a session that behaved the way you want.

    **You should see:** the scenario fly-in's tabs, and a nested panel showing a single session's transcript.
  </Step>

  <Step title="Author the Expected Output">
    On the **Expected Output** tab, define the contract in three sections: **Expected behavior** (LLM-judge evaluator cards, with a **Recommended for this scenario** group surfaced first), **Expected actions** (**Add action**, with parameter checks), and **Expected routes** (**Add route**). The pickers offer options observed in real sessions, so you choose what "good" looked like rather than typing it from memory.
    {/* CAPTURE-PENDING: expected-output-editor screenshot — the three sections (Expected behavior / Expected actions / Expected routes) with the learned-vocabulary picker open */}

    **You should see:** the three sections, and the action/route pickers offering values seen in your sessions.
  </Step>

  <Step title="Find the scenarios still missing one">
    The Scenarios list ships a built-in **Missing expected output** saved view — use it to sweep the scenarios (including freshly saved Playground captures) that can't be graded structurally yet.

    **You should see:** the saved view filter the table down to scenarios without a contract.
  </Step>
</Steps>

## Run and evaluate

<Note>
  *Ask the Trust Agent:* "Run and evaluate my refund-request scenario." It launches the graded run for you; the steps below do the same by hand and read the results.
</Note>

<Steps>
  <Step title="Launch a graded run">
    Use **Run & evaluate** — from the **Run** split-button, the fly-in **More** menu, or the Sessions tab. The launcher opens as **Run & evaluate scenario** (or **scenarios** for a bulk selection): pick the connected agent, check the fan-out preview, and launch. The evaluators come from each scenario's Expected Output — they're inherited into the run, not re-picked. A scenario run **always simulates** fresh conversations; there is no replay opt-in to remember.

    **You should see:** the launcher show your agent, the inherited evaluators, and the persona fan-out before you confirm.
  </Step>

  <Step title="Read the per-persona results">
    The finished run leads with a dual headline: **Pass rate** grades the *agent*, and **Evaluator accuracy** grades the *judges* — how often the automated evaluators agreed with a person's session-level verdict. Below it, a per-persona matrix scores every attached persona on its own, with plain-English hints where personas diverge. Each result's **Transcript** tab labels itself **Simulated** or **Replayed**, and **Open session** takes you to the exact session that was graded.
    {/* CAPTURE-PENDING: run-detail screenshot — the dual headline (Pass rate + Evaluator accuracy) over the per-persona matrix */}

    To grade conversations that *already happened* — synced or captured sessions, with no simulation — use the **Evaluate** action on the [Sessions](/concepts/sessions) surface instead.

    **You should see:** two headline numbers, one result per persona, and a **Simulated** label on the transcripts this run produced.
  </Step>
</Steps>

<Note>
  **The wire surface behind the flow.** Scenario reads and edits go through the `/v1/scenarios` API family (the legacy `/v1/datasets` container API survives underneath — the re-root is a rename over existing storage). Request and response shapes live in the [API Reference](/api-reference). All of your scenario and simulation data is **scoped to your organization** (see [Multi-tenancy](/concepts/multi-tenancy)).
</Note>

## Common stumbles

<AccordionGroup>
  <Accordion title="Where did scenario sets go?">
    They're gone as of v2026.06.30.1. Every project has one flat **Scenarios** list — there is no set to pick, and old dataset links redirect to `/projects/:id/scenarios`. A few vestigial in-app strings still say "scenario set"; the surface itself has no sets. See [Scenarios](/concepts/scenarios) for the current model.
  </Accordion>

  <Accordion title="I can't find the Simulate at eval time toggle">
    It no longer exists on any reachable surface. A scenario run **always simulates** — one fresh conversation per attached persona. Your choice is whether to grade: **Run & evaluate** simulates and grades; **Run only** simulates without grading. To grade existing conversations instead of simulating new ones, use **Evaluate** on the Sessions surface.
  </Accordion>

  <Accordion title="Simulate in the Playground only takes one persona and one goal">
    That is correct — the Playground's **Simulate** modal drives exactly one persona against the agent per run, so you can watch it turn by turn. For persona coverage, attach multiple personas to a scenario and run it: the run fans out one session per persona in a single launch.
  </Accordion>

  <Accordion title="My saved scenario says it's missing expected output">
    By design. **Save as a scenario** and **New scenario** deliberately defer the grading contract — the loop is define cheap, run, then set expected output from a session you like. Open the scenario's **Expected Output** tab to author it; until then the scenario shows up in the **Missing expected output** saved view.
  </Accordion>

  <Accordion title="I can't delete a scenario">
    A scenario referenced by evaluations is blocked from deletion, and the app tells you so ("Can't delete: used in N evaluations."). When a delete *is* allowed but the scenario has generated sessions, the confirm discloses what happens: the sessions are **kept** as frozen records (only their link to the scenario is removed), while the scenario's expected-output rubric goes away with it.
  </Accordion>
</AccordionGroup>
