An Annotation is a piece of **human judgment** attached to a [Session](/concepts/sessions) or to a single Turn within one. Pass/fail, rubric score, freeform note, or tag — Annotations are how reviewer judgment becomes data. They are the **signal** every [Evaluator](/concepts/evaluators) calibrates against, and the **filter** that selects rows worth promoting into a [Dataset](/concepts/datasets).

Without Annotations, the rest of Trust AI is theater: Evaluators would have nothing to calibrate to, Datasets would be unfiltered firehoses, and Evaluations would produce scores no one trusted.

## Annotation types

An Annotation can be any of:

- **Pass / fail label** — binary reviewer judgment. The most common form; cheap to apply, easy to aggregate.
- **Rubric score** — Likert (1-5), letter grade, or domain-specific scale. Useful when binary is too lossy.
- **Freeform note** — prose explaining *why* this Session is interesting, broken, or representative. The richest form; hardest to aggregate.
- **Tag** — a named label applied to a Session for filtering, slicing, or bulk operations ("regression," "happy-path," "high-priority").

A single Session can carry many Annotations across types: a pass/fail at the Session level, a 1-5 rubric on Turn 3, freeform notes on Turns 1 and 5, and a set of tags applied for filtering. The annotation surface is additive.

## Session-level vs Turn-level

Annotations attach at one of two granularities:

- **Session-level** — judges the whole interaction. "Did this Session as a whole resolve the user's intent?"
- **Turn-level** — judges a specific exchange or tool call. "Did Turn 3 — the refund lookup — return the correct value?"

The original Trust AI design (ADR 0005) considered a third level, "rule-level," where Annotations would attach to a specific rule firing within a Turn. We dropped it: it was conceptually pure but operationally painful (reviewer cognitive load, annotation queue branching). Session-level and Turn-level cover every real workflow we've observed.

## The annotation queue

The **annotation queue** is the shared workspace where QA Practitioners apply Annotations. It is the surface they live in.

A queue is typically:

- **Filtered** to Sessions needing review (no Annotation yet, or annotated by someone whose judgment another reviewer wants to verify)
- **Sliced** by criteria — by intent classification, by tag, by Agent Version, by time window
- **Bulk-actionable** — apply a tag or a label to many Sessions at once; bulk-add a filtered slice to a Dataset

A team's annotation rate is the practical bottleneck on Dataset quality. A reviewer churning through 50 Sessions a day will, over a quarter, produce enough signal to calibrate Evaluators well; one churning through five will not. The queue UX exists to optimize for the former.

## Annotations vs Evaluator scores — different things, paired

It's easy to confuse Annotations and Evaluator scores. They're structurally similar (a number or label attached to a Session/Turn) but they come from different sources and serve different purposes:

| | Annotation | Evaluator score |
|---|---|---|
| **Source** | Human reviewer | LLM Judge or Structured Match |
| **Purpose** | Calibrate Evaluators; curate Datasets | Compare Agent Versions at scale |
| **Cost per row** | Reviewer minutes | LLM tokens / compute |
| **When applied** | After ingestion, during review | At Evaluation time, programmatically |
| **Authoritative?** | Yes, by definition | Only insofar as it agrees with Annotations |

The relationship is **calibration**: Evaluator scores are trusted to the degree they agree with Annotations on the same rows. The calibration loop ([Evaluators → Calibration](/concepts/evaluators#calibration-the-loop-that-makes-llm-judges-trustworthy)) is the bridge between the two.

## What you do with annotated data

Annotations only justify their cost if they get used downstream. The three uses, in order of payoff:

1. **Curate Datasets** — filter to annotated Sessions of interest, bulk-add to a Dataset. The human judgment travels with each row.
2. **Calibrate Evaluators** — pick annotated rows, run the draft Evaluator against them, compare scores. Refine the rubric until agreement is high.
3. **Spot-check Agent Versions** — when you suspect a regression, filter to recent annotated Sessions of a specific category and inspect the deltas across Agent Versions before running a full Evaluation.

All three close the loop on the [evaluation loop](/concepts/evaluation-loop) — Annotations are the input that makes everything downstream trustworthy.

## API surface

Annotations appear in the `/v1/annotations` endpoint family of the [API Reference](/api-reference). Common operations:

- `POST /v1/sessions/{session_id}/annotations` — apply a Session-level Annotation
- `POST /v1/sessions/{session_id}/turns/{turn_id}/annotations` — apply a Turn-level Annotation
- `GET /v1/annotations` — list / filter Annotations in a Project
- `POST /v1/annotations/bulk-tag` — apply a tag to a filtered Session set
- `DELETE /v1/annotations/{annotation_id}` — revoke an Annotation (rare but supported)

Full request/response shapes live in the auto-generated [API Reference](/api-reference).
