A Dataset is a curated, versioned collection of test cases for a Trust AI [Project](/concepts/projects). Each row in a Dataset is something worth scoring an agent on: a Session, a Turn, a manually-written prompt, or a CSV-imported example. Every [Evaluation](/concepts/evaluations) runs against a Dataset, which is what lets results stay comparable across runs and Agent Versions.

## Why Datasets exist as a separate concept

You could imagine running Evaluators directly against all Sessions. That breaks down quickly:

- The Sessions list changes every minute as new ones get ingested
- Comparing Evaluator results across runs is meaningless if the input set changes
- You want to evaluate against the "hard cases" specifically, not the firehose
- Some test cases never come from real Sessions (edge cases you anticipate but haven't seen)

A Dataset is the **stable, curated, deliberately-selected test bed** — a unit of evaluation worth maintaining over time. Think of it like an integration test suite for agent behavior.

## What's in a Dataset

A Dataset has:

- **A name + description** — what this Dataset is for ("Order-status happy path," "Refund flow regression set," etc.)
- **Rows** — the test cases themselves
- **Version** — every Dataset is implicitly versioned; edits create a new revision (see "Snapshots" below)
- **Tags** — for filtering and organization
- **Annotations** baked in — when rows come from annotated Sessions, the human judgment travels with them

Each row in a Dataset contains:

- **Input** — the prompt or Session that gets sent to the agent at evaluation time
- **Reference output** (optional) — what "good" looks like for this row, used by some Evaluator types
- **Reference annotations** (optional) — the human signal that was attached to the source Session
- **Row-level tags** — for slicing results ("regression," "high-priority," "happy-path," etc.)

## Snapshots, not deep-links

A critical design decision: when you add a Session to a Dataset, Trust AI **snapshots the Session content into the Dataset row** rather than holding a pointer to the original.

Why this matters:

- **Edits upstream don't change historical results.** If you re-ingest a Session or its Annotation changes, prior Evaluations are unaffected. The row is what was scored.
- **Datasets stay reproducible across time.** An Evaluation from two months ago can be re-run with confidence today.
- **You can edit a Dataset row** without affecting the source Session — useful for prompt tweaks specific to a test case.

The cost: a Dataset and its source Sessions can diverge. If a Session is corrected upstream, the Dataset doesn't auto-update — that's a deliberate curation choice ("does the prior test case still represent what I want to test?").

(For the full architectural rationale, see ADR 0004 "Dataset snapshots, not deep-links" in the trust-ai-app repo.)

## Versioning + the regression-tracking guarantee

Every Dataset edit is treated as a new revision. Evaluations always pin to a specific Dataset revision, so:

- An Evaluation run yesterday is reproducible today (same Dataset version → same input rows)
- You can compare the same Agent Version across two Dataset revisions — useful when you've expanded the Dataset and want to know if the new rows surface anything
- You can compare two Agent Versions across the same Dataset revision — the **core regression-tracking workflow**

The combination of "Datasets are snapshots" + "Datasets are versioned" + "Evaluations pin to a Dataset version" is what makes Trust AI evaluations trustworthy as a regression signal.

## Working with Datasets

Common operations in the app:

- **Filter rows** by tag, source, annotation status, or content
- **Bulk-tag** subsets ("regression," "smoke-test")
- **Export to CSV** for offline analysis
- **Snapshot the Dataset** to freeze a state for an upcoming Evaluation run
- **Branch a Dataset** to experiment with variants without touching the original

## API surface

Datasets appear in the `/v1/datasets` endpoint family of the [API Reference](/api-reference). Common operations:

- `POST /v1/datasets` — create a Dataset
- `GET /v1/datasets/{dataset_id}` — fetch Dataset metadata
- `GET /v1/datasets/{dataset_id}/rows` — list rows (paginated)
- `POST /v1/datasets/{dataset_id}/rows` — add a row (manual entry or programmatic)
- `POST /v1/datasets/{dataset_id}/rows/bulk-add-from-sessions` — curate from a Session set
- `POST /v1/datasets/{dataset_id}/snapshot` — freeze the Dataset at the current revision

Full request/response shapes live in the auto-generated [API Reference](/api-reference).
