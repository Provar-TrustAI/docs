A **scenario** is one test case worth scoring an agent on — a curated [Session](/concepts/sessions), a hand-written prompt, a CSV-imported example, or a persona-and-goal pair: an actor with a system prompt and a goal it is trying to accomplish. A **Scenario set** is the curated, versioned collection of those rows — the stable test bed every [Evaluation](/concepts/evaluations) runs against. You build scenario sets by curating sessions, writing rows by hand, importing a CSV, generating rows from scratch with a wizard, or capturing them from a live [Playground](/concepts/playground) run, then score an agent against them.

<Note>
  **Formerly Datasets.** The surface and its collections were renamed to Scenarios — the left-nav row reads **Scenarios** and a collection is a **Scenario set** — but the route and the API did not move: you open the surface at `/projects/:id/datasets`, list calls hit `/v1/datasets`, wire fields keep names like `dataset_id`, and **there is no scenarios endpoint**. This page uses **scenario** and **scenario set** in prose, because that is what the UI says; where a route or field name matters, it is shown in code as its real `datasets` token.
  {/* ACCURACY-AUDIT-PENDING: nav label reads "Scenarios", route is /projects/:id/datasets, list calls hit /v1/datasets, and no /v1/scenarios endpoint exists — verify by driving the route + network tab on the release tag */}
</Note>

<Note>
  **Scenarios is not the Trust Agent, and not an Agent Version.** All three use the word "agent" nearby, so it is worth pinning them apart:

  - **Scenarios** (this page) are the *test cases* you score an agent against — curated sessions, hand-written prompts, persona-and-goal rows. A curation surface — a table of rows.
  - The **[Trust Agent](/concepts/trust-agent)** is your in-app *assistant* — the thing that helps you build sets, draft evaluators, and run evaluations. It lives at `/projects/:id/agent`.
  - An **[Agent Version](/concepts/agent-versions)** is the *agent under test* — an immutable snapshot of the customer's agent whose behavior you are judging.

  You score an Agent Version against a scenario set, and the Trust Agent can help you do it. None of the three is the same thing.
</Note>

## What's in a scenario set

A scenario set is a named, versioned collection of rows, with persona-and-goal columns layered onto the test-case content:

- **A name + description** — what this set is for ("Refund-flow personas," "Onboarding edge cases").
- **Rows** — the scenarios themselves. Each row is one test case.
- **Version** — every set is implicitly versioned; edits create a new revision, and Evaluations pin to a revision (see [Snapshots and reproducibility](#snapshots-and-reproducibility)).
- **Tags** — for filtering and organization.
- **Annotations baked in** — when rows are curated from annotated [Sessions](/concepts/sessions), the human judgment travels with them.

Each row in the detail view carries:

- **Persona** — the actor that role-plays the user for this scenario (shown by name as a chip). Rows curated from sessions may have no persona — they replay instead of simulate (see [Which rows are simulatable](#which-rows-are-simulatable)).
- **Goal** — what that persona is trying to accomplish.
- **Source** — the row's origin: `uploaded`, `generated`, or `simulated` (see [Where scenarios come from](#where-scenarios-come-from)).
- **Input** — the prompt or snapshotted Session content that gets sent to the agent at evaluation time.
- **Reference output** (optional) — what "good" looks like for this row, used by some Evaluator types.
- **Reference annotations** (optional) — the human signal that was attached to the source Session.
- **Row-level tags** — for slicing results ("regression," "high-priority," "happy-path").

## Personas

A **persona** is a reusable, project-scoped actor that role-plays the user in a scenario — a system prompt plus the goals and constraints that shape how it behaves. Personas are shared across the project, so the same actor can drive many scenarios.

### Manage Personas

You manage personas in a **Manage Personas** modal, opened from the **Persona** cell's **Manage personas** popover footer on a scenario-set detail. It is a modal that opens in place — the page does not navigate to a separate Settings screen.

## Where scenarios come from

Every row carries a **Source** that records how it got into the set — one of three origins:

  <Card title="Generated" icon="wand-magic-sparkles">

The **Source** column on the detail view reflects each row's origin, so you can always tell a hand-written case from a generated or captured one.

### Curating sessions into a scenario set

The richest `uploaded` path starts in [Sessions](/concepts/sessions). You could imagine running Evaluators directly against all Sessions, but that breaks down quickly: the Sessions list changes every minute as new ones get ingested, comparing results across runs is meaningless if the input set keeps shifting, and you want to evaluate against the hard cases specifically — not the firehose. Some test cases never come from real Sessions at all. A scenario set is the **stable, curated, deliberately-selected test bed** — think of it like an integration test suite for agent behavior.

The common flow: annotate Sessions, filter the Sessions list down to the ones worth testing against, and bulk-add them to a scenario set — either from the list itself or by asking the [Trust Agent](/concepts/trust-agent) to curate for you ("add the failing refund sessions to my regression set"). The human judgment travels with each row as reference annotations, and the row lands with an `uploaded` origin.
{/* ACCURACY-AUDIT-PENDING: sessions can be bulk-added to a scenario set from the Sessions list and via the Trust Agent, and session-curated rows land with an uploaded origin — verify the bulk-add affordance and the resulting Source value on the release tag */}

### CSV import and export

The other `uploaded` path is a spreadsheet. Drop in a CSV with columns that map to input, reference output, and tags — useful for migrating existing test suites from spreadsheets or other tools. The same toolbar takes rows back out: select rows (or the whole set) and export to CSV for offline analysis or sharing.

### The generate-from-scratch wizard

The **Generate scenarios from scratch** wizard builds a whole scenario set from your personas and goals. It runs in four logical steps, but the progress track shows **three pips** — generate and review share the last pip.

  </Step>
</Steps>

## Snapshots and reproducibility

Scenario-set rows are **snapshots, not deep-links**. Adding a Session to a set captures its content into the row rather than holding a pointer to the original, which buys three guarantees:

- **Edits upstream don't change historical results.** If a Session is re-ingested or its Annotation changes, prior Evaluations are unaffected — the row is what was scored.
- **Sets stay reproducible across time.** An Evaluation from two months ago can be re-run with confidence today.
- **You can edit a row** without affecting the source Session — useful for prompt tweaks specific to a test case.

The cost: a scenario set and its source Sessions can diverge. If a Session is corrected upstream, the set doesn't auto-update — that's a deliberate curation choice ("does this test case still represent what I want to test?").

Versioning completes the guarantee. Every edit to a set is a new revision, and Evaluations always pin to a specific revision, so:

- An Evaluation run yesterday is reproducible today (same revision → same input rows)
- You can compare the same Agent Version across two revisions of a set — useful when you've expanded the set and want to know if the new rows surface anything
- You can compare two Agent Versions across the same revision — the **core regression-tracking workflow**

The combination — rows are snapshots, sets are versioned, Evaluations pin to a version — is what makes Trust AI evaluations trustworthy as a regression signal.

(For the full architectural rationale, see ADR 0004 in the trust-ai-app repo.)

## Simulation at evaluation time

Normally an Evaluation **replays** each row — it sends the row's stored input turns to the agent under test. Scenario sets unlock a second mode: instead of replaying stored turns, an Evaluation can run a **fresh persona-and-goal simulation** for each row, generating a brand-new conversation between the persona and the agent at evaluation time.

You turn this on per run with the **Simulate at eval time** toggle on the new-evaluation-run form.
{/* ACCURACY-AUDIT-PENDING: a "Simulate at eval time" toggle on the new-run form switches each row from replaying stored turns to a fresh persona+goal simulation — verify by starting a run against a persona+goal set on the release tag */}

### Which rows are simulatable

A row is **simulatable** only if it has **both** at least one persona **and** a non-empty goal. Rows missing either are replayed, not simulated — so a mixed set degrades gracefully, simulating the rows it can and replaying the rest.

The toggle reflects a **server-computed count of simulatable rows** in the selected set. A replay-only set — rows with no persona or goal — shows the toggle as unavailable or with a zero simulatable-row count, so you are never offered a simulation that can't run.
{/* ACCURACY-AUDIT-PENDING: a row is simulatable only with both a persona and a non-empty goal; the toggle reflects a server-computed simulatable-row count; a replay-only set shows the toggle unavailable / zero — verify by selecting a replay-only set and a persona+goal set on the new-run form on the release tag */}

### The simulated transcript

When a simulate-at-eval-time run finishes, the **full persona-to-agent conversation** captured for each row is persisted onto the result and rendered inside the **evaluation result detail fly-in** — turns tagged with role and persona, not just a score. You can read exactly what the persona said, how the agent responded, and judge the behavior in context.
{/* ACCURACY-AUDIT-PENDING: the simulated conversation transcript is persisted onto the result row and renders inside the evaluation result detail fly-in with role/persona-tagged turns — verify by opening a result fly-in after a simulate-at-eval-time run on the release tag */}

## API surface

The rename is UI-layer only: scenario sets are served by the same endpoint family as before, under the pre-rename route names spelled out in the note at the top of this page — and **there is no scenarios endpoint**. The persona-and-goal additions surface as fields on the existing wire shapes: a row's `origin` (`uploaded` / `generated` / `simulated`), a set's `simulatable_row_count`, and an evaluation run's `simulate_at_eval_time` flag. The earlier `v2026.05.19` additions — duplicate, CSV export, and the tag vocabulary — live in the same family.

Full request/response shapes live in the [API Reference](/api-reference).
