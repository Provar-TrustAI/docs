This is the **deep how-to** on authoring LLM Judge grading rules. If you just want to author your first Evaluator end-to-end, start with [Build your first Evaluator](/tutorials/build-first-evaluator). Come here when you need rules that:

- Calibrate to high agreement with a careful human reviewer
- Stay stable when you re-run them next month
- Don't drift across scenarios that should score the same

Writing grading rules is a craft. The patterns below come from teams that have iterated on dozens of rubrics; they save real time when you internalize them.

## You write rules — the product writes the prompt

You never author a judge prompt in Trust AI. An LLM Judge evaluator is defined by plain-language **grading rules**:

- A **pass/fail** judge takes two rule lists: **Passes when…** and **Fails when…** — each rule one short sentence.
- A **1–5 scale** judge takes three anchors: **Rate 5 when…**, **Rate 3 when…**, and **Rate 1 when…**.

The app assembles your rules into the exact criteria the live judge reads — each rule becomes its own line, so the judge can work through them one at a time. You can inspect the result any time under **Advanced — view the prompt this produces**, but it's read-only: everything else about the judge is owned by the product and is not an authoring surface.

| You write | Trust AI owns |
|---|---|
| The grading rules (or scale anchors) | The judge's system prompt and framing |
| The score type and pass threshold | The judge model |
| The **Reason step by step** toggle | The output format and verdict parsing |
| | Temperature — pinned to `0` on every judge call |

This narrows the craft in a useful way. All the classic judge-prompt advice about neutral system-prompt framing, output schemas, and temperature tuning is handled for you. **Your entire authoring surface is the rules** — which is exactly where the judgment quality lives.

</Note>

## Start from a recommended evaluator when one fits

Before hand-authoring from scratch, check the starter catalog: **Add recommended** on the Evaluators library offers twelve curated LLM-judge presets — six Quality, six Safety — that adopt as ordinary, fully editable evaluators. A few carry a **Tailor first** badge, meaning they were written against a generic agent: add a rule about what *your* agent is specifically allowed or required to do before relying on them. Adopting, tailoring one or two rules, then calibrating is usually faster than a blank page — see [Evaluators](/concepts/evaluators) for the full catalog.

## Write rules, not vibes

The biggest single mistake new authors make is writing a rule as **a vague quality judgment**:

```
❌ Fails when the response is bad.
❌ Passes when the agent helps the user.
❌ Passes when the answer is correct and helpful.
```

These look like rules; they aren't. They're **vibes**. "Bad," "helps," and "helpful" are under-specified, so the judge picks an interpretation per call — which means the evaluator can't be calibrated and can't be trusted as a regression signal.

Good rules are **specific, observable, and one criterion each**:

```
Passes when…
  · the response identifies the order ID from the customer's message
    (or asks for it when the customer didn't provide one)
  · the response states the order's current status, not a generic
    acknowledgement like "let me check"

Fails when…
  · the status is anything other than "delivered" and the response
    doesn't explain the next expected step
  · the response contradicts the agent's tool-call evidence
```

This works because:

- Each rule is **independently checkable** — "did the response state the current status? yes / no"
- Each rule names an **observable behavior** in the transcript, not a quality
- A different reviewer applying these rules would reach the same verdict on most scenarios

A useful test: **could a mechanical check catch this failure?** If the thing you're checking is which route the session took or which tools the agent called, you may not need an LLM Judge at all — the **Structured Match** family (**Routing match** and **Tool match**) is the right fit; see [Evaluators](/concepts/evaluators). If the judgment really does need natural-language reasoning, keep it in rules — but make each rule as observable as the problem allows.

### One criterion per rule

Resist compound rules. "Passes when the response is accurate and friendly and ends with a next step" hides three criteria in one line — when the judge fails it, you can't tell which criterion tripped, and the disagreement table (below) gets much harder to read. Split it: three rules, each independently checkable. The rule lists are ordered and you can add as many as you need.

## Pick the score type

**Pass / fail** is the easiest to calibrate. Reviewers tend to agree on binary judgments; judges are most consistent at them. Start here.

**1–5 scale** is more nuanced — useful when pass and fail hide detail you care about (1 = "actively harmful," 5 = "fully resolves it," 3 = "technically correct but blunt"). You anchor the scale at three points — **Rate 5 when…**, **Rate 3 when…**, **Rate 1 when…** — each anchor a single sentence. Scores of 2 and 4 have no anchor text of their own; they land between yours, so write the three anchors as clearly distinguishable behaviors. A **Pass threshold** control (≥2 / ≥3 / ≥4 / ≥5, default **≥3**) converts the score into the pass/fail verdict the rest of the product reports: scores at or above the threshold count as a pass. Cost: reviewer agreement on a scale is markedly lower than on binary, so calibration takes longer.

**Rule of thumb:** start pass/fail. Move to the 1–5 scale only when you can name the specific gradations the binary view is collapsing — and write those gradations into the anchors.

## Draft with Tailor with AI

On the evaluator create form, the **Tailor with AI** box drafts the rules for you: describe what you want to check in one sentence under **Describe what this evaluator should check**, click **Draft for my agent**, and draft pass/fail rules (or scale anchors) prefill the rule lists. Nothing is saved until you save — the draft is a starting point for review, not a finished rubric.

Treat the draft the way you'd treat a colleague's first pass:

- **Make it yours.** Drafts are only as specific as your one-line description. Feed it your agent's actual constraints ("never promises a refund it can't authorize") rather than a generic goal ("is helpful").
- **Delete rules you can't observe.** If a drafted rule isn't checkable against the transcript, cut it or rewrite it.
- **Then calibrate.** A drafted rubric has earned exactly as much trust as an untested hand-written one — none yet.

## When to turn on Reason step by step

The **Reason step by step** toggle makes the judge think through each criterion before deciding, instead of returning a snap verdict. It's slower, but better at catching subtle failures like tool order or arguments. Turn it on when your rules reference *how* the agent did something (which tools, in what order, with what arguments) rather than just *what* the final response says — and leave it off for simple response-content checks, where it mostly adds latency.

## Calibrate — the loop that makes the rules trustworthy

Authoring is half the work. The other half is proving the judge agrees with you, on the **Calibrate** tab of the evaluator's panel (LLM Judge evaluators only — Structured Match has nothing to calibrate). A judge is only useful if its verdicts agree with yours, and human verdicts are the ground truth here — your verdict grades the judge; it never overrides the automated result (see [Annotations](/concepts/annotations)).

<Note>
  *Ask the Trust Agent:* "Update my Policy Quote Accuracy evaluator's rules and calibrate it against the scenarios I've marked." It proposes the rule edit for approval and follows it with a calibration check, so you see the agreement move in the same conversation.
</Note>

The loop:

1. **Mark scenarios pass or fail yourself.** Pick 10–20 spanning both passes and fails — don't only mark the easy cases. If nothing is marked yet, the Calibrate tab's empty state points you at the exact place to do it.
2. **Run Check agreement.** The **Check agreement** section runs the *exact production judge* — same model, same prompt, same parser as a real evaluation run — against your marked scenarios. Calibration samples at most 20 scenarios per check and takes about a minute.
3. **Read the agreement headline.** You get a plain-language agreement rate with a verdict breakdown. The vocabulary is *agreement*, deliberately — no F1 or precision to decode.
4. **Work the disagreements.** The **Row by row — disagreements first** table puts every scenario where you and the judge split at the top. The disagreements are the data — they show exactly where the rules are wrong.
5. **Tighten the rules and save.** Rule edits commit with **Save grading rules**. Once you've edited, the last result shows a stale-rubric warning — the agreement on screen reflects the *old* rules until you re-check.
6. **Re-check and repeat.** **Calibration history** keeps every check, newest first, so you can watch agreement climb as the rules tighten.

Don't chase 100%. Perfect agreement on a small set usually means the rules have over-fit those specific scenarios. A well-calibrated rubric tolerates a small disagreement rate — and on a healthy rubric, the remaining disagreements are typically the genuinely ambiguous cases a human reviewer would also have flagged for discussion.

## Common pitfalls

<AccordionGroup>
  <Accordion title="The 'agree with everything' judge">
    **Symptom:** the judge passes nearly every scenario, including ones you'd clearly fail.

    **Common causes:**
    - The pass rules are vibes ("the response is reasonable") — the judge can always find a reading that satisfies them
    - There are no **Fails when…** rules at all, so nothing names what failure looks like

    **Fix:** rewrite each pass rule as an observable behavior, and add explicit fail rules for the failures you've actually seen. The judge's framing and temperature are already pinned by the product — leniency here is always a rules problem.
  </Accordion>

  <Accordion title="The 'over-reject' judge">
    **Symptom:** the judge fails scenarios you'd pass.

    **Common causes:**
    - A fail rule is broader than you intended ("fails when the response is incomplete" — incomplete by what standard?)
    - You're grading against an implicit requirement the rules never state (the agent had to close with an offer to help — but no rule says so)

    **Fix:** re-read the rules as if you'd never seen the responses. Every requirement you actually hold should be its own explicit rule; everything else should be deleted.
  </Accordion>

  <Accordion title="The 'flaps' judge">
    **Symptom:** the same scenario scores differently across checks.

    **Cause:** ambiguous rules. The judge runs at temperature 0 with a fixed model and prompt, so the model settings are never the variable — the usual culprit is rules loose enough that the judge picks a different interpretation per call.

    **Fix:** find the flapping scenarios in the disagreements table, identify which rule they hinge on, and tighten it until two careful humans reading it would agree.
  </Accordion>

  <Accordion title="The 'category-blind' judge">
    **Symptom:** a whole category of responses (e.g., "agent returned an error") all score the same way, ignoring real differences within the category.

    **Cause:** no rule accounts for the category, so the judge applies the same default verdict across it.

    **Fix:** add an explicit rule for the category. "Fails when the agent returns an error, unless the error message correctly identifies that the customer's input was missing required information."
  </Accordion>
</AccordionGroup>

## Two worked examples

### Example 1 — Order-status accuracy (pass/fail)

**Use case:** evaluating a customer-support agent's order-status responses.

```
Passes when…
  · the response identifies the order ID from the customer's message,
    or correctly asks for it when the customer didn't provide one
  · the response states the identified order's current status, not a
    generic acknowledgement
  · if the status is anything other than "delivered," the response
    explains the next expected step (estimated delivery date, carrier,
    or an action the customer needs to take)

Fails when…
  · any of the above is missing or incorrect
  · the response contradicts the agent's tool-call evidence
```

This calibrates well because each rule is testable independently, failure is explicitly named, and nothing depends on an unstated quality bar.

### Example 2 — Tone and completeness (1–5 scale)

**Use case:** evaluating whether an agent's *style* matches your brand voice. Note this is intentionally a *style* rubric, not a *correctness* rubric — combining the two in one Evaluator is an anti-pattern.

```
Rate 5 when… the response is friendly, complete, jargon-free, and ends
  with a clear next step or an offer to help further.

Rate 3 when… the response is factually adequate but flat in tone, or
  skips a sub-question the customer asked.

Rate 1 when… the response is off-tone (sarcastic, condescending,
  jargon-heavy) or so incomplete it fails to answer.
```

Set the **Pass threshold** to **≥3** so an adequate-but-flat response still passes while tone regressions below it fail. Two separate Evaluators (this style one plus the accuracy one from Example 1) beat one combined rubric, because regressions in tone don't drown out regressions in accuracy.
