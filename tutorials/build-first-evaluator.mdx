This tutorial continues directly from [Your first 10 minutes in Trust AI](/tutorials/quickstart). You'll author your first [LLM Judge](/concepts/evaluators) — a plain-English rubric scored by Claude or GPT at evaluation time — and **calibrate** it against the [Annotation](/concepts/annotations) you applied in the quickstart. By the end you'll have an Evaluator that agrees with you on the cases you care about.

The whole walkthrough should take about 15 minutes. Most of that time is **iterating on the rubric**, which is also the most important skill you'll develop as an Evaluator author.

<Tip>
  **Why LLM Judge first?** It's the family that does the work of qualitative judgment — politeness, completeness, accuracy of unstructured text — which is most of what real agents need to be evaluated on. [Structured Match](/concepts/evaluators#anatomy-of-a-structured-match) is the right pick when judgment is exhaustively specified (intent classification, tool-call shape). Most teams need both eventually; start here.
</Tip>

## Before you start

You should have, from [the quickstart](/tutorials/quickstart):

- A [Project](/concepts/projects) with a connected runtime (or demo data)
- At least one Session with an [Annotation](/concepts/annotations) on a Turn — ideally a pass/fail with a short freeform note explaining why
- A starter [Dataset](/concepts/datasets) with that annotated Session as a row

If you skipped any of that, go back — calibration only works against real Annotations.

## The five steps

<Steps>
  <Step title="Pick a Dataset to anchor the Evaluator">
    Open your Project, navigate to **Datasets**, click the Dataset you created in the quickstart ("Order-status happy path" or similar).

    The rows in this Dataset are what your Evaluator will be calibrated against. You don't need many — even one well-annotated row is enough to write a draft rubric. Three to five is the sweet spot for first-pass calibration; you can grow the set later.

    **You should see:** the Dataset detail view with at least one row, the row showing the original Session content + your Annotation.
  </Step>

  <Step title="Open the Evaluator builder">
    From the Project home, navigate to **Evaluators → New Evaluator**. Select **LLM Judge** as the evaluator kind, then choose **pass/fail** as the score type.

    The create flow walks you through:

    - **Rubric editor** — system prompt, criteria, and output format
    - **Score type** — pass/fail (binary) or 1–5 scale. Start with pass/fail; it's the easiest to calibrate.
    - **Calibration** — pick a Dataset and see live agreement with Annotations as you edit the rubric

    **You should see:** the rubric editor with placeholder text, and a dataset selector for calibration.
  </Step>

  <Step title="Write a plain-English rubric">
    The rubric has four parts. Here's a working example for the order-status flow we curated in the quickstart:

    **System prompt:**

    ```
    You are reviewing a customer-support agent's response to a user asking
    about order status. You are judging whether the agent's response
    correctly resolved the user's question.
    ```

    **Criteria:**

    ```
    A passing response must:
    (a) Correctly identify the order ID from the user's message
    (b) Provide the actual current status of that order (not a generic
        "we'll check on it")
    (c) If the status is anything other than "delivered," explain what
        the status means and the expected next step

    Mark "fail" if any of (a), (b), or (c) is missing or wrong.
    Mark "pass" only if all three are present and accurate.
    ```

    **Scoring scale:** Binary (`pass` / `fail`). Binary is the easiest to calibrate; once it's working, you can revisit Likert if you need nuance.

    **Output format:** JSON with `{ "score": "pass" | "fail", "reasoning": "..." }`. The builder fills this in for you; you don't need to author it from scratch.

    <Tip>
      **The most common rubric-writing mistake** is writing criteria as "is this good?" — a vague yes/no. Good criteria are testable: "did the agent (a)... (b)... (c)..." with each clause individually checkable. If a thoughtful human reviewer couldn't apply your rubric and produce the same score twice, the LLM Judge won't either.
    </Tip>

    Click **Save draft**.
  </Step>

  <Step title="Calibrate against your Annotations">
    In the calibration panel on the right, pick the Dataset from step 1. The Evaluator runs against every row in the Dataset and shows you, per row:

    - The Evaluator's score (pass/fail) + the reasoning it captured
    - The human Annotation on that row
    - Whether they agree

    **You should see:** an agreement rate at the top (e.g., `4 / 5 rows agree — 80%`), and per-row detail below.

    Look at the rows where they **disagree** — that's where calibration happens. Common patterns:

    - The Evaluator passed a row a human failed → rubric is too lenient. Add a tightening clause.
    - The Evaluator failed a row a human passed → rubric is too strict. Loosen or remove an over-specified clause.
    - Both pass or both fail with different reasoning → calibration is roughly aligned; you can move on or refine further.

    Edit the rubric, click **Re-calibrate**, and watch the agreement rate move. **The goal is ~90% agreement** for a binary rubric on a small set. Don't chase 100% — perfect agreement on a small Dataset often means you've over-fit the rubric to the specific rows.

    Iterate. This is the work.
  </Step>

  <Step title="Save and reuse">
    Once agreement is high, click **Save Evaluator**. Give it a name (e.g., "Order status — three-part rubric"), an optional description, and confirm.

    The Evaluator is now in your Project's Evaluator library. Any future [Evaluation](/concepts/evaluations) — against any Dataset, against any Agent Version — can apply this Evaluator. It's reusable; that's the whole point.

    Or via the API:

    <CodeGroup>
      ```bash curl
      curl -X POST https://api.provar.dev/v1/evaluators \
        -H "Authorization: Bearer $TRUST_AI_TOKEN" \
        -H "Content-Type: application/json" \
        -d @evaluator.json
      ```

      ```typescript TypeScript
      const evaluator = await client.evaluators.create({
        project_id: project.id,
        name: "Order status — three-part rubric",
        type: "llm_judge",
        system_prompt: "You are reviewing a customer-support agent...",
        criteria: "A passing response must: (a)... (b)... (c)...",
        scale: "binary",
      });
      ```
    </CodeGroup>

    **You should see:** the Evaluators list now shows your new Evaluator, ready to apply.
  </Step>
</Steps>

## When Structured Match would have been a better pick

If the judgment you're encoding is **exhaustive and precise** — "did the agent emit `intent: order_status`?" — Structured Match is the right family, not LLM Judge. A Structured Match for that case would be:

```yaml
path: output.intent
operator: equals
expected: order_status
```

Three lines. No calibration. No LLM cost per evaluation. **If you find yourself trying to encode something exact into a plain-English rubric, switch families.** See [LLM Judge vs Structured Match](/concepts/evaluators#the-two-families) for the full decision heuristic.
