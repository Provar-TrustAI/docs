A Session is one real conversation between a user and an agent — every message, tool call, and intermediate step captured as-it-happened. Sessions flow into Trust AI from your connected agent runtime and become the raw material everything downstream depends on: Annotations, Datasets, Evaluations, regression comparisons.

## What's in a Session

A Session is a tree of nested structures:

- **The Session itself** — top-level metadata (timestamps, user ID, Agent Version, runtime source, intent classification if available)
- **Turns** — each message exchange. A Turn is the smallest unit an Annotation or Evaluator can attach to.
- **Tool Calls inside a Turn** — function name, arguments, return value, latency. Often the most diagnostic data in a Session.
- **Skill activations** — when the agent invoked a known capability (`refund_check`, `lookup_order_status`, etc.). Useful for evaluating coverage and routing.

Think of a Session as a fully-resolved trace: not just what the user said and what the agent said back, but every decision the agent made on the way to its response.

## Where Sessions come from

Trust AI ingests Sessions from the agent runtime configured at the **[Project](/concepts/projects)** level. Trust AI supports AWS Bedrock AgentCore today; Salesforce Agentforce and Microsoft Copilot Studio connectors are on the roadmap.

The ingestion path for AgentCore:

1. Your AgentCore Runtime emits invocation logs to CloudWatch (built-in).
2. Trust AI's ingestion pipeline pulls those logs via a cross-account IAM role you grant during [Connect an AWS Bedrock AgentCore agent](/how-to/connect-agentcore).
3. Each invocation becomes a Session row in your Project.

Ingestion is read-only — Trust AI never modifies the source runtime or its logs.

## Sessions vs Traces — same data, different name?

If you've used observability tools (Langfuse, Phoenix, LangSmith) you may have heard the data called a "trace." Trust AI deliberately uses **Session**:

| Term | Implies |
|---|---|
| Trace | Engineering observability — debugging a single execution |
| Session | A complete user interaction — a unit of agent behavior worth judging |

The distinction matters because the workflows differ. Traces are for engineers reading logs. Sessions are for **QA Practitioners** building Datasets and **Agent Owners** judging behavior. Both are valid lenses; Trust AI deliberately picks the term that maps to its core users.

## Session metadata you can filter on

Out of the box, every ingested Session carries:

- Source: runtime (AgentCore, Salesforce, etc.), Agent Version, runtime-specific IDs
- Time: ingestion timestamp, original invocation time
- Outcome: response status (success, error, partial), latency, total tokens
- Routing: intent classification (if your agent emits one), skill activations
- Annotations: any human signal already attached
- Dataset memberships: which Datasets this Session has been curated into

Custom metadata your agent emits also surfaces — anything in the AgentCore invocation's `attributes` block flows through to the Session.

## Working with Sessions

Everything you do with a Session happens from the Sessions surface or by asking the Trust Agent. You list and filter Sessions in the table, open a row to read the full detail with Turns and Tool Calls, attach an Annotation, curate a Session into a Dataset, or replay it against a new Agent Version — all without leaving the UI. You can also ask the Trust Agent to do any of these for you, for example "show me failed sessions from yesterday" or "add these sessions to the regression dataset."

Developers who want the underlying endpoints can find full request and response shapes in the [API Reference](/api-reference).
