An Annotation is a piece of **human judgment** attached to a [Session](/concepts/sessions). You record it in typed **annotation columns** on the Sessions surface — a pass/fail verdict, a severity, a score, tags, or a reviewer note — and it becomes data the rest of Trust AI can use. The human **Verdict** is the ground truth every [Evaluator](/concepts/evaluators) is measured against: it grades the judges.

Without Annotations, Evaluator scores are unaccountable — the app can tell you what its judges decided, but not how often those judges agree with a person.

## The annotation column model

Annotations live in a project-wide schema of typed columns. Each column holds one value per Session, and every project starts with three seeded columns:

- **Verdict** — a Pass / Fail judgment of the whole Session. This column is reserved: you can't delete it or change its type, because it is the human ground-truth channel that evaluator accuracy is measured against.
- **Severity** — a single choice among **minor**, **major**, and **blocker**.
- **Reviewer note** — a free-text comment.

You add your own columns from the Sessions table's add-column menu with **New annotation column**. Five types are available:

| Type | Holds |
|---|---|
| **Pass / Fail** | A binary verdict |
| **Score** | A numeric rating (1–5) |
| **Category** | A single choice from options you define |
| **Tags** | Multiple tags from a palette |
| **Comment** | A free-text note |

Annotations are **Session-level**: each value judges the conversation as a whole. There is no per-Turn annotation — if a specific exchange is what matters, say so in a **Reviewer note** or a **Comment** column.

## Where you annotate

Annotation happens where the Sessions already are — there is no separate review surface:

- **Inline in the Sessions table.** **Verdict** and **Severity** cells are editable directly in the table, so a reviewer can sweep down the list without opening anything. The Verdict column shows by default; other annotation columns can be shown from the column menu.
- **In the Session fly-in.** The fly-in's **Annotations** tab is the full schema-driven editor for every column, alongside the **Transcript**, **Evaluations**, and **Details** tabs. The same panel appears when you drill into a session from a scenario. An edit made in a table cell repaints any open panel for that Session, so the two never disagree.

To work through Sessions in volume, filter the list to the slice you care about — by Source, tag, or agent version — and save the filter as a [saved view](/concepts/saved-views). Selecting multiple Sessions raises the bulk-actions bar (**Evaluate** · **Tag** · **Download**); **Tag** applies tags from a vocabulary shared across Scenarios and Sessions, with autocomplete.

## Two verdicts, one rule

A Session can carry two verdicts at once, and they never overwrite each other:

- The **automated verdict** — the three-state pill (**Pass** / **Fail** / **Not evaluated**) produced by [Evaluations](/concepts/evaluations). Only automated results set it.
- The **human Verdict annotation** — your Pass / Fail judgment in the reserved Verdict column.

The rule: **your Verdict grades the judges, and never overrides the automated verdict.** When a Session carries both, Trust AI compares them, and the agreement rate surfaces as the **Evaluator accuracy** headline on evaluation results — how often the automated judges reached the same conclusion a person did. A Session where the two disagree is exactly the signal worth investigating: either the agent's behavior is subtle, or the Evaluator's rubric needs work.

## Annotations vs Evaluator scores — different things, paired

Annotations and Evaluator scores are structurally similar — a judgment attached to a Session — but they come from different sources and answer different questions:

| | Annotation | Evaluator score |
|---|---|---|
| **Source** | Human reviewer | LLM Judge or Structured Match |
| **Question answered** | "Was this actually acceptable?" | "Does this pass the rubric, at scale?" |
| **Cost per Session** | Reviewer minutes | LLM tokens / compute |
| **When applied** | During review, any time after capture | At Evaluation time, programmatically |
| **Role** | Ground truth for evaluator accuracy | Sets the automated verdict |

The relationship is **calibration**: Evaluator scores earn trust to the degree they agree with human Verdicts on the same Sessions. That agreement is what the calibration surfaces on [Evaluators](/concepts/evaluators) measure.

The common annotation patterns:

## What annotated data feeds

Annotations justify their cost downstream:

1. **Evaluator accuracy** — every human Verdict on an evaluated Session sharpens the judge-vs-human agreement number on evaluation results. This is the primary payoff: it tells you whether to trust the automated scores.
2. **Evaluator calibration** — when accuracy is low, the disagreeing Sessions show you exactly where the rubric misjudges. Refine the Evaluator and re-run.
3. **Review-driven evaluation** — filter to the Sessions your review flagged, select them, and hit **Evaluate** to grade them with your judges. Annotation is not a prerequisite: you can evaluate Sessions you haven't annotated at all.

All three close the loop described in [the evaluation loop](/concepts/evaluation-loop) — Annotations are what keep the automated half of that loop honest.

## Working with Annotations

Annotating is one of the few genuinely hands-on tasks in Trust AI — the judgment is the point, so no agent does it for you. You can ask the [Trust Agent](/concepts/trust-agent) to find the Sessions worth reviewing ("which sessions from this week have no verdict yet?"), then grade them yourself inline or in the fly-in.

Earlier releases attached annotations to entries in the legacy dataset container; those values survive on the wire through the legacy API, but the Sessions annotation columns are the product's annotation surface. For reading or writing annotation columns and values programmatically, the endpoint details live in the [API Reference](/api-reference).
