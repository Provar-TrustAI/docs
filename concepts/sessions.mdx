A Session is one complete conversation between a user and an agent — every message, tool call, and intermediate step captured as it happened. Sessions are the evidence layer of Trust AI: [Annotations](/concepts/annotations) grade them, [Evaluations](/concepts/evaluations) can run directly against them, and evaluator accuracy is measured against the human verdicts you record on them. [Scenarios](/concepts/scenarios) are where you author what to test; Sessions are where you review and grade what actually happened.

## What's in a Session

A Session is a tree of nested structures:

- **The Session itself** — top-level metadata: its Source, timestamps, the Model and agent version it ran against, and where it was simulated from.
- **Turns** — each message exchange between the user and the agent, with an optional response-latency readout per turn.
- **Tool calls inside a Turn** — function name, arguments, return value. Often the most diagnostic data in a Session.
- **Agent steps** — the trace between question and answer: **Thinking** steps show the model's reasoning in plain language, and **Model call** steps summarize each underlying model request ("Model call · 1,180 in / 142 out") with the raw payload behind an opt-in **Input** / **Output** disclosure. Reasoning is provider-dependent — where a runtime can't emit it, the transcript simply shows no Thinking steps rather than a blank frame.

Think of a Session as a fully-resolved record: not just what the user said and what the agent said back, but every decision the agent made on the way to its response. Annotations and evaluation results attach to the Session as a whole — you grade the conversation, not an individual Turn.

## Where Sessions come from

Sessions have three origins, and every session wears a **Source** chip that tells you which:

| Source | How the session was created |
|---|---|
| **Scenario** | Simulated from a scenario — by running it, generating sessions from it, or evaluating it. The session links back to the scenario it came from. |
| **Playground** | A [Playground](/concepts/playground) conversation you didn't save as a scenario — saving one flips its Source to **Scenario**. |
| **External** | Synced from a connected agent runtime — a real conversation your agent already had. |

A session linked to a scenario always shows **Scenario**, and its **Details** tab carries a **Simulated from** card naming the scenario and persona behind it.

Session IDs reflect origin too: sessions generated in-product get `RUN-` ids (`RUN-000014`), synced external sessions get `SES-` ids (`SES-000013`). Both draw from one per-project counter, and old links keep resolving either way.

### Connected runtimes

External sessions sync from the agent platforms connected in your [Project](/concepts/projects): AWS Bedrock AgentCore, Claude Managed Agent, and Salesforce Agentforce. For AgentCore, your runtime emits invocation logs to CloudWatch and Trust AI's pipeline pulls them through a cross-account IAM role you grant during [Connect an AWS Bedrock AgentCore agent](/how-to/connect-agentcore); each invocation becomes a Session in your Project.

**Sync now** in the Sessions header opens a picker of your project's connected agents, grouped by platform — pick one to sync that agent on demand instead of waiting for the next scheduled pull. Ingestion is read-only: Trust AI never modifies the source runtime or its logs.

## In the app

The Sessions list is where you review, grade, and evaluate captured conversations. The table and panel behaviors shared across list surfaces — sorting, the column menu, range selection, the keyboard contract — are described in [Tables and panels](/concepts/tables-and-panels); here is what the surface offers:

- **Sortable, filterable table** — the ID column shows the readable `RUN-` / `SES-` id, **Source** shows the origin chip, **Routed To** shows where the agent routed the conversation, and annotation columns (starting with a default **Verdict**) sit alongside the session data. Filter configurations persist as [saved views](/concepts/saved-views) your team can share.
- **Typed annotation columns** — a default **Verdict** (pass / fail) column ships on every project, and **+ New annotation column** adds custom columns typed **Pass / Fail**, **Score**, **Category**, **Tags**, or **Comment**. Verdict and severity cells edit inline, right in the table.
- **Column control** — one column menu handles show/hide and drag-to-reorder, and the order is remembered per saved view. Shift-click a second checkbox to select a whole range of sessions at once.
- **Bulk actions** — selecting sessions raises a bar with **Evaluate** (the primary action), **Tag**, and **Download** (CSV export of the selection).
- **Fly-in detail panel** — click any session to open a panel with four tabs: **Transcript** (default), **Annotations**, **Evaluations**, and **Details**. The header carries a one-off **Evaluate** button, and the overflow menu offers **Download transcript** and **Copy session ID**.
- **Keyboard navigation** — <kbd>J</kbd> / <kbd>K</kbd> cycles through sessions, <kbd>F</kbd> toggles fullscreen, <kbd>Esc</kbd> exits fullscreen first and then closes the panel.
- **Details tab** — the session's Source, **Simulated from** provenance, **Model**, agent version, and **Routed to** route in one place.
- **Transcripts are immutable** — there is no Delete Session action, and a session's conversation can't be edited. What you *can* write are the annotation values and tags layered on top; tags autocomplete from one vocabulary shared with Scenarios.

## What you do with a Session

The four primary workflows:

Earlier releases had you curate sessions into datasets and replay them against new agent versions. That workflow is retired: regression testing now runs through [Scenarios](/concepts/scenarios), which simulate fresh conversations against the [Agent Version](/concepts/agent-versions) you pick at evaluation time — and Sessions became something you evaluate directly, in place.

## Sessions vs traces — same data, different name?

If you've used observability tools (Langfuse, Phoenix, LangSmith) you may have heard the data called a "trace." Trust AI deliberately uses **Session**:

| Term | Implies |
|---|---|
| Trace | Engineering observability — debugging a single execution |
| Session | A complete user interaction — a unit of agent behavior worth judging |

The distinction matters because the workflows differ. Traces are for engineers reading logs. Sessions are for **QA Practitioners** grading behavior and authoring scenarios, and **Agent Owners** judging results. Both are valid lenses; Trust AI deliberately picks the term that maps to its core users.

## Session metadata you can filter on

Out of the box, every Session carries:

- Origin: the **Source** chip (Scenario / Playground / External) and the readable `RUN-` / `SES-` id
- Provenance: the linked scenario and persona (for simulated sessions), Model, and agent version
- Time: when the conversation happened and when it was captured
- Routing: the **Routed To** route, plus tool calls made along the way
- Human signal: annotation column values (Verdict, severity, and any custom columns) and tags
- Automated signal: the session's evaluation verdict — **Pass**, **Fail**, or **Not evaluated**

## Working with Sessions

Everything you do with a Session happens from the Sessions list or by asking the [Trust Agent](/concepts/trust-agent). You filter the table, open a session to read its transcript, set a Verdict inline, tag it, or evaluate it — all without leaving the surface. You can also ask the Trust Agent to find and read sessions for you — "show me failed sessions from yesterday" — and evaluating them is launcher-driven from the list or the panel (see [Evaluate sessions](/how-to/evaluate-sessions)).

Developers who want the underlying endpoints can find full request and response shapes in the [API Reference](/api-reference).
