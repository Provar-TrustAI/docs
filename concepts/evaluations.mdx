An Evaluation is a run: take one or more [Evaluators](/concepts/evaluators), apply them to the [scenarios](/concepts/scenarios) or [Sessions](/concepts/sessions) you select, against a specific [Agent Version](/concepts/agent-versions). The output is a scored, reviewable record per result — a verdict alongside the reasoning the Evaluator captured.

An Evaluation is **the artifact regressions get caught in**. It's also the artifact every other Trust AI surface ultimately serves: Sessions exist to become scenarios and evaluation targets, Annotations exist to grade the judges, scenarios and Evaluators exist to produce Evaluations.

## What an Evaluation can grade

An Evaluation targets one of two sources:

- **Scenarios** — Trust AI simulates each selected scenario live against the Agent Version (via the [Project's runtime connection](/concepts/projects)). Every persona attached to a scenario is simulated and scored on its own, so one scenario with three personas produces three results.
- **Sessions** — Trust AI grades conversations that already happened. The judges read the captured transcript; no agent is invoked and the session is never modified. Because there's no live conversation to steer, session grading uses LLM Judge evaluators only. The result is homed on the session itself, and an **Open session** button on the result takes you back to the exact session that was graded.

The second source is what makes Evaluations more than a synthetic-test tool: yesterday's real production conversations are as gradeable as your authored test cases.

## Launching a run

One **Evaluation Launcher** drives every run. It opens from five places:

1. **Sessions list** — select sessions and click **Evaluate** (bulk).
2. **A single session's panel** — **Evaluate** in the header (one-off).
3. **Scenarios list** — select scenarios and use the **Run** split button (**Run & evaluate**, or **Run only** to simulate without grading).
4. **A single scenario's fly-in** (one-off).
5. **Evaluations list** — click **New evaluation** and pick the source yourself: Sessions or Scenarios, then a [saved view](/concepts/saved-views) of that surface to scope the run.

However you arrive, the launcher shows a **fan-out preview** — exactly what will run, including the per-persona expansion for scenarios — before you commit.

One launcher detail worth knowing:

- **Re-run** — on any finished run, **Re-run** reopens the launcher pre-seeded with the original run's full configuration: same targets, same Evaluators, same Agent Version. Change one variable (usually the Agent Version) and launch.

## One-off evaluation in place

You don't need a named run to answer "does this one pass?". Evaluating a single scenario or session grades it in place: the verdict lands directly on the item, and no run is added to the Evaluations list. Bulk runs are for measurement; one-off evaluation is for the quick check while you're already looking at the thing.

## The run lifecycle

- **Launched** — the run records a snapshot of your scenarios as they were at launch, so a finished run stays reproducible even after you edit or delete scenarios later.
- **In progress** — results fill in as they complete. If you launched a bulk run from the Scenarios list, you stay on the list and watch each scenario's live **Generating…** badge settle as it finishes. You can open the run and watch results arrive, or click **Cancel run** — the run stops at a clean boundary and keeps every result already scored.
- **Complete** — every result has a verdict, and the aggregate headlines resolve.

Finished runs offer **Export** (download the results as CSV) and a primary **Re-run**; the **…** menu offers **Delete run**, which removes the run from your lists without destroying the sessions it produced.

## Reading the results

Run detail leads with **two headline numbers**, and they answer different questions:

- **Pass rate** grades the *agent*: of the results that count, how many passed.
- **Evaluator accuracy** grades the *judges*: how often the automated Evaluators agreed with a human's verdict on the same sessions. Human Pass / Fail verdicts, recorded as [Annotations](/concepts/annotations), are the ground truth here — a human verdict grades the judges, and never overrides the automated verdict.

A high pass rate means little if evaluator accuracy is low: you'd be trusting judges that a human reviewer routinely disagrees with. The two numbers together tell you whether to believe the run.

### Pass^K reliability

A run whose test cases were attempted more than once leads with a **Pass^K** rate instead of a plain pass rate — the share of test cases that passed *every* attempt, rendered literally as `Pass^2`. Hover the headline for the full curve from pass-once to pass-every-time. This is the honest number for non-deterministic agents: an agent that passes 87% of cases on one attempt can pass far fewer on all four. At this release the launcher runs each test case once — repeated-trial runs render Pass^K when present, but the repeat selector isn't exposed on the launcher.

### Per-persona results

Because every attached persona is scored separately, run detail includes a per-persona matrix with plain-English divergence hints — when the same scenario passes for one persona and fails for another, the matrix points at it. That divergence is usually the most actionable finding in a run: the agent didn't break everywhere, it broke for a *kind of user*.

### Verdicts are three-state

Every verdict is **Pass**, **Fail**, or **Not evaluated**. The third state is honest bookkeeping — a result that no Evaluator scored is not silently a pass or a fail.

## When it's not the agent's fault

Not every non-pass means the agent got something wrong, and the results no longer pretend otherwise:

- **Guardrail blocked** — the runtime's guardrails blocked the response before the agent's answer could be judged. The result carries an amber **Guardrail blocked** label and is excluded from the pass-rate denominator.
- **Environment problem** — the runtime, the harness, or a judge call failed for infrastructure reasons. The result renders as a neutral **Environment problem — not the agent · Excluded from the score** state, never a red Fail.

The pipeline also defends itself before classifying anything as an environment problem: model-capacity throttling is retried automatically with backoff, one bad result can't fail the whole run, and judge calls run in parallel within a safe bound. A capacity hiccup at 2 a.m. can no longer masquerade as a drop in agent quality.

## What's in an Evaluation result

Open any result to see:

- **Transcript** — the full conversation, labeled **Simulated** (the run drove a live conversation) or **Replayed** (the judges graded a captured session).
- **Final verdict** — the three-state verdict, with per-Evaluator score cards beneath it.
- **Reasoning** — for LLM Judges, the rubric reasoning that led to the score. For Structured Match, the assertion that fired (or didn't).
- **Open session** — jump to the exact session this result was graded against.

## Evaluation results are Sessions, too

The conversations produced when an Evaluation simulates scenarios are themselves Sessions — structurally the same as production Sessions. They appear in your Sessions list with a **Source** chip of **Scenario** (captured production traffic shows **External**; unsaved Playground conversations show **Playground**, and saving one as a scenario flips it to **Scenario**), and their **Simulated from** provenance shows which scenario and persona produced them.

This is what unlocks:

- **Re-grading without re-running** — evaluate those sessions again with different Evaluators; the judges read the transcript, no agent invocation needed.
- **Annotating them** like any other session, which feeds the evaluator-accuracy headline.
- **No special-case surface** — one Sessions list, filtered by Source when you want only one kind.

## Comparing Agent Versions

The regression-tracking pattern is two runs with one variable changed:

1. Run an Evaluation against **Agent Version A** (say, production).
2. On the finished run, click **Re-run**, switch the Agent Version to **B** (the candidate), and launch. Everything else — targets, Evaluators, repeat count — carries over faithfully.
3. Read the two run details side by side: headline deltas first, then the per-persona matrices for where the movement happened.

There is no dedicated run-vs-run comparison view in the app today; the faithful Re-run is what makes the manual comparison trustworthy, because you know the only variable that changed is the agent.

## Working with Evaluations

You drive Evaluations from the Trust AI app, or by asking the [Trust Agent](/concepts/trust-agent). Tell the Trust Agent to evaluate your scenarios against an Agent Version — or click **New evaluation** on the Evaluations list and configure the launcher yourself. Evaluating captured sessions is launcher-driven: start it from the Sessions list or a session's panel (see [Evaluate sessions](/how-to/evaluate-sessions)). Open a run to watch progress, read results, cancel it mid-flight, or re-run it against a different Agent Version. The Trust Agent can also pull aggregate stats and slice results by persona for you.

Developers who want the underlying endpoints and request/response shapes can find them in the [API Reference](/api-reference).
