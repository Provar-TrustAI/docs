The [Trust Agent](/concepts/trust-agent) is the in-app assistant that helps you do evaluation work on a project. When it wants to read your data it just streams the answer — but when it wants to **write** (create, update, delete, or run a record) it stops and asks you first. This is a recipe for that moment: recognizing the card, reading the write preview, and approving so the change actually executes.

<Note>
  The Trust Agent is the in-app assistant, not an [Agent Version](/concepts/agent-versions) (the agent under test). The two share the word "agent" but are different concepts — this page is about approving writes the *in-app assistant* proposes.
</Note>

This how-to picks up where [Work with the Trust Agent](/how-to/work-with-the-trust-agent) leaves off — that page covers driving the agent and the clarify-first flow. Here, you'll learn to handle the three interrupt cards the agent uses to ask for your input:

- A **permission card** — gates a write before it runs
- A **single-question card** — one open-ended clarification
- A **multi-question flow card** — a stepped set of clarifications

After this, you'll be able to review a proposed change, approve it (and watch it land for real), reject it with a note, or answer the agent's questions — and find the resolved decision in the transcript afterward.

## How the approval gate works

When the agent decides to perform a write, it does not act silently. It surfaces a **permission card** in the place where the composer normally sits — the text input disappears and the card takes its slot. At that point the agent run **halts** at the tool call and waits: no further turns run until you decide.

Permission cards gate **writes specifically**. A read-only request — "list my sessions," "what failed in this evaluation" — streams its result straight back (often as a [clickable object table](/concepts/trust-agent)), with no card and no approval needed. The gate only appears when the agent proposes to change something.

<Warning>
  Approving a write **executes it for real**. The Trust Agent runs the approved change on the live agent run path — the mutation persists in your project; it is not a preview or a dry run. Treat every approval as a real edit to your data.
</Warning>

## The permission card

When a write is proposed, the permission card greets you with a **write preview** — not a bare record ID. Before you approve, you can see:

- **The diff** — what specifically will change
- **The dependencies** — the linked records the write would affect
- **The proposed end state** — what the record will look like after the change

The write preview always speaks in human terms — a delete shows the record by its label and lists what depends on it, so you can see the blast radius before you commit to it. To inspect the full proposed end state, choose the card's **View proposal** link: it opens a right-side panel showing the record as it would look after the change.

The preview also keeps proposing tense. Until you approve, nothing has happened — a pending delete is badged **Pending delete** (and other writes **Pending approval**), never a false "Deleted." If the panel says a record is gone but you haven't approved anything, that's a bug, not the design.

## Approve, dismiss, or reject

Once you've read the preview, you have three ways to resolve a permission card.

<Steps>
  <Step title="Approve to execute">
    Pick an approval scope, then confirm. Every card offers a **one-time approval** — approve this write, and only this write. When the agent expects to repeat the same kind of write, the card can also offer a **for-this-chat** option, worded specifically for what it would keep doing, so approving it grants a standing pass for the rest of the conversation. Delete requests never offer the chat-wide option — every delete is approved one at a time. The confirm button's label follows the scope you've selected, so read it before you click.

    Once you confirm, the agent run resumes from where it halted and performs the write — the change lands in your project. Watch the [side rail](/concepts/trust-agent) **Outputs** section: the record the agent created, updated, or ran shows up there as a row you can open in a fly-in.

    **You should see:** the card resolve, the run continue, and the affected record reflect the change when you re-open it. The write is real — re-query the record and the mutation is there.
  </Step>

  <Step title="Dismiss to decline">
    If you don't want the change, **dismiss** the card. The write does not run, and the agent waits for your next message instead of acting. A resolved row still lands in the transcript — it reads "You dismissed the request" — so the declined decision is on the record.

    **You should see:** no change to the record, and the composer return so you can redirect the agent.
  </Step>

  <Step title="Reject and tell the agent why">
    If the proposed write is close but wrong, choose **Reject — and tell me why (optional)**. An inline note field opens on the card; type your reason (or leave it blank) and choose **Send rejection**. Your reason goes back to the agent, so it can adjust rather than guess.

    **You should see:** your rejection and your reason echoed into the transcript, and the agent respond to your reasoning on its next turn.
  </Step>
</Steps>

<Note>
  A permission card is not the same as a plan confirmation. In Plan mode the agent proposes a task list and waits at a plan-confirm bar — one tap on **Approve & run** starts the work, or you can type a go-ahead in the composer and it does the same thing. That's a different gate: it approves a plan, not a write. A permission card is the per-write checkpoint, and it's what executes the change. See [Work with the Trust Agent](/how-to/work-with-the-trust-agent) for Plan mode.
</Note>

## Answer a question card

Sometimes the agent doesn't have enough to act yet. Instead of a permission card it raises a **question card** — also in the composer slot — and pauses until you answer. There are two shapes.

### A single-question card

For one open-ended clarification, you get a single-question card. It offers the **drafted choices** the agent thinks are likely, plus a write-in field ("Or type your own answer…") so you're never boxed into the offered answers. The card is dismissible if you'd rather steer the agent a different way.

### A multi-question flow card

When the agent needs several inputs, it raises a **multi-question flow card**. It's stepped: questions come one at a time with a progress row ("Question N of M"), and you can go **back** to a previous question to change an earlier answer before submitting. Nothing is sent until you submit the batch — then all your answers go to the agent together. Each question carries the same drafted choices and write-in option as the single-question card.

## Find the decision in the transcript

Every resolution — an approval, an answer, or a dismissal — is recorded back into the transcript as a **resolved row** that echoes what you chose. An approval's resolved row names the scope you granted, so a chat-wide grant is distinguishable from a one-time approval later. For a multi-question flow, the resolved row echoes each per-question answer. This is your audit trail: it persists after the turn completes, so reopening the chat later still shows what was asked and what you decided.

## A pending approval survives a reload

You don't lose the gate if your connection drops, you reload, or you close the tab before deciding. When you reopen the chat, a card that was still awaiting your answer **rehydrates as a live, answerable card** — back in the composer slot, with the composer still gated — and resolving it resumes the halted run exactly as if you had never left. Already-resolved approvals rehydrate too, in the order they happened, so a reopened chat reads the same as it did live.

**You should see:** after a mid-decision reload, the same permission or question card waiting in the composer slot; answer it and the run picks up where it halted.

## When the agent is minimized

You don't have to keep the full agent open to catch an approval. If you've collapsed the agent to its **docked panel** or **pill** when a write needs your sign-off, a pulsing **needs-you badge** appears on the shell so you know a decision is pending — and the docked view auto-expands to surface the permission card, so you can act without manually reopening the agent.

<Note>
  There may be a brief wait before a card appears. That's the model thinking — its time-to-first-token — not a hang. The agent fills the wait with a live **Thinking** timer and a working indicator rather than a dead gap, and the card paints as soon as the model reaches the write. There's no fixed timing; it depends on the model in use.
</Note>

## Accessibility

The approval flow is keyboard- and screen-reader-friendly. When a card appears, focus moves into it automatically. Resolving or dismissing a card should return focus to where it was, and the interrupt is designed to be announced to assistive technology and to meet WCAG AA contrast.
