An Evaluation is a single run: take one or more [Evaluators](/concepts/evaluators), apply them to every row of a [Dataset](/concepts/datasets), against a specific [Agent Version](/concepts/overview#agent-version-in-one-paragraph). The output is a scored, reviewable record per row — pass/fail or rubric score, alongside the reasoning the Evaluator captured.

An Evaluation is **the artifact regressions get caught in**. It's also the artifact every other Trust AI surface ultimately serves: Sessions exist to feed Datasets, Annotations exist to calibrate Evaluators, Datasets and Evaluators exist to produce Evaluations.

## The run lifecycle

Every Evaluation moves through four states:

1. **Configured** — you've picked the Dataset, the Evaluator(s), and the Agent Version. Nothing's running yet.
2. **In progress** — Trust AI is sending each Dataset row to the live agent (via the [Project's runtime connection](/concepts/projects)), collecting structured responses, scoring them with the Evaluators. Long-running; the app surfaces progress as rows complete.
3. **Complete** — every row has an Evaluator result. Aggregate stats are available (pass rate, score distribution, common failure patterns).
4. **Compared** *(optional)* — paired with a prior Evaluation against a different Agent Version, surfaces the per-row delta.

The lifecycle is observable. You can open an in-progress Evaluation and watch rows fill in.

## What's in an Evaluation result

Per Dataset row, an Evaluation captures:

- **The input** sent to the agent (from the Dataset row)
- **The agent's full response** — what it said back, including tool calls and intermediate steps. This **is itself a Session** — see "Evaluation results are Sessions, too" below.
- **Per-Evaluator scores** — one per Evaluator applied to this run
- **Reasoning** — for LLM Judges, the rubric reasoning that led to the score. For Structured Match, the assertion that fired (or didn't).
- **Reference comparison** *(optional)* — if the Dataset row has a reference output, the delta between expected and actual

At the Evaluation level:

- Aggregate pass/fail rate, score distribution, latency stats
- Failure categories (rows that failed Evaluator A but not Evaluator B, etc.)
- Comparison to prior Evaluations against the same Dataset

## Evaluation results are Sessions, too

A subtle but important pattern (ADR 0006 in the trust-ai-app repo): **the agent responses produced during an Evaluation are themselves Sessions**, indistinguishable structurally from production Sessions. They appear in your Sessions list, taggable as "from-eval," and you can re-annotate them, re-evaluate them, or compare them to production Sessions side-by-side.

This is what unlocks:

- **Re-evaluating an Evaluation's outputs with a different Evaluator** without re-running against the live agent
- **Curating the "evaluation Sessions" into a Dataset of their own** for higher-resolution regression testing
- **Treating Evaluation outputs as just another agent invocation** for downstream tooling — no special-case API surface

The trade: Sessions get noisier (you'll see both production and evaluation Sessions in lists). Tags + filters handle this; the structural simplicity is worth more than the namespace cost.

## Comparing two Agent Versions

The headline workflow Trust AI exists for. The pattern:

1. Pick the **same Dataset** and **same Evaluators**.
2. Pick **Agent Version A** (e.g., production) and **Agent Version B** (e.g., a candidate replacement).
3. Run an Evaluation against each.
4. Open the comparison view.

The comparison surfaces:

- **Per-row deltas** — rows where A passed and B failed (potential regressions), rows where A failed and B passed (improvements), rows where both passed/failed
- **Flapping rows** — rows where a non-deterministic Evaluator scored them differently (a signal the rubric needs tightening, or the test case is on the edge)
- **Aggregate-level deltas** — pass rate change, score distribution shift
- **Per-Evaluator breakdowns** — if one Evaluator regressed harder than others, that often localizes the kind of regression

This is **the regression-tracking primitive**. For the practical how-to, see [Compare two Agent Versions](/how-to/compare-agent-versions).

## Reading an Evaluation

Common failure patterns and what they usually mean:

- **A row that passed yesterday and fails today, same Evaluator, same prompt** → either the agent regressed, or the Evaluator is non-deterministic and needs tightening. Open the reasoning; the Evaluator's narrative usually tells you which.
- **A row where two Evaluators disagree** → often a calibration gap. The strict Evaluator is over-rejecting or the lenient one is over-accepting. Look at the reasoning side-by-side.
- **A category of rows that all fail together** → usually a category regression in the agent (a whole intent broke, a tool's not being called, etc.). Filter by row tag to localize.
- **A row where the agent didn't respond at all** (timeout, error) → infrastructure problem; not a judgment problem. Re-run those rows; if persistent, debug the runtime.

## API surface

Evaluations appear in the `/v1/evaluations` endpoint family of the [API Reference](/api-reference). Common operations:

- `POST /v1/evaluations` — kick off a new Evaluation run (Dataset + Evaluators + Agent Version)
- `GET /v1/evaluations/{evaluation_id}` — fetch run state + aggregate results
- `GET /v1/evaluations/{evaluation_id}/rows` — per-row scored results (paginated)
- `POST /v1/evaluations/{evaluation_id}/compare/{other_evaluation_id}` — generate the per-row comparison
- `GET /v1/evaluations` — list / filter Evaluations in a Project
- `POST /v1/evaluations/{run_id}/cancel` — cancel an in-progress Evaluation run *(v2026.05.19+)*
- `GET /v1/evaluations/stats` — aggregate Evaluation stats across the Project (pass-rate trends, latency distributions) *(v2026.05.19+)*
- `POST /v1/evaluations/cohort-breakdown` — slice an Evaluation's per-row results by cohort (intent, tag, custom dimension) *(v2026.05.19+)*

Full request/response shapes live in the auto-generated [API Reference](/api-reference).
