An Agent Version is a **pinned snapshot of the agent under test** — its prompt, model, tool set, retrieval config, and deployment target captured as a single, named, immutable reference. Every [Evaluation](/concepts/evaluations) runs against exactly one Agent Version, and every regression-tracking workflow is a diff between two.

If [Sessions](/concepts/sessions) are the raw material, [Scenario sets](/concepts/scenarios) the test bed, and [Evaluators](/concepts/evaluators) the rubric, then Agent Versions are **what's being judged**.

<Note>
  **An Agent Version is not the Trust Agent.** They share the word "agent," but they are opposite ends of the evaluation:

  - An **Agent Version** (this page) is the *thing being evaluated* — a pinned snapshot of the customer's agent under test.
  - The **[Trust Agent](/concepts/trust-agent)** is the *in-app assistant* that helps you run those evaluations.

  The Trust Agent helps you judge Agent Versions; it is never itself "a version."
</Note>

## The regression-tracking pattern

The core regression-tracking pattern:

1. Pick a stable **Scenario set**.
2. Pick the same set of **Evaluators**.
3. Pick **two Agent Versions** — the production one and a candidate replacement.
4. Run an Evaluation against each.
5. Compare per-row deltas.

When the same Scenario set and the same Evaluators are held constant, **the only variable left is the agent itself**. Whatever changes in the Evaluation results between the two runs is attributable to the version change. That's the loop the Agent Version concept exists to make legible.

Hold any of those three constants and you get a different signal:

- **Hold Agent Version, vary Scenario set** → "does this version regress on a different slice?"
- **Hold Agent Version, vary Evaluator** → "does the new rubric judge this version differently?"
- **Hold both, vary Agent Version** → **the regression diff**, the daily workflow

## What's in an Agent Version

An Agent Version captures everything that influences agent behavior at runtime:

- **Prompt** — system prompt, persona, instructions
- **Model** — Claude Opus 4.7, GPT-4o, etc., with model-side config (temperature, max tokens)
- **Tool set** — which functions the agent can call, with their schemas
- **Retrieval config** — knowledge base, RAG sources, search parameters (if applicable)
- **Deployment target** — the runtime the version is deployed to (AgentCore alias, Agentforce environment, etc.)
- **Metadata** — version name, description, owner, creation time, tags

Two Agent Versions that differ only in metadata are functionally identical; two that differ in any runtime input are not. The schema makes this distinction explicit.

## Version discovery

For platforms Trust AI has a connector for, **versions are discovered automatically**:

- **AWS Bedrock AgentCore** — Trust AI reads the AgentCore alias and runtime version. Each new deployment becomes a new Agent Version row in the Project, with the AgentCore version ID pinned to the metadata.
- **Salesforce Agentforce** *(roadmap)* — analogous: discovery via the Agentforce metadata API.
- **Manual entry** — for agents on platforms without a connector, you create the Agent Version row manually and attach the relevant config. The Project then knows how to invoke it for evaluation-time runs.

Discovery is read-only — Trust AI never modifies the source platform. Discovered Agent Versions stay in sync with what's deployed; archiving an Agent Version in Trust AI does not affect the source runtime.

## The VersionPill — what users see in the app

In the Trust AI app, an Agent Version surfaces as a compact pill on every Session, Evaluation, and comparison view: a colored badge showing the version name (and sometimes its short SHA-equivalent). Two Agent Versions next to each other in a comparison view show as two pills side by side, color-keyed so the eye can track them.

The pill is the **anchor** users orient around. Reviewing a regression report, you'll see something like:

```
[v2026.05.11]  →  [v2026.05.19]    pass rate: 87.2% → 84.8%  (−2.4)
```

Two pills, one delta, immediately legible. The pill UI is intentionally minimal: it's the most-clicked element in a comparison view, and every detail of the Agent Version is one hop away.

## Lifecycle

An Agent Version is **created** on discovery (or manual entry), **active** while it's the target of new Evaluations, and **archived** when superseded. Archiving doesn't delete:

- Historical Evaluations against the archived version remain readable and re-runnable
- The version stays comparable from any new Evaluation
- Snapshot exports include archived versions

Versions don't expire on their own; archive is a manual or policy-driven action. A team that ships weekly might keep the last six versions hot and archive everything older; a team that ships quarterly might keep every version forever.

## Comparison semantics across more than two versions

The common pattern is a two-version comparison, but the underlying primitive supports more:

- **A vs B vs C** — same Scenario set, three versions. Useful for evaluating prompt-A vs prompt-B vs prompt-C in parallel. The comparison view surfaces each version's row-level score; rows where one stands out localize where each version excels or fails.
- **A baseline + N candidates** — useful when iterating on a candidate replacement; the baseline stays pinned, candidates rotate.
- **A vs A across time** — re-running the same Evaluation on the same Agent Version weeks apart. Surfaces non-determinism in the Evaluator (a calibration smell) and any drift in the agent's runtime behavior between the two runs.

All three patterns reduce to the same Evaluation diff primitive; the UI just shows more pills.

## Working with Agent Versions

You do everything around Agent Versions from the app, usually by asking the Trust Agent:

- **Browse them** — open a Project to see its Agent Versions, or ask the Trust Agent to list the versions in a Project.
- **Inspect one** — click a VersionPill to open the full config snapshot, or ask the Trust Agent what's in a given version.
- **Create one manually** — for a platform without a connector, ask the Trust Agent to create an Agent Version and attach its config (or add the row from the Project view).
- **Archive one** — archive a superseded version from its detail view, or ask the Trust Agent to archive it.
- **Refresh discovery** — trigger a discovery refresh against connected runtimes from the Project view, or ask the Trust Agent to re-check what's deployed.
- **Check run options** — the available runtime invocation options for connected platforms surface in the run setup UI *(v2026.05.19+)*.

If you need the underlying endpoints and their full request/response shapes, the [API Reference](/api-reference) is the authoritative source.
