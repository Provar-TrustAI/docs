# Lessons from M2 (for M4 and beyond)

Captured at the end of M2 execution as part of the DEV-1829 M3 kickoff spike. Updates this file as M3 and later milestones produce new evidence.

---

## What worked well

### The "branch → write → commit → push → PR → merge → pull main" pattern
A single discipline followed across every content ticket. No exceptions. Every PR small enough to review in one sitting. No `--no-verify`, no `--delete-branch`. The Mintlify preview-build race condition stayed dormant because we never gave it a chance to fire.

### Concept-first sequencing
Track C (concept pages) shipped before Track D (tutorials/how-tos). Every tutorial and how-to got to cross-link to a real concept page instead of a placeholder. Internal-link graph stayed coherent throughout, no rewrites needed.

### Promoting H2s to first-class pages
DEV-1821 (Annotations) and DEV-1822 (Agent Versions) were originally just sections in `concepts/overview.mdx`. Promoting them to standalone pages with their own nav slots was the right call — the topics carried 100+ lines of real content each and would have made overview unreadable.

### One-OpenAPI-snapshot pivot
DEV-1808 pivoted from URL-pinning trust-ai-app's openapi.json (which needed expiring tokens on a private repo) to committing the snapshot directly in the docs repo. Two consequences:
- Renders auto-update on PR merge instead of requiring an external token rotation
- Versioning becomes copying a file (`v2026.05.11/api-reference/openapi.json`) instead of negotiating cross-repo URL signing

### Cross-linking discipline
Every new page ends in a "Related concepts" or "What's next" CardGroup. The result: the docs site is genuinely navigable now — you can land on any concept page and reach the rest of the conceptual model in 2-3 clicks.

## What surprised us

### Content authoring cost was lower than estimate
Estimates for content tickets were 0.5-1 person-day each. Real execution was ~5-10 minutes per concept page in this loop pattern. The estimates were anchored against a "research + write + review" cycle, not against the "research already done in the spike phase, just write" execution pattern that the loop enabled.

**Implication for M4+:** for content-only tickets with the source material already mapped (which is the case after the audit/spike phase), the realistic estimate is closer to 0.1-0.2 person-day per ticket. Reserve the higher estimates for tickets requiring fresh research or design decisions.

### The "Evaluation results are Sessions too" pattern (ADR 0006) was a key payoff
This pattern shows up in three different concept pages (Sessions, Evaluations, evaluation-loop) and one tutorial (run-first-evaluation). It's the conceptual primitive that makes the docs feel coherent — without it, "evaluation outputs" would have needed its own bespoke surface area. Worth flagging in the docs as more than incidental.

### "Annotations as the signal, not the metadata" framing was strong
The Annotation vs Evaluator-score comparison table in `concepts/annotations.mdx` ended up clarifying something the docs hadn't said clearly anywhere else: humans annotate, Evaluators score, and the latter is only trusted when calibrated against the former. Several tutorials reach for this concept; making it explicit on the canonical page helped.

### The IAM-policy "External ID" requirement was almost undocumented
DEV-1826 (connect-agentcore expansion) caught that the existing placeholder doc didn't mention the External ID condition. That's a real security requirement and a real first-time-setup blocker — if a customer had followed the placeholder doc end-to-end, they'd have ended up with a less-secure IAM trust relationship and not known it. **General lesson:** when expanding placeholder docs, audit for missing security-relevant fields, not just missing content.

## What didn't work / friction we hit

### Mintlify preview-build race condition (resolved by feedback memory)
Empirically verified via PR #9 in DEV-1810-prep work: `gh pr merge --delete-branch` races the async preview build. The branch gets deleted before Mintlify finishes processing, the build fails silently, and the preview URL 404s. **Permanent fix:** never use `--delete-branch`. Captured as a feedback memory; followed religiously through the M2 loop with zero failures.

### Trust-ai-app pre-push verify failing on unrelated broken test
DEV-1812 (`docs-paths.yml` for cross-repo CI) committed locally as `484c64b2` but couldn't push because the trust-ai-app `pnpm test` was failing on a pre-existing broken test unrelated to the change. The feedback memory prohibits `--no-verify`. Four other trust-ai-app tickets (DEV-1812, 1813, 1809, 1806, 1807) blocked until that test suite is green.

**Process implication:** for cross-repo work in a loop pattern, verify the target repo's CI is green BEFORE starting work. A broken upstream test in the target repo can block an entire track without warning.

### Subagent 529 errors on heavy implementation tickets
Two subagent dispatches died on API overload during M2 prep (DEV-1767 implementation, DEV-1768 implementation). Pivoted to orchestrator-direct execution. **Implication:** for small mechanical tickets in a loop, orchestrator-direct is faster and more reliable than subagent dispatch. Save subagents for tasks that genuinely benefit from parallelism or context isolation.

### Branding & layout tickets ate more time than content
DEV-1772 favicon centering took 3 iterations because the gear's bounding-box center was at (501.40, 142.97), not (488, 141) where I'd placed the circle. Visual-polish tickets often hide this kind of "the data isn't what you assumed" friction; budget more iteration than for content tickets.

## Process learnings for M4 (automation)

### Cross-repo trigger via repository_dispatch
M4's planned automation (trust-ai-app on `v*` tag → docs repo workflow regenerates OpenAPI + opens a version-cut PR) needs:
- A bot identity that can push to `Provar-TrustAI/docs` from CI
- The version-cut PR to be a draft initially (not auto-merged) — the runbook's step 10 (friction notes) requires a human to walk through the new state before merge
- The runbook itself should stay the manual-procedure source of truth — automation should match it step-for-step, not diverge

### Don't over-automate the archive-banner step
The runbook's step 3 (add archive banners) is a candidate for automation, but the manual sweep on the first transition (M3) is valuable for catching per-file weirdness. **M4 automation should default to "open a PR with the mechanical changes" rather than "auto-merge" — the human review is still cheap and the failure modes are not.**

### CI gating on the docs side
DEV-1813 (`docs-paths.yml` in trust-ai-app) is the basis for soft-warning CI on PRs that touch API surface without updating docs. M3+ should treat this as a "review prompt" not a "block merge" — soft-warnings build the habit; hard-blocks build the resentment.

## What to do differently next loop

- **Start with a working state inventory.** This loop began with implicit knowledge of M2's state; an explicit "here's what's merged, here's what's open, here's the next ticket in priority order" snapshot at the start would have removed ~5 min of orientation overhead per resume.
- **Write retro notes inline, not at the end.** The "lessons from M2" pattern this file is following would be richer if I'd captured one bullet per ticket as I went. End-of-loop retros remember the highlights, not the friction.
- **Pre-flight the cross-repo dependencies.** DEV-1812's blocked state would have been caught at the start if the loop's first action had been "run trust-ai-app's test suite to confirm it's green."
