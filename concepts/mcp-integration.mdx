Trust AI runs a [Model Context Protocol](https://modelcontextprotocol.io) (MCP) server that exposes a curated subset of the **Trust Agent's** tools — the same read and write operations the agent uses inside the product — so they can be driven over a standard protocol instead of a bespoke integration.

## What MCP is

MCP is an open protocol that standardizes how AI applications discover and invoke tools exposed by external systems. A server publishes a tool surface; any client that speaks the protocol can connect, discover the available tools, and call them — no custom SDK or plugin per client. For protocol details, see [modelcontextprotocol.io](https://modelcontextprotocol.io).

## What the Trust AI MCP server exposes

The server exposes a **deliberate subset** of the Trust Agent's tool registry — the data and write operations a caller needs, without the agent's internal prompt-scaffolding tools.

**Read tools** — queries against your Trust AI data, executed immediately:

- `get_project_state` — a structured summary of a Project: entity counts, setup flags, and the most recent evaluation run
- `query_data` — list and filter Sessions, Scenarios, Evaluators, Evaluations, Personas, and Agent Versions
- `read_file` — retrieve the extracted text of a task-attached file
- `analyze` — run a structured failure analysis over a completed evaluation run

**Write tools** — operations that modify Trust AI state, each wrapped with an approval gate:

- Sync Sessions from a connected platform, curate Sessions into scenarios, update or delete scenarios, and start an Evaluation. These are the Trust Agent's own write operations, surfaced over MCP and held behind the same confirmation step.

## Authentication

The MCP server enforces the **same auth boundary as the rest of the Trust AI API** — there is no separate permission shape for it. A request is authenticated by your Trust AI **WorkOS session**, and access is scoped to your organization: a caller sees only the Projects, Sessions, and Evaluations that belong to your org, exactly as in the UI. That tenancy enforcement now rides Trust AI's shared authorization engine — the same fail-closed permission resolver that governs the rest of the product — a behavior-preserving cutover that changes nothing about who can call what.

Because that session is the credential, the server today serves callers that already carry it — the in-product Trust Agent and Trust AI's in-process runtime. A **standalone external client cannot present that session**, so direct third-party connection is gated on a dedicated, OAuth-issued credential that is on the roadmap (see the roadmap note above). There is no anonymous access path.

## The approval gate

Write operations pass through a `can_use_tool` confirmation step before they execute:

- **Read tools execute immediately.** Querying data, reading a file, and running analysis need no extra approval beyond a valid session.
- **Write tools pause for confirmation.** When a caller attempts a write — curating Sessions into scenarios, syncing Sessions, starting an Evaluation — Trust AI builds a preview of exactly what would change and holds the operation until it is explicitly approved. Approvals have a short time-to-live; an unanswered request times out rather than executing.

This gate exists because a caller — an agent or an IDE — should not silently modify Trust AI data. The same write tools invoked through the Trust Agent inside the product follow the same approval pattern.

## Observability

Every MCP tool invocation emits an **OpenInference tool span**:

- Each call is traced with its tool name, tenant context (project / org / user), input arguments, output, and status
- Spans follow the OpenTelemetry / OpenInference conventions, so they slot into a Phoenix-style trace view alongside the agent runtime's own tool spans
- Failed calls (auth, validation, approval rejection or timeout) are traced with error metadata

Every tool call — including a denied one — is recorded, so MCP activity is auditable rather than a black box.

## Conformance

The exposed tool surface is **checked against the Trust Agent's tool registry on every build**:

- A conformance test asserts the MCP-exposed allowlist matches the set actually registered on the transport, so the documented surface can't silently drift from what's live
- Tools deliberately held back stay out of the external surface by design. That includes the agent's internal prompt-scaffolding, and also some of the Trust Agent's newer write operations — generating scenarios, simulating a scenario, and saving an evaluator — which are excluded until their shapes are final. The exposed subset deliberately lags the agent's newest capabilities.

## Supported clients

The server speaks standard MCP, so any conformant client can — in principle — drive it. **Direct external connection (Claude Desktop, Cursor, VS Code) is not yet available**; it ships with the OAuth authentication work described above. Today the agent's capabilities reach you through the product's own Trust Agent.
