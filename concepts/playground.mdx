The Playground is where you run the agent under test directly: pick a connected agent, **chat** with it live, or **simulate** a persona pursuing a goal against it — and save any conversation worth keeping as a [scenario](/concepts/scenarios). It sits between building scenarios and running [evaluations](/concepts/evaluations): you drive an agent by hand here before grading it at scale.

The Playground is a per-project surface at `/projects/:id/playground`. It is project-scoped — the agents you can pick, the sessions you can drive, and the History you can browse all belong to the project you are in.

<Note>
  **The Playground is not the Trust Agent.** They are two different chats, on two different routes:

  - The **[Trust Agent](/concepts/trust-agent)** is *your* in-app assistant at `/projects/:id/agent` — it does evaluation work for you (finds sessions, builds scenarios, runs evaluations) and stops for your approval before it writes.
  - The **Playground** (this page) at `/projects/:id/playground` is where you chat with and simulate against the *agent under test* — the customer's [Agent Version](/concepts/agent-versions) you are judging.

  They do not share a thread. When this page says "the agent," it means the connected agent under test, not the Trust Agent.
</Note>

The Playground is a full-screen chat surface — there is no floating dock here. The topbar carries the agent picker; the composer carries the mode toggle; a History menu lets you reopen past sessions.

### The agent picker

The topbar carries a grouped agent picker. Connected agents are grouped by their source system, and picking one shows the agent's display name alongside the system it came from — never a raw ID. Once a session starts, the picker locks to the agent you chose, so the whole conversation runs against one agent.

### The Chat / Simulate toggle

Before a session starts, the composer carries a segmented **Chat / Simulate** toggle — two options, labeled exactly **Chat** and **Simulate**. This is a mode switch, not an "Ask" mode and not the Trust Agent's mode toggle. Once you send a first message, the toggle collapses out of the way — a session is committed to the mode it started in.

## Chat — manual streamed conversation

**Chat** is a manual conversation with the connected agent. You send a message; the agent streams a real reply back live, the same way it would in production — nothing is canned or pre-recorded. Send another message and it continues the same thread; the agent picker stays locked to your chosen agent and the conversation accumulates.

Under the surface, a Chat session is a [simulation](#how-the-playground-runs) with no persona: the first message you send opens the session, and each message after that drives the connected agent for the next reply. You don't have to think about that to use Chat — it behaves like an ordinary conversation — but it is why a Chat conversation can be saved as a scenario and re-run later, just like a simulated one.

## Simulate — persona and goal driven

**Simulate** turns the conversation over to a persona. Toggle to **Simulate** and a **Simulate a conversation** config modal opens, where you set up the run:

<Steps>
  <Step title="Pick exactly one persona">
    A single-select persona picker. A simulation needs a concrete persona to role-play, so the project's default persona is deliberately not offered here — you choose a specific persona, not the implicit fallback. You can also create a new persona inline from the picker; a persona you create is auto-selected for the run.
  </Step>
  <Step title="Write one goal">
    A single goal field describing what the persona is trying to accomplish. **Simulate** stays disabled until you have both a persona and a non-empty goal.
  </Step>
  <Step title="Choose a run mode">
    Two run modes control how much you stay involved:

    - **Full auto** — simulates the whole conversation end-to-end. The persona's user turns and the agent's replies alternate automatically until the conversation resolves or hits its turn cap.
    - **Stay in the loop** — drafts each user turn for you. The persona's next message lands in the composer as a suggestion; you confirm it or edit it before it sends, so you can steer the conversation as it goes.
  </Step>
</Steps>

Press **Simulate** and the conversation auto-drives turn by turn: the chosen persona role-plays the user pursuing the goal, the agent replies, and both stream live while you watch. The run continues for a bounded number of turns rather than forever — long enough to see the agent's behavior, short enough to stay legible.

### The agent's trace

The transcript shows more than the agent's replies. Where the agent's runtime supports it, each turn also surfaces what the agent did to produce its answer: plain-language **Thinking** steps carrying the model's reasoning, the tools it called, and **Model call** entries summarizing each underlying model request — with the raw request and response tucked behind an **Input** / **Output** disclosure rather than spilled onto the surface. Reasoning is provider-dependent: when the connected agent's runtime can't emit it, the trace simply shows no Thinking steps instead of an empty frame.

## Save a conversation as a scenario

Any conversation worth keeping — a **Chat** or a **Simulate** session — can be saved as a [scenario](/concepts/scenarios). Once a session starts, a **Save as a scenario** button (a database icon) appears in the topbar and stays visible for the whole session. It becomes actionable once the conversation has at least one user message and one agent reply; before that, its tooltip reads **Send a message and get a reply first**.

Trigger it and a **Save as a scenario** modal opens — the same lightweight intake a new scenario gets anywhere else:

- **Name** (required) — a short, descriptive scenario name.
- **Goal** (required) — what the user was trying to achieve. A Simulate session pre-fills this with the run's goal; for a Chat you write it yourself.
- **Personas** (optional) — a Simulate session pre-selects the persona that drove the run; a Chat starts empty. Each attached persona means one simulated session per run later.

{/* CAPTURE-PENDING: playground-save-scenario-modal.png — the 'Save as a scenario' modal with the turn-count summary, Name, Goal, and Personas fields */}

The modal summarizes what you're capturing as a turn count, and **Save scenario** does two things at once: it creates a flat scenario in the project's Scenarios list, and it attaches the conversation you just ran as that scenario's linked [session](/concepts/sessions). Nothing is copied or re-typed — the live conversation itself becomes the scenario's session, with its **Source** shown as **Scenario**.

Notice what the modal does *not* ask for: an expected output. Its own sub-text says why — "You'll set expected output later — usually from a session you like." A freshly saved scenario is deliberately incomplete: it shows up as missing expected output (the Scenarios list ships a built-in **Missing expected output** view for exactly these), and you author its grading contract later in the scenario's Expected Output editor — often from the very conversation you just saved. Define cheap, run, then learn the expected output from a real session.

<Note>
  **Saving creates two things you can find later.** The scenario lands in the project's flat [Scenarios](/concepts/scenarios) list with a stable id like `SCN-000123`; the conversation lands on the [Sessions](/concepts/sessions) surface as that scenario's linked session. There is no separate copy — the scenario's Sessions tab and the Sessions list are showing the same conversation.
</Note>

### Where Playground conversations show up as sessions

Every session carries a **Source** — exactly one of **Scenario**, **Playground**, or **External** — plus a **Simulated from** card in its details recording where it came from. A conversation you save as a scenario reads Source **Scenario**; a Playground session you never save reads Source **Playground**. This is the same shape the Scenarios surface's **Generate sessions** flow produces, so a conversation you captured by hand and one the generate flow simulated land identically: real, scenario-linked sessions.

Because a saved conversation is a real session, you can also grade it directly, as it is: sessions are first-class evaluation targets, and the Sessions surface's primary **Evaluate** action grades the conversations that already happened. The [Sessions](/concepts/sessions) page owns that surface.

## History — past sessions

The Playground keeps a **History** view of the project's past sessions, newest first. Each entry shows a turn count and a title — a Simulate session takes its title from the run's goal and is labeled with its persona name; a manual chat takes its title from the first user message and is labeled with just the agent. It lists a scannable recent set rather than every session ever run.

History is real, persisted data — the project's actual sessions, not mock entries and not something kept only in your browser. Reopen a session from History and its full transcript reloads from the server; it survives a hard reload and shows up no matter which browser you signed in from. Starting a new session clears the previous conversation from the surface — past conversations live in History, not in the composer.

## Running a saved scenario at evaluation time

The persona-and-goal simulation you drive by hand here is the same thing an evaluation runs automatically. Once a conversation is saved as a scenario, that scenario can simulate *fresh* conversations on demand: from the Scenarios list or the scenario's fly-in, **Run & evaluate** opens the Evaluation Launcher, you pick the agent to run against, and the run simulates a new conversation for each persona attached to the scenario — grading each one with the evaluators the scenario carries. A goal-only scenario with no persona still simulates one conversation from its goal. The split button's chevron also offers **Run only** — simulate without grading — when you just want more conversations to look at.

Two paths, one distinction: a scenario-sourced run always simulates fresh conversations; grading conversations that *already happened* — including your captured Playground sessions — is the launcher's sessions mode, opened from **Evaluate** on the Sessions surface. Either way, every conversation a run grades is a real session: **Open session** on any result lands you on the exact conversation that was judged. The full execution anatomy — the launcher's entry points, the fan-out preview, and the run results — lives on the [Evaluations](/concepts/evaluations) page. This page names the capability; Evaluations owns the depth.

## How the Playground runs

You don't need the plumbing to use the Playground, but a little context helps. A Playground session is a **simulation**: the simulations domain backs the surface, every session is scoped to its project, and a session is in one of three states — running, ended, or abandoned. That's why History can list, label, and reopen your sessions, and why saving a conversation as a scenario needs no copy step: each session is a real persisted record, not a transient chat.
{/* ACCURACY-AUDIT-PENDING: a Playground session is a project-scoped simulation whose status is running / ended / abandoned, and History reflects those states — verify behaviorally on the release tag; do not publish the internal route table as a stable public API */}

The Playground reuses the same streaming infrastructure the Trust Agent's runtime sits on, but it is a **separate surface** — it is not the Trust Agent runtime, and the two don't share a conversation. Any wait you feel before a reply streams in is upstream **model thinking time** — how long the connected agent's model takes to respond. It is not a fixed latency the Playground promises and not a bug; the time depends on the model and the infrastructure it runs on, which is exactly why the Playground streams the reply in as it lands rather than showing a countdown.
{/* ACCURACY-AUDIT-PENDING: the Playground reuses the kept streaming infra (use-simulation-stream.ts) but is NOT the AG-UI Trust Agent runtime; any wait is upstream model time with no published SLA or duration — verify framing against ADR-0019 and confirm no latency figure is shown on the release tag */}

<Note>
  Your Playground sessions are **tenant-scoped**. Every session belongs to your organization and project, and you only ever see and drive your own org's data — a session from another organization isn't addressable, even by its identifier. The platform side of that boundary lives in [Multi-tenancy](/concepts/multi-tenancy).
</Note>
