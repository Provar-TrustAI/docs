An Evaluator is a reusable judgment function. You define it once in a [Project](/concepts/projects), attach it to the [Scenarios](/concepts/scenarios) it should grade, and it gets reused across many [Evaluations](/concepts/evaluations) — against whatever [Agent Version](/concepts/agent-versions) you're testing. The same Evaluators scoring the same scenarios across two Agent Versions is **the apples-to-apples comparison** Trust AI is built around.

## The two families

Trust AI has two kinds of Evaluators:

<CardGroup cols={2}>
  <Card title="LLM Judge" icon="brain">
    Plain-language grading rules applied by an AI judge at evaluation time. Good for **qualitative judgment** — politeness, completeness, helpfulness, staying in scope. This is the kind you create today, from scratch or from the recommended library.
  </Card>
  <Card title="Structured Match" icon="check-square">
    A deterministic check of what the agent actually did — which skill or intent it routed to, or which tools it called and with what parameters — compared against the scenario's expected turn.
  </Card>
</CardGroup>

New evaluators you create are LLM Judges. You no longer create standalone Structured Match evaluators — structural expectations (**Expected routes**, required or disallowed actions, parameter checks) are authored directly in a scenario's **Expected Output** editor, where they belong to the scenario being graded. Existing Structured Match evaluators keep working and stay editable; each one is either a **Routing match** (which skill or intent the agent routed to) or a **Tool match** (which tools it invoked, including their parameters).

## Start from the recommended library

A new project doesn't start from a blank slate. Trust AI ships a curated library of recommended defaults — the Trust Library — including 12 recommended evaluators — six **Quality** and six **Safety** presets, each a plain-language pass/fail judge:

- Click **Add recommended** in the Evaluators header (or the **Add recommended evaluators** call-to-action on an empty project) to open the adopt modal.
- Each preset shows a provenance chip — a friendly category like **Answer quality** or **Safety & security** — with the underlying method cited in its tooltip. Presets already in your project show a muted **In your project** label, so adopting twice never creates a duplicate.
- Check the ones you want and confirm with **Add N evaluators**.

Three things to know about adopted presets:

1. **They're ordinary evaluators.** Fully editable, no special status — they just keep a friendly **Recommended** chip on the list and detail header so their origin stays clear.
2. **Some are marked "Tailor first."** The presets that judge goal completion, scope, or unauthorized commitments are written against a generic agent. The **Tailor first** badge is a nudge — add a rule about what *your* agent is allowed to do before relying on the check. It never blocks adoption.
3. **They pair with scenarios.** A matching catalog of recommended starter safety scenarios exists on the Scenarios surface, and each is paired with the evaluators that score it well — surfaced as guidance when you attach evaluators, not auto-attached.

## Anatomy of an LLM Judge

You never write a judge prompt. An LLM Judge is defined by:

1. **A name and description** — what the check is, in your team's words.
2. **A score type** — **pass/fail**, or a **1–5 scale** with a passing threshold you pick from **≥2 ≥3 ≥4 ≥5** (default ≥3).
3. **Grading rules** — short plain-language sentences. Pass/fail judges take two lists, **Passes when…** and **Fails when…**; 1–5 judges take three anchors, **Rate 5 when…**, **Rate 3 when…**, and **Rate 1 when…**.

The app assembles your rules into the exact criteria the live judge uses. If you want to see the result, it's read-only behind **Advanced — view the prompt this produces**. A **Reason step by step** toggle asks the judge to reason before scoring.

Two properties make the judges dependable:

- **Deterministic scoring.** Judges run at temperature 0 — the same transcript and the same rules produce the same verdict. This is pinned by the product, not a setting.
- **Environment problems don't count against the agent.** If a judge call fails for infrastructure reasons, the result is excluded from the score as an environment problem rather than recorded as a failure.

### Tailor with AI

On the create form, the **Tailor with AI** box drafts the rules for you: describe what the evaluator should check in one sentence, click **Draft for my agent**, and draft pass/fail rules prefill the editor. You review and edit them — nothing is saved until you save.

For how to write rules that hold up, see [Write an effective LLM-judge prompt](/how-to/write-llm-judge-prompt).

## Attaching Evaluators to Scenarios

Evaluators meet scenarios in the scenario's **Expected Output** editor. The **+ Add evaluator** picker leads with a **Recommended for this scenario** group — the evaluators paired with that scenario's kind of failure — ahead of the full Quality/Safety catalog, and a recommended preset can be adopted and attached in one step from inside the picker. Removing an evaluator from a scenario is clean, and re-adding it is a plain attach.

At run time, an [Evaluation](/concepts/evaluations) applies each scenario's attached evaluators; you can also grade a single scenario or session in place, without creating a run in the Evaluations list.

## Calibration — the loop that makes LLM Judges trustworthy

An LLM Judge is useful only if it agrees with informed humans. Trust AI measures this in two places — the **Calibrate** tab checks the judge against the pass/fail verdicts you recorded on scenarios, and **evaluator accuracy** on run results measures agreement with the human **Verdict** annotations on sessions. In both, the ground truth is **your verdicts**: a human verdict grades the judges — it never overrides the automated verdict.

### The Calibrate tab

Every LLM Judge's side panel has a **Calibrate** tab (Structured Match evaluators don't — they're deterministic, so there's nothing to calibrate):

1. In the **Check agreement** section, pick the scenarios that carry your own pass/fail verdicts — calibration reads a pass/fail verdict column on your scenario data, and the picker shows how many scenarios carry a verdict.
2. Click **Calibrate**. Calibration never runs automatically — it costs judge calls — and each check samples up to 20 annotated scenarios.
3. Read the results: a plain-language agreement headline with a verdict breakdown, and a **Row by row — disagreements first** table so the mismatches are the first thing you see.
4. Refine your grading rules where the judge and you disagree, then check again — the button reads **Re-calibrate** once a result exists. **Calibration history** keeps every run, newest first, so you can watch agreement climb.

Calibration runs the exact production judge — same model, same assembled prompt, same parser — so the agreement you measure is the agreement you'll get in real Evaluations. If you edit the grading rules after a check, the tab warns you that the last result is stale.

### Evaluator accuracy on run results

Evaluation run results lead with a dual headline: the **Pass rate** grades the agent, and evaluator accuracy grades the judges — how often the automated verdicts agreed with a human's session verdicts. Calibrate before you trust; an uncalibrated judge produces scores that look authoritative but may not track reality.

## In the app

- **Evaluators list** — every evaluator in the project, with kind, score type, and any **Recommended** provenance chips. **Add recommended** and **New evaluator** live in the header.
- **Create flow** — name and describe the evaluator, pick the score type, then author the grading rules by hand or with **Tailor with AI**. The kind is fixed once created.
- **Side panel** — click any evaluator to open its panel: **Overview** (recent trend), **Criteria** (the grading rules, committed with **Save grading rules**; for Structured Match, what it checks), **Runs** (run history), and **Calibrate** on LLM Judges.
- **Safe delete** — the app checks references before deleting. If active evaluation runs still depend on the evaluator, the delete is refused (**Can't delete**), so you can't silently break a comparison.

## Reusing Evaluators across scenarios, Agent Versions, time

A defined Evaluator is reusable; that's the whole point. You apply it:

- **Across scenarios** — attach the same judgment criteria to every scenario where it's relevant
- **Across Agent Versions** — same scenarios, same criteria, different agent under test = **regression signal**
- **Across time** — re-running the same Evaluator on the same scenarios two months later tells you about drift

Each calibration run snapshots the rules it tested along with its agreement metrics, so **Calibration history** answers "which version of the rules earned this agreement score?" — and the app tracks when the criteria were last edited, so a rename doesn't falsely mark a calibration stale.

## Why we explicitly reject other framings

Trust AI's Evaluator vocabulary deliberately differs from some adjacent ecosystems:

- **Not "metric."** Metrics are continuous scalars (latency, token count). Evaluators produce judgments — pass/fail or rubric-anchored — and the product distinguishes these.
- **Not "test."** Tests pass or fail without nuance. Evaluators carry reasoning — when an LLM Judge fails a result, the reasoning is part of it, which is how you debug agent regressions.
- **Not "guardrail."** Guardrails are runtime controls (block this output before it reaches a customer). Evaluators are post-hoc judgment. The two even meet cleanly in results: a response blocked by a runtime guardrail shows as **Guardrail blocked** and is excluded from the score rather than counted as a failure.

## Working with Evaluators

Everything you'd want to do with an Evaluator is available through the Trust Agent and the app:

- **Adopt** recommended evaluators — ask the Trust Agent, or click **Add recommended** on the Evaluators surface.
- **Define** one from scratch — ask the Trust Agent to create it, or use the create flow and author the **Passes when… / Fails when…** rules.
- **Tailor** a preset — heed the **Tailor first** badge and add a rule about what your agent is allowed to do.
- **Attach** evaluators to scenarios from the scenario's **Expected Output** editor.
- **Calibrate** an LLM Judge on the **Calibrate** tab — **Check agreement**, work the disagreements, re-check.
- **Check references** before deleting — the app blocks a delete that would break active evaluation runs.

Developers who want the underlying endpoints and request/response shapes can find them in the [API Reference](/api-reference).
