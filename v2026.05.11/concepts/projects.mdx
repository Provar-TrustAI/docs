A Project is the workspace boundary around a single agent (or fleet of related agents) under evaluation. Members, runtime credentials, and every Trust AI artifact — Sessions, Annotations, Datasets, Evaluators, Evaluations, Agent Versions — belong to one Project.

## What's in a Project

Every Project owns five things:

<CardGroup cols={2}>
  <Card title="Members + access" icon="users">
    Who can read, annotate, define Evaluators, or run Evaluations. Identity is federated via WorkOS — your organization's SSO is the source of truth.
  </Card>
  <Card title="Runtime connection config" icon="plug">
    Credentials for the agent platform you're evaluating — AWS Bedrock AgentCore IAM role, Salesforce Agentforce OAuth, etc. One set per Project (you can't half-connect).
  </Card>
  <Card title="Inventory of artifacts" icon="folder-open">
    Sessions ingested, Annotations applied, Datasets curated, Evaluators defined, Evaluations run, Agent Versions tracked. All scoped to this Project.
  </Card>
  <Card title="Settings" icon="sliders-horizontal">
    Project name, description, default Evaluator pool, evaluation defaults, and the snapshot/restore configuration covered below.
  </Card>
</CardGroup>

## Who creates Projects, and when

The two personas who reach for "+ New Project" most:

- **SI Consultant** — setting up Trust AI for a new customer engagement. The Consultant creates one Project per agent the customer cares about evaluating. Their first hour in the product is: create Project → connect runtime → import the first Sessions.
- **Agent Owner** — adding a new agent to an existing customer's Trust AI workspace. Same shape, narrower scope: they know what they're evaluating and just need the workspace to put it in.

In both cases, you're answering the question: "Do these things deserve to share an Evaluator pool, an annotation queue, and an Agent Version timeline?" If yes, one Project. If no, two.

## One Project per logical product

Practical guidance:

- **Different production agents** → different Projects. Even if they share an underlying model, the Datasets and Evaluators that judge them are different. Don't conflate.
- **Multiple environments of the same agent** (staging vs prod) → **one Project**, two Agent Versions. The Datasets and Evaluators want to be shared across environments; what differs is the version under test.
- **A fleet of similar agents** (one per customer, one per region) → judgment call. If the evaluator criteria are identical, consider sharing. If they diverge, split.

## Runtime connection config

A Project knows how to reach one or more agent runtimes. The connection is configured once per Project; from then on, ingestion, evaluation-time invocation, and version discovery all flow through it.

V0 ships **AWS Bedrock AgentCore** as the supported connector. For setup, see [Connect an AWS Bedrock AgentCore agent](/how-to/connect-agentcore).

Salesforce Agentforce and Microsoft Copilot Studio connectors are on the roadmap; the Project model is intentionally pluggable.

## Project Snapshots

A Project is the unit of restore. The snapshot system captures the full state of a Project at a moment in time — its Sessions, Annotations, Datasets, Evaluators, Evaluations, and Agent Versions — so it can be restored or audited later.

Snapshots are useful for:

- **Audit trails** — proving what a Project looked like at the time a regression-test run "passed"
- **Disaster recovery** — restoring a Project after accidental destructive edits
- **Forking** — branching an Evaluator pool to experiment without affecting the live Project

Snapshots are immutable and named. Restoring a snapshot creates a new Project rather than overwriting the original.

## Where this fits in the evaluation loop

<Tip>
  Projects are the **boundary**, not a step. Every step of the [evaluation loop](/concepts/evaluation-loop) — ingest, review, curate, define, run, compare — happens inside one Project. If you find yourself wanting to cross a Project boundary mid-workflow, that's a signal the workflow belongs in a single Project (or the Projects need merging).
</Tip>

## API surface

Projects appear in the `/v1/projects` endpoint family of the [API Reference](/api-reference). Common operations:

- `POST /v1/projects` — create a Project
- `GET /v1/projects/{project_id}` — fetch Project details
- `GET /v1/projects` — list Projects in the org
- `POST /v1/projects/{project_id}/members` — invite members
- `POST /v1/projects/{project_id}/snapshots` — create a Snapshot
- `POST /v1/projects/{project_id}/restore-from-snapshot` — restore from a Snapshot

Full request/response shapes live in the auto-generated [API Reference](/api-reference).
