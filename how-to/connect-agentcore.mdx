This how-to connects an **AWS Bedrock AgentCore** agent to a Trust AI [Project](/concepts/projects). When you're done:

- Your AgentCore agent's conversations import into the Project as [Sessions](/concepts/sessions) — on demand with **Sync now**, or automatically if you set up streaming
- The connected agent appears as an agent build you can pick when you launch an Evaluation, and on each session's Details

AgentCore is one of three platforms in the **Add connection** picker, alongside [Salesforce Agentforce](/how-to/connect-agentforce) and [Claude Managed Agent](/how-to/connect-claude-managed-agent) — each has its own setup guide. This page covers AgentCore only.

Before you start, it's worth asking the [Trust Agent](/concepts/trust-agent) whether an agent is already connected — try *"Is an agent connected to this project?"* It checks the project's connected agent builds and tells you what it finds, so you don't reconnect something that already works. The AgentCore connection itself is made in a guided wizard in Project Settings, described below.

<Warning>
  This setup creates a **cross-account IAM role** that Trust AI assumes to reach your AWS account, protected by an External ID baked into the setup package. Coordinate with whoever owns AWS security in your team before creating the role. Use the setup package the wizard generates — don't hand-author the policies.
</Warning>

## Prerequisites

- An **AWS account** with a deployed AgentCore Runtime agent. Trust AI discovers agents once the AgentCore runtime and its Memory are ready.
- An **AWS admin** who can create IAM roles in that account — or you, if that's your role. The wizard is built for handoff: you can send the whole setup to your admin and come back when they reply with a role ARN.
- A **Trust AI Project**. If you don't have one yet, start with [Your first 10 minutes in Trust AI](/tutorials/quickstart).
- The **AWS region** your agent runs in (for example `us-east-1`).

## Connect the agent

<Steps>
  <Step title="Open the setup wizard">
    In your Project, open **Project Settings**, select the **Connection** tab, and click **Add connection**. Choose **AWS Bedrock AgentCore** from the platform picker.

    **You should see:** a "Connect your AgentCore agent" pane with three progress chips — **Region**, **AWS setup**, **Role ARN**. The wizard walks you through one step at a time.
  </Step>

  <Step title="Enter the region">
    Enter the **AWS region** where your agent runs (like `us-east-1` or `eu-west-1`) and click **Continue**. The region is included in the setup package your AWS admin receives, so get it right here.
  </Step>

  <Step title="Send the AWS setup to your admin">
    On the **AWS setup** step, click **Copy setup for AWS admin**. This copies a self-contained setup package to your clipboard — send it to the person who manages AWS. It contains everything they need:

    - A **cross-account trust policy** that lets Trust AI assume the role, locked to an External ID
    - The **minimum permission policy** the connection needs
    - A ready-to-use **CloudFormation template** that creates the role in one stack. The package notes that the template body must be pasted into a new CloudFormation stack (or uploaded to an approved S3 bucket) — there is no one-click Quick Create link yet.
    - Console steps and CLI commands for the optional streaming setup (see [Stream sessions automatically](#optional-stream-sessions-automatically) below)

    The stack creates a role named `TrustAI-AgentCore-<region>` and outputs `TrustAIAgentCoreRoleArn` — that ARN is what your admin sends back to you.

    Wondering what access you're granting? The wizard's **What access does this setup use?** expander answers it: Trust AI can import AgentCore conversation history and run evaluation prompts against the selected agent. The setup does not grant permission to change, deploy, or delete AgentCore resources. In IAM terms, the policy has five statements: verify the role assumption worked, discover agents and their versions and memories, read conversation history from AgentCore Memory, invoke the agent runtime for evaluations, and run CloudWatch Logs Insights queries to pull session traces.

    When your admin has created the role, click **I have the role ARN** to move on. If you're waiting, you can leave and come back — the wizard is where you left it in the Connection tab.
  </Step>

  <Step title="Paste the role ARN and check access">
    On the **Role ARN** step, paste the ARN your admin sent back (it looks like `arn:aws:iam::123456789012:role/TrustAI-AgentCore-us-east-1`) and click **Check access**.

    Trust AI assumes the role and verifies it, then shows three facts — **Account**, **Region**, and **Access check** — plus one of these outcomes:

    - **Access confirmed** — Trust AI can connect to AgentCore in this account, and shows the agent it found.
    - **Choose the agent to connect** — the account has multiple AgentCore runtimes. Pick the one this Project should track; each runtime carries a **Ready** or **Needs review** badge. A connection tracks exactly one runtime — add another connection for a second agent.
    - **AWS role needs an admin update** — the role is reachable but missing required access. The pane lists the exact missing IAM actions as chips. Send the setup package to your admin again, then click **Check access** to recheck.
    - **AWS couldn't assume this role** — double-check the role ARN, the AWS account number, and that the role's trust policy lets Trust AI assume it.
    - **Role connected, no ready agent found yet** — the role works but no runtime is ready. You can click **Save role connection** now and return to Connection settings after an AgentCore runtime and Memory are ready.

    One guardrail to know about: the wizard rejects AWS SSO reserved roles and warns on roles that look person-bound (names containing things like `sso`, `federated`, or `personal`). Saved connections need a dedicated Trust AI cross-account role protected by an External ID — a role tied to a person signing in won't work.

    Once you've picked the agent, click **Save connection**.
  </Step>

  <Step title="Import your first sessions">
    Open **Sessions** in the Project. In the page header, click the **Sync now** dropdown and pick your AgentCore agent from the list (agents are grouped by connected system). While it runs, the button reads **Syncing…**; when it finishes, a toast confirms how many sessions were synced from that agent.

    Sync is on-demand and per-agent — picking an agent syncs only that agent, and there is no "sync all" option. Each sync pulls recent AgentCore activity from your account's CloudWatch logs using the saved role.

    To confirm the connection end to end, you can also ask the Trust Agent — *"Which agent build is connected now?"* — and it will report the newly connected agent.
  </Step>
</Steps>

## Optional: stream sessions automatically

**Sync now** works as soon as the role and runtime are connected. If you want sessions to import automatically instead, the setup package includes a second handoff for your AWS admin — **Send AgentCore sessions to TrustAI** — which subscribes two CloudWatch log groups to a Trust AI destination:

- `aws/spans` — span records with timing, trace IDs, and session context
- `/aws/bedrock-agentcore/runtimes/<agent_id>-<endpoint>` — runtime events with the customer and assistant messages

Both are required to reconstruct sessions, and your AWS account keeps control of the subscription filters. The connection's streaming status shows as **Not set up**, **Partially streaming** (only one log group is arriving), or **Ready** — at Ready, sessions stream into Trust AI in real time and you no longer need to sync by hand.

If Trust AI hasn't enabled session import for your region yet, the package says so — you can still connect the role now, and Trust AI provides the CloudWatch setup steps once the region is enabled.

## Other sign-in methods

The admin-handoff role is the recommended way to connect, but the AgentCore connection form supports other **Sign-in method** options: **Local AWS profile** (local development only), **AWS access keys** (advanced fallback), **OAuth token**, and **JWT token**.

Whatever method you use, secrets are write-only: a saved access key or token is shown only as a masked value and is never displayed back — to change it, you enter a new one.

## Troubleshooting

<AccordionGroup>
  <Accordion title="Check access fails with 'AWS couldn't assume this role'">
    Trust AI couldn't assume the role at all. Check, in order:

    1. The **role ARN** is exactly the `TrustAIAgentCoreRoleArn` output from the stack — not a different role in the account.
    2. The **AWS account number** in the ARN matches the account where the stack was created.
    3. The role's **trust policy** is the one from the setup package — it must allow Trust AI's principal to assume the role, with the External ID condition intact. If your admin edited the policy or created the role by hand, resend the package and have them create it from the CloudFormation template body.
  </Accordion>

  <Accordion title="Check access warns 'AWS role needs an admin update'">
    Trust AI can reach the role, but required read access is missing. The pane lists the missing IAM actions as chips — send the setup package to your AWS admin again and ask them to recreate or update the role from it, then click **Check access** to recheck. This usually means the role was authored by hand from an older or trimmed policy rather than the current package.
  </Accordion>

  <Accordion title="The role ARN is rejected as an SSO or personal role">
    AWS SSO reserved roles are tied to a person signing in and can't be used for saved Trust AI access. The same goes for roles that look person-bound. Ask your AWS admin for the dedicated Trust AI cross-account role from the setup package — it's protected by an External ID and isn't tied to any person's sign-in.
  </Accordion>

  <Accordion title="Role connected, but no agent is found">
    The role works, but discovery found no ready AgentCore runtime. Confirm:

    1. The **region** you entered in step 2 matches where the agent is actually deployed — the role and discovery are region-scoped.
    2. The AgentCore **runtime and its Memory are ready** in that region — check the AgentCore console. You can save the role connection now and come back to Connection settings once they're ready.
  </Accordion>

  <Accordion title="Sync now imports nothing or misses recent conversations">
    A sync pulls recent activity from your account's CloudWatch logs, so:

    1. Make sure the agent has had **invocations recently** — a quiet agent syncs zero sessions, which the toast reports honestly.
    2. If conversations exist but don't arrive, recheck access from the Connection tab — a role that lost its CloudWatch Logs Insights access can discover agents but not pull sessions. The missing-actions chips will show what's gone.
    3. If you expected sessions to appear **without syncing**, that's the streaming setup — see [Stream sessions automatically](#optional-stream-sessions-automatically) above. Until streaming shows **Ready**, sessions arrive only when you sync.
  </Accordion>
</AccordionGroup>
