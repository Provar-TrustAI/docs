An Agent Version is a **pinned snapshot of the agent under test** — its prompt, model, tool set, retrieval config, and deployment target captured as a single, named, immutable reference. Every scenario-driven [Evaluation](/concepts/evaluations) runs against exactly one Agent Version, and every regression-tracking workflow is a diff between two.

One exception up front: an Evaluation can also grade [Sessions](/concepts/sessions) that already happened. Those runs judge recorded transcripts and never invoke your agent, so they carry no pinned Agent Version.

If Sessions are the raw material, [Scenarios](/concepts/scenarios) the test bed, and [Evaluators](/concepts/evaluators) the rubric, then Agent Versions are **what's being judged**. In UI copy you'll often see a version called an *agent build* — same thing.

<Note>
  **An Agent Version is not the Trust Agent.** They share the word "agent," but they are opposite ends of the evaluation:

  - An **Agent Version** (this page) is the *thing being evaluated* — a pinned snapshot of the customer's agent under test.
  - The **[Trust Agent](/concepts/trust-agent)** is the *in-app assistant* that helps you run those evaluations.

  The Trust Agent helps you judge Agent Versions; it is never itself "a version."
</Note>

## The regression-tracking pattern

The core regression-tracking pattern:

1. Pick a stable selection of **Scenarios**.
2. Pick the same set of **Evaluators**.
3. Pick **two Agent Versions** — the production one and a candidate replacement.
4. Run an Evaluation against each.
5. Read the two finished runs side by side and compare per-scenario results.

When the same Scenarios and the same Evaluators are held constant, **the only variable left is the agent itself**. Whatever changes in the Evaluation results between the two runs is attributable to the version change. That's the loop the Agent Version concept exists to make legible.

<Note>
  There is no dedicated run-vs-run comparison view in the product yet — you compare by opening the two run details. **Re-run** on a finished run restores its full configuration, including the pinned agent and judges, so repeating the same setup against a different version is a one-field change.
</Note>

Hold any of those three constants and you get a different signal:

- **Hold the Agent Version, vary the Scenarios** → "does this version regress on a different slice?"
- **Hold the Agent Version, vary the Evaluators** → "does the new rubric judge this version differently?"
- **Hold both, vary the Agent Version** → **the regression diff**, the daily workflow

## What's in an Agent Version

An Agent Version captures everything that influences agent behavior at runtime:

- **Prompt** — system prompt, persona, instructions
- **Model** — Claude Opus 4.7, GPT-4o, etc., with model-side config (temperature, max tokens)
- **Tool set** — which functions the agent can call, with their schemas
- **Retrieval config** — knowledge base, RAG sources, search parameters (if applicable)
- **Deployment target** — the runtime the version is deployed to (an AgentCore runtime, a Salesforce Agentforce org, a Claude managed agent)
- **Metadata** — version name, description, owner, creation time, tags

Two Agent Versions that differ only in metadata are functionally identical; two that differ in any runtime input are not.

## Where Agent Versions come from

Agent Versions arrive through **connectors**. Three platforms connect today, as peers, from **Project Settings → Connection → Add connection**:

- **AWS Bedrock AgentCore** — a guided setup flow connects the runtime, and its builds sync into the Project automatically. Discovery stays reconciled with what's deployed: a build that no longer matches the connected runtime is archived, and pointing the connection back at it revives the record.
- **Salesforce Agentforce** — connect your Salesforce org with JWT-based credentials, and Trust AI drives the live agent through evaluations.
- **Claude Managed Agent** — connect an Anthropic managed agent on the Claude Platform. Trust AI observes the agent and relays its responses; the agent runs its own tools.

The fastest path is to ask the **Trust Agent** to connect your agent: its guided connect flow covers Claude-managed and Salesforce Agentforce agents today, and it hands you a plain-English checklist before the connection form. For a platform without a connector, you can register an agent build directly through the API.

Two things hold across every connector:

- **Sync is on demand and per agent.** The **Sync now** button in the Sessions page header opens a picker grouped by connected system, so you refresh exactly the agent you care about.
- **Connection secrets stay secret.** Private keys, access keys, and API keys are write-only — once saved, they are never shown back.

Archiving an Agent Version in Trust AI does not affect the source runtime.

## Pinned at run time

When you launch a scenario-driven Evaluation, the launcher's **Agent** section asks which connected agent simulates each scenario — the run pins that build, and its results carry the pin. When you drive the run through the Trust Agent instead, it confirms which agent build it will run against before simulating, and tells you plainly if none is connected. **Re-run** on a finished run restores the original run's full configuration, including its pinned agent and judges.

Evaluations launched from Sessions skip this step entirely: grading a recorded transcript needs no agent, so no version is pinned.

You can also see the pin after the fact: a session's **Details** tab shows its **Model** — resolved from the pinned agent version — alongside its "Simulated from" provenance.

## Lifecycle

An Agent Version is **created** on discovery (or registration), **active** while it's the target of new Evaluations, and **archived** when superseded. Archiving doesn't delete:

- Historical Evaluations against the archived version remain readable, and **Re-run** can restore any finished run's configuration
- The version's results stay comparable against any newer run

Archiving follows discovery: reconciling against the connected runtime archives builds that are no longer deployed, and re-registering the same build revives the record. Versions don't expire on their own — a version stays active until the runtime it points at moves on.

## Comparing more than two versions

The common pattern is a two-version comparison, but the underlying primitive supports more:

- **A vs B vs C** — the same Scenarios, three versions, three runs. Useful for evaluating prompt-A vs prompt-B vs prompt-C in parallel; reading each run's per-scenario results localizes where each version excels or fails.
- **A baseline + N candidates** — useful when iterating on a candidate replacement; the baseline run stays pinned, candidates rotate.
- **A vs A across time** — re-running the same Evaluation on the same Agent Version weeks apart. Surfaces non-determinism in the Evaluator (a calibration smell) and any drift in the agent's runtime behavior between the two runs.

All three reduce to the same primitive: Evaluation runs that held the Scenarios and Evaluators constant, read against each other.

## Working with Agent Versions

You do everything around Agent Versions from the app, usually by asking the Trust Agent:

- **See what's connected** — ask the Trust Agent which agent builds are connected in the Project; the manual view is **Project Settings → Connection**.
- **Connect a new platform** — ask the Trust Agent to connect your agent, or add it yourself from **Connection → Add connection**.
- **Sync on demand** — click **Sync now** in the Sessions page header and pick the connected agent to refresh.
- **Pin one at run time** — pick the build in the launcher's **Agent** section, or tell the Trust Agent which build to evaluate.
- **Check what a session ran on** — open the session's **Details** tab for its Model and "Simulated from" provenance.

If you need the underlying endpoints — including registering a build for a platform without a connector — the [API Reference](/api-reference) is the authoritative source.
