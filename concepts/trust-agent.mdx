The Trust Agent is the in-app assistant that helps you do evaluation work inside a [Project](/concepts/projects). You ask it to find sessions, generate scenarios, draft an evaluator, run an evaluation, or analyze a regression — and it works in the open: streaming its reasoning, surfacing real records you can click into, planning before it acts, and pausing for your approval before it changes anything. On a brand-new project it goes further: it can guide you from connecting your agent all the way to your first scored evaluation, inside one conversation. See [The guided first hour](#the-guided-first-hour) below.

It lives at `/projects/:id/agent` and is the default, flag-free experience on every project. The legacy agent implementation is gone — there is no `?v2`, `?agui`, or `?legacy` flag to opt into; the Trust Agent you see is the only one there is.

<Note>
  **The Trust Agent is not an Agent Version.** They share the word "agent," but they are different concepts:

  - The **Trust Agent** (this page) is *your* assistant inside Trust AI — the thing that helps you build and run evaluations.
  - An **[Agent Version](/concepts/agent-versions)** is the *customer's agent under test* — a pinned snapshot of the agent you are judging.

  The Trust Agent helps you evaluate Agent Versions. It is never itself "a version." Everywhere this page says "the agent," it means the Trust Agent.
</Note>

## The three shells

The Trust Agent runs from a single instance that can show up in three shells, and your conversation moves with you between them:

- **Full page** — the agent occupies the project content pane, with the streaming transcript, composer, and side rail.
- **Docked panel** — a compact panel overlaying whatever tab you're on, so you can keep a conversation going while you work elsewhere.
- **Collapsed pill** — a minimized indicator that gets out of your way until you need it.

It is one mount, not three separate widgets. Switching from full to docked to pill and back preserves the transcript and the conversation state — it is the same session throughout. Because the agent is mounted at the app root, the chat survives navigation between core nav tabs: move to **Scenarios** or **Evaluations** and back, and your conversation is still there, not reset.
{/* ACCURACY-AUDIT-PENDING: chat/thread state persists across full→docked→pill transitions AND across nav-tab navigation (one app-root mount, not remounted per page) — verify by driving the shells on the release tag */}

When a project has no active agent task, the agent resolves to the minimized pill so it stays out of the way. If a backgrounded conversation hits an approval gate, the docked view auto-expands to surface the card that needs your decision.
{/* ACCURACY-AUDIT-PENDING: no-active-task resolves to the pill; an approval gate while minimized auto-expands the docked view — verify by driving the app on the release tag */}

### Recent chats and persistence

A floating chat selector lists your recent chats, most-recent first, each with a per-chat status. An unread-indicator dot appears only on chats that actually have unread messages, so the list reads clean. Those recent chats are unified onto the project's canonical task list — the same conversations show up wherever the project tracks work.
{/* ACCURACY-AUDIT-PENDING: chat selector sorts most-recent-first, each shows a status, unread dot only on unread chats, and recent chats are the project's canonical task list — verify against the app on the release tag */}

Transcripts are durable, not just saved. Streamed turns, the agent's response blocks, and approval decisions persist in their true event order — through a page reload, a dropped connection, or closing the tab mid-stream — so a reopened chat matches what you saw live. That durability covers the parts that matter most: a pending approval gate comes back as an answerable card with the composer held until you decide, and a message you send while the agent is still responding is queued, not lost.

If you leave while a reply is still streaming, reopening the chat shows a quiet **This reply was interrupted.** note with a one-tap **Continue** action. Continue resumes from the saved context — it picks the work back up rather than reattaching to the original live stream.

Starting a *new* chat opens a Welcome page — it does not start streaming on its own. The Welcome page offers setup-aware starter prompts (see [The guided first hour](#the-guided-first-hour)); picking one pre-fills the composer, and the prompt *you* submit is auto-sent as the conversation's first turn. Nothing runs until you send it.

In the full view the composer is pinned to the bottom of the screen, and the project's left nav (**New chat**, **Playground**, **Scenarios**, **Sessions**, **Evaluators**, **Evaluations**, **Requirements**, **Project Settings**) stays visible beside the agent — the assistant is seated *inside* the app, not floating over it. **New chat** is the agent's entry in that nav.

## The streaming transcript

The agent thinks and answers in the open. Reasoning, text, and tool-call arguments all stream live, token by token — a turn does not buffer silently until it finishes and then dump a wall of text. Because of that, an approval card can paint mid-turn, the moment the agent reaches the tool call that needs your decision, rather than only after the whole turn completes.
{/* ACCURACY-AUDIT-PENDING: text/reasoning/tool-args stream incrementally and a HITL card can paint mid-turn at the tool call — verify by driving a streaming + write turn on the release tag */}

Reasoning is persisted with the chat, so a reopened conversation rebuilds its Thinking rails rather than losing them.

    The trace goes one level deeper than tools: each underlying model request appears as a **Model call** step with a plain-language summary — a friendly model name and the token counts, like *Model call · 1180 in / 142 out* — and the raw request and response tucked behind an opt-in **Input**/**Output** disclosure. A raw model identifier or a JSON dump never reaches the surface.

    Any wait you feel before a response or an approval card is upstream **model thinking time**, not a runtime delay or a bug. The Thinking timer and a working indicator are there to make that unavoidable wait legible — there is no fixed duration or service-level promise, because the time depends on the model and the infrastructure it runs on. ## Tool results as clickable records

The Trust Agent doesn't describe your records in paragraphs — it hands you the real things. When a tool result returns multiple records, they render as a **clickable object table** right in the chat: real, human-readable records, no bare IDs. Click one and the real record's fly-in opens in context, the same fly-in you'd get from the Sessions or Evaluations surface.

## The side rail

While the agent plans and works, a side rail tracks what's happening, in four sections in this order:

1. **Progress** — the plan. Step counts and live execution status tick as the agent works through its task list.
2. **Context** — the entities the agent is referencing for this task.
3. **Outputs** — what the agent created, updated, or ran, per entity, each deep-linking into a fly-in.
4. **Attachments** — the files attached to the active task. Click a file and a readable preview opens in place — no download required — for PDF, Word, Markdown, text, CSV, and JSON.

The rail stays visible across the whole conversation — it doesn't reset or vanish when you send a follow-up — so it remains a stable map of the work from planning through execution. Counts in the rail show the latest count per item type, not a confusing running sum.

## Human-in-the-loop approvals

The Trust Agent stops for you before it does anything consequential. Approvals use an **interrupt shell** that replaces the composer in its slot and hosts three kinds of card:

- A **permission card** — gates a write. When the agent emits a write tool, the run halts at that tool call and waits for your decision before continuing.
- A **single-question card** — one clarifying question.
- A **multi-question flow card** — a short sequence of clarifying questions with back-navigation between them.

Permission cards are rich: the write preview shows the **diff**, the **dependencies**, and the **proposed end state** before you decide — not a bare ID you'd have to trust blindly. Approving a write actually executes it on the real agent run path: your **Approve** resumes the run and the mutation lands. It is not a preview that quietly does nothing.
{/* ACCURACY-AUDIT-PENDING: the interrupt shell replaces the composer and hosts permission / single-question / multi-question cards; permission cards halt the run; write previews show diff/dependencies/proposed state; Approve executes the real write — verify by triggering writes and clarifies on the release tag */}

Approval gates are durable. A pending gate survives a reload or a dropped connection: reopen the chat and the card comes back answerable, with the composer held until you decide, and your answer resumes the run. Resolved approvals persist too, in the order they happened, so the audit trail a reopened chat shows matches what happened live.

Single- and multi-question cards offer a write-in **Other** option alongside the drafted choices, and the cards are dismissible. Once you answer, the decision is recorded in the transcript as a resolved state that echoes back what you chose (and the per-question answers for a multi-question flow). And if an approval gate is reached while the agent is minimized, a backgrounded-approval-request indicator surfaces on the shells, so you know a decision is pending without the full view open.
{/* ACCURACY-AUDIT-PENDING: question cards offer a write-in "Other" and are dismissible; resolved decisions echo back into the transcript; a backgrounded-gate indicator surfaces on the shells — verify by driving these flows on the release tag */}

{/* CAPTURE-PENDING: trust-agent-permission-card.png — a permission/write-preview interrupt card in the composer slot (re-add inside an Accordion titled "See a permission card" once captured) */}

## The composer, modes, and skills

The composer is where you drive the agent, and it's rich: an **Attach files** chip for pulling in documents, an **@** picker for mentions, a **/** palette for slash commands and skills, an inline model selector chip, an **Ask** / **Plan** / **Act** mode toggle, and a plan-confirm bar. It's also forgiving: the textarea auto-grows as you type, an unsent draft survives navigating away — even leaving the project or reloading — and a message you send while the agent is mid-reply is queued for the next turn, not dropped.

{/* CAPTURE-PENDING: trust-agent-composer.png — the composer with mode toggle, model selector, attach chip, and @/slash pickers */}

### Attach files

The **Attach files** (+) chip accepts document and text files — PDF, Markdown, plain text, CSV, JSON, and Word (`.docx`). Images and other media are deliberately not supported. Attached files are handed to the agent with the run so it can actually read them, and they persist in the side rail's **Attachments** section with an in-place preview. Attach is disabled until the conversation has a first message — the chip's tooltip says **Send a message first to attach files**.

### Ask, Plan, Act

The mode toggle picks how the agent works. The labels are human — **Ask**, **Plan**, **Act** — never raw wire tokens:

| Mode | What it does |
|---|---|
| **Ask** | Clarifies first and pauses for approval before each action — the most cautious mode. |
| **Plan** | Researches and proposes a task list for you to confirm. No changes happen yet. |
| **Act** | Plans and executes end-to-end without pausing between steps. |

In **Plan** mode the agent refines the *existing* plan instead of throwing it away and starting over, and a typed confirmation is what triggers the transition from planning to execution — nothing changes until you confirm.
{/* ACCURACY-AUDIT-PENDING: Plan refines the existing plan and a typed confirmation moves planning→execution — verify by driving each mode on the release tag */}

### Clarify-first

Underneath the modes, the runtime enforces two gates. On an open-ended request the agent's first move is to **ask a clarifying question** — no preamble before the question, just the question. And planning stops for your confirmation before any write executes. The agent would rather ask than guess.
{/* ACCURACY-AUDIT-PENDING: an ambiguous request surfaces a clarifying-question card as the first action (no preamble), and plans stop for confirmation before any write — verify by sending an ambiguous request on the release tag */}

### Next-step chips

After a reply, the agent can offer up to three **next-step chips** — one-tap follow-ups in plain English. Clicking a chip pre-fills the composer with a ready-to-send prompt; it never sends on its own. You always get the last look before anything runs.

### The model selector

The inline model selector lets you pick the model for the session as a price-performance control. Its default is the **quality** model. It deliberately shows you **no cost or latency figure** — those stay internal. You choose how much horsepower a conversation gets; you don't get a billing readout in the middle of your work.

### Skills

Skills are first-class. A skills registry backs the **/** palette, and a skills editor modal lets you create, edit, and delete skills — with the slash palette live-refreshing on every change. When you edit a skill, the editor pre-fills with its existing content. An **@**-mention is pre-loaded and grounded: mention a known entity and it's handed to the agent directly, so the agent skips a redundant search to go find it.
{/* ACCURACY-AUDIT-PENDING: the skills editor modal does create/edit/delete with a live-refreshing / palette and pre-fills on edit; an @-mentioned entity is passed to the agent without a redundant search — verify by driving the skills editor and an @-mention on the release tag */}

The agent also ships with built-in skills, including three that power the guided first hour below: one for connecting an agent, one for observe-only exploration, and one for drafting security coverage.

## The guided first hour

On a brand-new project, the Trust Agent doesn't wait for you to know what to ask. It teaches a guided cold-start arc — inside one conversation — from nothing to a first scored result:

1. **Connect** — it helps you connect the agent you want to test (a Claude-managed agent or Salesforce Agentforce).
2. **Explore** — it runs an **observe-only exploration**: a conversation played against your connected agent purely to learn how it behaves. No scoring — just watching.
3. **Secure** — it proactively proposes adversarial security coverage, drawing on scenarios aligned to the OWASP Agentic AI Top 10, rather than waiting for you to think of attacks.
4. **First evaluation** — it drafts behavior scenarios alongside the security ones and runs your first scored evaluation against the connected agent.

The Welcome page is setup-aware to match. A brand-new project (no sessions or data yet) gets four cold-start starter pills — **Connect agent**, **Explore behaviour**, **Create scenarios**, **Run evaluation** — that steer into this arc; a warmed-up project gets a curated set instead (**Curate data**, **Annotate data**, **Evaluate agent**, **Analyze failures**). Picking a pill expands its example prompts, and picking a prompt pre-fills the composer — the same never-auto-send contract as the next-step chips.

When the agent generates test coverage, it creates flat, persona-and-goal [scenarios](/concepts/scenarios) — no transcripts yet — and simulation happens later, at evaluation time, when fresh conversations are played out per persona against the pinned agent build. Generated scenarios land directly in your project's one flat Scenarios list — there is no intermediate collection to manage.

## How the Trust Agent runs

You don't need to know the plumbing to use the agent, but a little context helps. The Trust Agent runs on one unconditional streaming runtime: the front end consumes a streaming event protocol, and the back end emits those events from the agent loop, with the model streaming over an OpenAI-compatible path. The response markdown is pre-rendered to HTML server-side before it reaches you. This is the single runtime on every project — there's no second implementation behind a flag.

This is the seam where latency lives, and it's worth being precise about it: any wait you feel is the **upstream model's thinking time**, surfaced honestly by the Thinking timer and a working indicator. It is not a runtime timer, not an SLA, and not a bug. How long it takes depends on the model you selected and the infrastructure it runs on, which is exactly why the agent shows a live timer instead of promising a number.

<Note>
  The internal routes and runtime details behind the Trust Agent are implementation, not a public API. There is no agent-runtime endpoint to call directly — you drive the agent through the app.
</Note>

## Security and scope

Write execution is **tenant-scoped and server-anchored**. An approved write executes only within your own organization and project; the organization's tool-permission allowlist is enforced at agent turn time; and execution is bound to the specific approval you gave, not trusted from the client. The agent cannot act on another organization's data. The platform side of that boundary lives in [Multi-tenancy](/concepts/multi-tenancy).
{/* ACCURACY-AUDIT-PENDING: approved writes execute only within the user's org/project, the permission allowlist is enforced at turn time, and execution is bound to the approval (not client-trusted) — verify behaviorally on the release tag */}

The Trust Agent surface is designed to meet WCAG AA contrast. The interrupt cards move focus into the card when they appear, restore it on dismiss, and announce interrupt events to assistive technology — so approvals are reachable by keyboard and screen reader, not just by mouse.
{/* ACCURACY-AUDIT-PENDING: interrupt cards move focus in and restore it on dismiss and announce to AT; surface meets WCAG AA contrast — verify focus behavior by triggering an interrupt card on the release tag */}
