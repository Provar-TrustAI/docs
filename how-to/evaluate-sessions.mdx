An [evaluation](/concepts/evaluations) doesn't have to simulate anything. When the conversations already exist — synced from your runtime, saved from the [Playground](/concepts/playground), or produced by earlier scenario runs — you can grade them exactly as they happened: the judges read the captured transcript, no agent is invoked, and the session is never modified. This page is the task recipe for that; the model behind it lives on [Evaluations](/concepts/evaluations) and [Sessions](/concepts/sessions).

You'll do the job two ways:

- **Bulk** — select sessions and click **Evaluate**, so one named run grades the whole batch
- **One-off** — click **Evaluate** in a single session's panel, so the verdict lands on the session with no run created

<Note>
  Session grading is **judge-only**. The launcher's evaluator picker offers [LLM Judge evaluators](/concepts/evaluators) exclusively — a captured conversation has no authored expected output to structurally check. Sessions that came from a scenario also run that scenario's automatic routing and tool checks, per the launcher's own helper text.
</Note>

## Sessions or scenarios — which to evaluate

Both are launcher sources; they answer different questions.

| Evaluate **sessions** when… | Run **scenarios** when… |
|---|---|
| You want to grade what *actually happened* — yesterday's production traffic, a batch of synced conversations, a Playground save. | You want to test what *would happen* — simulate fresh conversations against an [Agent Version](/concepts/agent-versions) before it ships. |
| No agent is invoked; the judges read stored transcripts. | The agent is driven live, one conversation per attached persona. |
| You pick the judges in the launcher. | Evaluators are inherited from each scenario's expected output. |
| Re-grading is free — evaluate the same sessions again with a different judge, no re-run needed. | Each run produces new sessions you can later evaluate or annotate. |

If you're chasing a regression between agent builds, use [scenarios](/concepts/scenarios). If you're asking "how is my agent really doing?", evaluate the sessions.

## Prerequisites

- A **Trust AI account** with access to a [Project](/concepts/projects) that has sessions — synced, generated, or saved from the Playground
- At least one **LLM Judge evaluator** in the project. If your Evaluators list is empty, **Add recommended evaluators** on the [Evaluators](/concepts/evaluators) surface gives you a curated starting roster in one step.

## Evaluate sessions in bulk

<Steps>
  <Step title="Select the sessions to grade">
    Open **Sessions** in the project nav and select the conversations you want graded — tick their checkboxes, and shift-click a second checkbox to select the whole range between. A bulk-actions bar appears with **Evaluate** as its primary action, alongside **Tag** and **Download**.

    You can also skip selecting entirely: the **Evaluate** button in the Sessions page header evaluates the sessions in your current view, so a filtered [saved view](/concepts/saved-views) — say, last week's external sessions — becomes the run's target in one click.

    **You should see:** the bulk-actions bar rise when sessions are selected, with **Evaluate** as the primary button.
  </Step>

  <Step title="Pick judges in the launcher">
    Clicking **Evaluate** opens the Evaluation Launcher in sessions mode, titled **Evaluate sessions**. Three things to check before you commit:

    - **Name** — pre-filled from your selection, judge count, and date; edit it if you want the run recognizable later.
    - **Evaluators** — a searchable multi-select of your project's LLM Judges, pre-filled with up to three. Add or remove until the panel lists exactly the judges you want applied to every session.
    - **This will run** — the fan-out preview: how many sessions, how many evaluators, and the total evaluator runs that multiplies out to. It also confirms no conversations are simulated.

    There is no agent picker — nothing is invoked, so there's nothing to pick.

    **You should see:** the **Evaluate sessions** modal with a pre-filled name, a judge multi-select, and a fan-out preview that matches your selection.
  </Step>

  <Step title="Launch and watch the run fill in">
    Click **Run evaluation**. You land on the new run's detail page, and results fill in as the judges finish each session. The run also appears in the **Evaluations** list under the name you gave it.

    **You should see:** a run detail page whose results resolve one by one, each with a three-state verdict — **Pass**, **Fail**, or **Not evaluated**.
  </Step>
</Steps>

## Evaluate one session

For the quick check while you're already reading a conversation, you don't need a named run at all.

<Steps>
  <Step title="Open the session and click Evaluate">
    Click any session on the Sessions list to open its fly-in panel. The header's primary action is **Evaluate** — click it and the launcher opens titled **Evaluate session**, with the judge picker but no name field: a one-off is deliberately anonymous.

    **You should see:** the fly-in header's **Evaluate** button open a launcher scoped to just this session.
  </Step>

  <Step title="Confirm and read the verdict in place">
    Pick your judges and click **Evaluate session**. No run is created and you stay exactly where you are — the verdict lands on the session itself once grading completes: the fly-in header's verdict pill reflects it, and the **Evaluations** tab records the grade in the session's history.

    **You should see:** the modal close, the Evaluations list stay unchanged, and the session's own panel carry the new verdict.
  </Step>
</Steps>

Bulk runs are for measurement; the one-off is for "does this one pass?" without ceremony.

## Read the results

### On the run

A finished bulk run leads with two headline numbers, and they answer different questions:

- **Pass rate** grades the *agent* — of the sessions that count, how many passed.
- **Evaluator accuracy** grades the *judges* — how often their combined verdict agreed with a human's verdict on the same sessions.

Evaluator accuracy only resolves when humans have recorded verdicts: it compares the judges against the pass/fail **Verdict** [annotations](/concepts/annotations) people set on those sessions, and sessions without a human verdict are excluded. If nobody has verdicted anything, the headline simply doesn't render — record human verdicts (the Annotate sessions guide covers this) and accuracy starts resolving. A human verdict grades the judges; it never overrides the automated one.

Open any result in the run and use **Open session** to jump to the exact conversation it was graded against — the Sessions list opens with that session's panel already up. And on any finished run, **Re-run** reopens the launcher pre-seeded with the same sessions and the same judges, so re-grading after a judge edit is one click plus confirm.

### On the session

Every graded session carries its results with it:

- The fly-in header shows the **automated verdict** pill — **Pass**, **Fail**, or **Not evaluated** — derived from automated evaluations only.
- The fly-in's **Evaluations** tab lists every run that graded this session, newest first, each expandable into its individual check results.

*Ask the [Trust Agent](/concepts/trust-agent):* "Analyze the failures in that run." On a completed run it open-codes each failing result and groups the notes into ranked failure themes — a fast first pass before you read transcripts yourself.

## Common stumbles

<AccordionGroup>
  <Accordion title="The evaluator picker only offers LLM Judges">
    By design. A captured session has no authored expected output, so structural checks have nothing to assert against — grading a transcript is judge work. Sessions that link back to a scenario additionally run that scenario's automatic routing and tool checks.
  </Accordion>

  <Accordion title="Evaluator accuracy is missing from my run">
    The headline self-hides when there's nothing to measure against: accuracy compares the judges to human **Verdict** annotations, and none of the graded sessions carry one yet. Set pass/fail verdicts on some of those sessions and re-run, and the number resolves.
  </Accordion>

  <Accordion title="My one-off evaluation isn't in the Evaluations list">
    Also by design. A one-off mints no run — the verdict is homed on the session. Find it on the session itself: the verdict pill in its fly-in header and the graded checks in its **Evaluations** tab.
  </Accordion>

  <Accordion title="The launcher won't run — it says I need a judge">
    A sessions launch requires at least one selected LLM Judge; the launcher renders the validation message inline rather than failing silently. If the picker itself is empty, your project has no LLM Judge evaluators yet — create one, or adopt the recommended set from the Evaluators surface.
  </Accordion>

  <Accordion title="Will evaluating change my sessions?">
    No. Grading is read-only: the judges read the stored transcript, the session's conversation is never edited, and there is no delete-session action for a run to trigger. What accrues on the session is its evaluation history and verdict — evidence, not mutation.
  </Accordion>
</AccordionGroup>

Developers who want the underlying launch endpoint and result shapes can find them in the [API Reference](/api-reference).
