An Evaluator is a reusable judgment function. You define it once in a [Project](/concepts/projects), and it gets reused across many [Evaluations](/concepts/evaluations) — applied to whatever [Dataset](/concepts/datasets) you point it at, against whatever [Agent Version](/concepts/agent-versions) you're testing. The same Evaluator scoring the same Dataset across two Agent Versions is **the apples-to-apples comparison** Trust AI is built around.

## The two families

Trust AI ships two kinds of Evaluators. Pick based on what you're judging:

<CardGroup cols={2}>
  <Card title="LLM Judge" icon="brain">
    A plain-English rubric sent to a strong LLM (Claude, GPT-4) at evaluation time. Good for **qualitative judgment** — politeness, completeness, helpfulness, accuracy of unstructured text.
  </Card>
  <Card title="Structured Match" icon="check-square">
    A deterministic check on agent output metadata. Good for **structured judgment** — did the agent call the right tool, return the right intent code, hit the right schema shape.
  </Card>
</CardGroup>

**When you don't know which:** start with LLM Judge if the judgment is something a careful human would describe in words. Switch to Structured Match if the judgment is something a careful human would write as a regex or a JSON path assertion.

## Anatomy of an LLM Judge

An LLM Judge has four parts:

1. **System prompt** — describes the role ("you are a customer-support quality reviewer"). Sets the tone of judgment.
2. **Criteria** — what specifically you're judging. Phrased as a rubric, not as "is this good?". Example: *"Did the agent (a) acknowledge the user's specific order ID, (b) provide the correct order status, (c) explain what the status means, in that order?"*
3. **Scoring scale** — pass/fail or 1–5 scale (the two score types available in the app). Pass/fail is the easiest to calibrate; 1–5 is more nuanced but harder to get reviewer agreement on. A freeform-with-extraction option also exists, though it's less common.
4. **Output format** — JSON with a structured score field, so the result is machine-readable.

The create/edit flow in the app walks you through these four parts — you pick the evaluator kind (LLM Judge or Structured Match), choose a score type (pass/fail or 1–5 scale), and author the criteria in a dedicated editor. The rubric gets stored as a template that re-runs against every Dataset row at Evaluation time.

For how to author rubrics that actually work, see [Write an effective LLM-judge prompt](/how-to/write-llm-judge-prompt).

## Anatomy of a Structured Match

A Structured Match assertion has three parts:

1. **Path** — a JSONPath or dotted-key reference into the agent's structured output. Example: `output.intent`, `tool_calls[0].name`, `metadata.confidence`.
2. **Operator** — equals, not-equals, contains, regex, exists, schema-validates.
3. **Expected** — the value or pattern the path should resolve to.

Structured Match is exact-match by default — useful when the agent emits structured metadata you can pin to (intent classification, skill activation, tool call shape). It's the right choice when "good" is defined precisely and exhaustively. It's the wrong choice when "good" is judgment-laden.

## Calibration — the loop that makes LLM Judges trustworthy

LLM Judges are useful only if their scores agree with informed humans. Calibration is the loop:

1. **Write a draft rubric** (the first pass — often too vague or too strict)
2. **Score it against 10 annotated rows** from a Dataset (rows that already have human Annotations)
3. **Compare** Evaluator pass/fail vs Annotation pass/fail
4. **Refine** the rubric where they disagree — usually by tightening criteria
5. **Repeat** until agreement is high (90%+ for binary scales)

A well-calibrated LLM Judge agrees with a thoughtful human reviewer most of the time. An uncalibrated one is theater — it produces scores that look authoritative but don't track reality.

Trust AI's Evaluator builder shows the calibration agreement rate live as you edit the rubric, which is why getting a usable Evaluator is mostly an iterative authoring task, not a "spec it once and forget" task.

## Reusing Evaluators across Datasets, Agent Versions, time

A defined Evaluator is reusable; that's the whole point. You point it at:

- **Different Datasets** — apply the same judgment criteria to different test beds
- **Different Agent Versions** — same Dataset, same criteria, different agent under test = **regression signal**
- **Different time horizons** — re-running the same Evaluator on the same Dataset two months later tells you about drift

Evaluators are versioned (lightly) — major rubric edits create a new revision. This matters mostly for audit ("which version of the rubric scored this Evaluation?"); minor wording tweaks don't bump.

## Why we explicitly reject other framings

Trust AI's Evaluator vocabulary deliberately differs from some adjacent ecosystems:

- **Not "metric."** Metrics are continuous scalars (latency, token count). Evaluators produce judgments — pass/fail or scored — which are categorical or rubric-bound. The product distinguishes these.
- **Not "test."** Tests pass or fail without nuance. Evaluators carry reasoning — when an LLM Judge fails a row, the reasoning is part of the result, which is how you debug agent regressions.
- **Not "guardrail."** Guardrails are runtime controls (block this output before it reaches a customer). Evaluators are post-hoc judgment. Different system, different stage.

The product DESIGN.md (in `trust-ai-app/docs/`) explicitly rejects `Gauge` and `Ruler` icons for Evaluators in favor of `Wand2` — the rationale is the same: Evaluators are judgments, not measurements.

## Working with Evaluators

Everything you'd want to do with an Evaluator is available through the Trust Agent and the app:

- **Define** an Evaluator (LLM Judge or Structured Match) — ask the Trust Agent to create one, or use the create flow in the Evaluators surface.
- **List** the Evaluators in a Project, or open one to review its rubric or assertion definition, from the Evaluator list view.
- **Calibrate** an LLM Judge against your annotated rows — the builder runs this as you edit and shows the agreement rate live.
- **Edit** a rubric — major edits create a new revision (see [Reusing Evaluators](#reusing-evaluators-across-datasets-agent-versions-time) above).
- **Check references** before archiving or making rubric-breaking edits — ask the Trust Agent, or read the references shown in the side panel.

Developers who want the underlying endpoints, request/response shapes, and version notes (such as the `v2026.05.19` body-shape rename and the reference-check addition) can find them in the auto-generated [API Reference](/api-reference).
