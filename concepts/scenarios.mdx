A **scenario** is one test case worth scoring your agent on: a name, the goal a simulated user is trying to accomplish, the personas who pursue it, and an authored **expected output** — the grading contract that says what a passing conversation looks like. Every project has one flat **Scenarios** list, and each scenario carries a stable, reader-visible id like `SCN-000123`. You define a scenario cheaply, run it to simulate real conversations, and learn its expected output from a session you like — that loop is the heart of evaluating an agent in Trust AI.

<Note>
  **Formerly Datasets.** Earlier releases organized test cases as rows inside collections. As of v2026.06.30.1 those collections are gone: there is one flat Scenarios list per project, each scenario is a first-class entity, and the conversations it produces are real [Sessions](/concepts/sessions) linked back to it. The surface lives at `/projects/:id/scenarios`; the old `/projects/:id/datasets` address redirects there, so stale bookmarks still land in the right place.
</Note>

<Note>
  **Scenarios is not the Trust Agent, and not an Agent Version.** All three use the word "agent" nearby, so it is worth pinning them apart:

  - **Scenarios** (this page) are the *test cases* you score an agent against.
  - The **[Trust Agent](/concepts/trust-agent)** is your in-app *assistant* — the thing that helps you create scenarios, draft evaluators, and run evaluations.
  - An **[Agent Version](/concepts/agent-versions)** is the *agent under test* — an immutable snapshot of the customer's agent whose behavior you are judging.

  You score an Agent Version against your scenarios, and the Trust Agent can help you do it. None of the three is the same thing.
</Note>

## What's in a scenario

- **A name and a stable id** — the id renders as `SCN-000123` everywhere, so a scenario keeps the same handle in results, links, and conversations with teammates.
- **A user goal** — what the simulated user is trying to accomplish ("Get a refund for a duplicate charge"). The goal drives the simulation; a scenario with a goal and no persona still simulates.
- **Personas** (optional) — reusable, project-scoped actors that role-play the user. Attach more than one and a run produces one session per persona (see [Personas](#personas)).
- **Expected output** — the authored grading contract: expected behavior, expected actions, and expected routes (see [Expected output](#expected-output-the-grading-contract)).
- **Tags** — drawn from one shared project vocabulary, so the same tag autocompletes on Scenarios and Sessions alike.
- **Linked sessions** — the real conversations this scenario has produced, listed on the scenario itself.

## The Scenarios list

Open **Scenarios** in the left nav to get the flat table: an **ID** column with the `SCN` handle, the scenario name, the **User Goal**, **Personas** as removable pills, **Expected routes** and **Expected actions** chips, attached **Evaluators**, and a **Sessions** count. Name, goal, personas, and evaluators are editable inline — click the cell, change it, done.

{/* CAPTURE-PENDING: scenarios-list.png — the flat Scenarios table at /projects/:id/scenarios with SCN ids, persona pills, and Expected routes chips */}

The page ships with two built-in [saved views](/concepts/saved-views): **All scenarios** and **Missing expected output** — the second is your worklist of scenarios that can simulate but don't yet say what "good" means. The header carries the page's four actions: **Manage personas**, **Add recommended scenarios**, a **Run** split-button, and **New scenario**.

During a bulk run you stay on the list: each scenario shows a live in-flight badge that settles as it finishes. Runs ride out model-capacity throttling with automatic retry, and a capacity hiccup is excluded from quality metrics — it never masquerades as a drop in agent quality.

## The scenario fly-in

Click a scenario to open a two-level fly-in over the list. The first level is the scenario itself, with workflow-ordered tabs:

- **Personas & Goals** — edit who pursues the goal and what the goal is.
- **Expected Output** — the grading contract editor (next section).
- **Sessions** — every conversation this scenario has produced, with a live count in the tab label. Click a session to stack a second-level panel over the first with that session's transcript, annotations, and evaluations — you drill in without ever losing your place in the list.

The panel's **More** menu carries **Run & evaluate**, **Run only**, and **Delete scenario**.

## Expected output — the grading contract

The keystone of a scenario is its **Expected Output** tab: the place you author what a passing conversation looks like. The editor has three sections:

- **Expected behavior** — LLM-judge [evaluator](/concepts/evaluators) cards that grade qualities like tone, scope, or safety. The picker surfaces the evaluators best suited to this scenario first, under **Recommended for this scenario**, ahead of the full catalog.
- **Expected actions** — the tools the agent must (or must not) use, with per-action parameter checks. Add one with **Add action**.
- **Expected routes** — where the conversation should be routed. Add one with **Add route**.

You don't type action and route names by hand. The options come from **vocabulary the product learned from your real sessions** — the actions and routes your agent has actually used — so the contract is grounded in observed behavior rather than guesses.

This is why the recommended order is to define the scenario cheaply first and author expected output *after* you have sessions: run the scenario, read the conversations, find one that went the way you want, and write the contract from what you see. The **Missing expected output** saved view keeps track of the scenarios still waiting for that step.

## Running a scenario

Running a scenario simulates one session per attached persona — each persona genuinely pursues the goal against the agent under test. There are two run modes, both available from the list's **Run** split-button, the fly-in's **More** menu, and the empty Sessions tab:

- **Run & evaluate** — simulate and grade. Creates an evaluation whose results land in [Evaluations](/concepts/evaluations), scored per persona with a dual headline: how often the agent passed, and how often the automated judges agreed with a human verdict.
- **Run only** — simulate without grading. You get the sessions to read, and no run appears in the Evaluations list. This is the cheap way to produce the raw material for expected output.

The split-button's primary **Run** action is the default graded run; the chevron offers the explicit choice.

Every conversation a run produces is a **real Session**, linked back to its scenario. It appears on the scenario's Sessions tab and on the [Sessions](/concepts/sessions) surface, carrying a **Source** chip that reads **Scenario** and a **Simulated from** provenance card pointing back at the scenario that produced it. Nothing is throwaway: you can read, annotate, and re-grade these sessions like any other.

### The define → run → learn loop

Putting the pieces together, the opinionated workflow is:

<Steps>
  <Step title="Define cheap">
    Create the scenario with just a name, a goal, and personas. Skip expected output — the create modal tells you so: you'll set it later, usually from a session you like.
  </Step>
  <Step title="Run">
    **Run only** to simulate one session per persona without grading.
  </Step>
  <Step title="Learn from a good session">
    Read the sessions on the scenario's Sessions tab. When one goes the way you want, open **Expected Output** and author the contract from the actions and routes you just watched.
  </Step>
  <Step title="Run &amp; evaluate">
    From now on, graded runs score the agent against the contract — and every [Agent Version](/concepts/agent-versions) can be held to the same standard.
  </Step>
</Steps>

For the step-by-step recipe, see [Simulate with scenarios](/how-to/simulate-with-scenarios).

## Where scenarios come from

- **New scenario** — the header button opens a short form: name, goal, personas. Expected output is deliberately deferred.
- **Save as a scenario** from the [Playground](/concepts/playground) — an exploratory conversation you liked becomes a scenario with that conversation attached as its first linked session.
- **Add recommended scenarios** — a starter catalog of ten safety scenarios mapped to the OWASP Agentic AI Top 10, adoptable in one move. Each comes paired with the recommended evaluators that score it well, surfaced as guidance in the Expected Output picker.
- **Ask the [Trust Agent](/concepts/trust-agent)** — describe the behavior you want covered and let the assistant draft scenarios for you; it pauses for your approval before it writes anything.

Note that **Source** on a session is a different idea from where a *scenario* came from: a session's Source chip is exactly one of **Scenario**, **Playground**, or **External**, and records how that conversation entered the system.

## Personas

A **persona** is a reusable, project-scoped actor that role-plays the user — a character with the traits and constraints that shape how it pursues a goal. Personas are shared across the project, so the same actor can drive many scenarios, and they always render by name.

Open **Manage personas** in the Scenarios page header to list, create, edit, and delete them in place. Personas live at the project level; the project's default persona is protected and can't be deleted.

Attach several personas to one scenario to test the same goal from different angles — an impatient customer, a confused first-timer, an adversarial probe. Each run scores every persona on its own, so the results tell you *which kind of user* your agent fails, not just whether it fails.

## Deleting a scenario

Delete is safe by design, with three outcomes:

- **Blocked** — a scenario referenced by evaluations can't be deleted; you'll see "Can't delete: used in N evaluations." instead of a confirm.
- **Confirm with disclosure** — if the scenario has produced sessions, the confirm tells you the sessions are **kept** (they are frozen records of what happened; only the link to the scenario is removed) while the scenario's expected-output contract goes with it.
- **Instant** — a scenario with no evaluations and no sessions deletes immediately.

Because sessions are frozen snapshots and evaluations pin what they scored, deleting or editing a scenario never rewrites history: past results stay exactly as they were recorded.

## API surface

Scenario reads and edits go through the `/v1/scenarios` family — list, get, update, delete (with a pre-delete check), the recommended starter catalog, and each scenario's expected output. The legacy `/v1/datasets` family remains on the wire underneath as the storage the re-design was built over, so you may still see `dataset_id` fields in older payloads. Full request and response shapes live in the [API Reference](/api-reference).
