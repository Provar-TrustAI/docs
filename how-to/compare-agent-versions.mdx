This how-to walks through the **regression-tracking workflow** Trust AI is built around: evaluating two [Agent Versions](/concepts/agent-versions) on the same scenarios under the same [Evaluators](/concepts/evaluators), so the only variable between the two runs is the build itself.

One honest framing note before you start: **Trust AI does not have a dedicated run-vs-run comparison view today.** There is no Compare button and no side-by-side diff surface. The comparison is a workflow — two evaluation runs with identical configuration except the Agent Version, read against each other on the run detail pages and the Evaluations list. This page shows you how to do that read quickly and trust what it tells you.

Two nouns to keep straight on this page: an **Agent Version** is a pinned snapshot of *your* agent under test — the thing being graded. The **Trust Agent** is the in-app assistant you delegate the work to. They are different things.

It picks up where [Run your first Evaluation](/tutorials/run-first-evaluation) leaves off — you already have at least one finished [Evaluation](/concepts/evaluations) run and a second Agent Version to test against. If you don't, run that tutorial first.

## When to compare versions

The three most common triggers:

- **Before deploying a new agent build.** Run the candidate against the same scenarios the production version ran against; ship only if the differences are acceptable.
- **After deploying.** A weekly cadence comparing this week's production build against last week's surfaces drift that wasn't visible in the candidate review.
- **Ranking parallel candidates.** Multiple prompt variants competing for production; run each against the same scenarios and Evaluators and read the results on the same axes.

If you find yourself wanting to compare two runs that **don't share their scenarios or don't share their Evaluators**, stop — the difference isn't apples-to-apples and won't tell you what you think it does. Re-run with matching configuration instead.

## The comparison workflow

<Steps>
  <Step title="Run the baseline against the first Agent Version">
    <Note>
    *Ask the Trust Agent:* "Run an evaluation of my order-status scenarios against the current production build."
    </Note>

    The Trust Agent proposes the run — confirming which scenarios, Evaluators, and agent build it will use — and you approve it before anything launches (see [Approve agent writes](/how-to/approve-agent-writes)).

    To do it by hand: from the Project's **Evaluations** list, click **+ New evaluation**. The evaluation launcher opens; choose your scenarios, keep their attached Evaluators, and pick the first build in the **Agent** section — that's the connected agent that simulates each scenario. The launcher previews the fan-out (every persona on every scenario gets its own conversation) before you confirm.

    **You should see:** the run appear on the Evaluations list, with the Agent Version's label shown under the run name.
  </Step>

  <Step title="Re-run the same configuration against the candidate">
    This is the step that makes the comparison trustworthy. Open the finished baseline run and click **Re-run**. The launcher reopens pre-filled with the original run's configuration — the same scenarios, the same Evaluators, and the original agent — all still editable. Change **only the Agent** to the candidate build, then launch.

    <Note>
    *Ask the Trust Agent:* "Run the same evaluation again, but against the new candidate build."
    </Note>

    Because everything except the Agent Version is held constant, any difference between the two runs is attributable to the build — not to a drifted scenario list or a different judge.

    <Warning>
    If your baseline repeated each test case (its headline reads **Pass^K**, for example `Pass^2`), make sure the candidate run repeats the same number of times. A Pass^2 reliability rate and a single-attempt pass rate are different metrics and can't be compared directly.
    </Warning>
  </Step>

  <Step title="Read the two headline pairs">
    <Note>
    *Ask the Trust Agent:* "Summarize the results of my last two evaluation runs — pass rates and what changed."
    </Note>

    The Trust Agent reads both runs and reports the numbers back in plain language; the same numbers lead each run's detail page for you to check yourself. A finished run's header carries a **dual headline**:

    - **Pass rate** grades the agent — how often it passed each case. This is the number you're comparing between versions.
    - **Evaluator accuracy** grades the judges — how often the Evaluators' combined verdict agreed with a human's session verdict. It appears when annotated cases exist to measure against.

    Read them together. If pass rate dropped but evaluator accuracy held steady, the agent likely regressed. If evaluator accuracy moved sharply between two runs on the same scenarios, treat the pass-rate delta with suspicion — the judges, not the agent, may be the story. Calibrate the Evaluator before making a ship decision (see below).

    Two things that keep the pass rates honest across runs:

    - **Environment problems never count against the agent.** A result blocked by a guardrail shows a **Guardrail blocked** state, and infrastructure or harness failures show as an environment problem — excluded from the score, never a red fail. A capacity hiccup on one run can't masquerade as a regression.
    - **Judging is deterministic.** The LLM judges run with no sampling randomness, so the same response gets the same verdict. A case that scored differently across the two runs reflects a different agent response — or a rubric edge the two responses landed on opposite sides of — not judge noise.

    **You should see:** each run detail leading with its pass rate (or Pass^K reliability rate) and, where human verdicts exist, an evaluator-accuracy headline beside it.
  </Step>

  <Step title="Drill into the cases that flipped">
    <Note>
    *Ask the Trust Agent:* "Which scenarios failed on the candidate run but passed on the baseline? Analyze the failures."
    </Note>

    The Trust Agent can pull both runs' results and run a structured failure analysis on the candidate. To read it yourself, open the two run details (two browser tabs work well) and walk the results tables against each other. Each table scores every scenario-persona pair on its own line, led by the persona's name; failing results carry a plain-English **divergence hint** describing how the actual response diverged from the expected one. Click into a result to read the full transcript — labeled **Simulated** or **Replayed** so you know how it was produced — with the Evaluator's score and reasoning, and use **Open session** to jump to the exact session that was graded.

    Reading the flipped cases is the work. Most differences fall into one of three buckets:

    - **Real regression** — the candidate's response is clearly worse. The build change is the problem.
    - **Calibration edge** — both responses are reasonable; they landed on opposite sides of a subjective rubric line. Tighten the rubric, don't blame the agent. The Evaluator's **Calibrate** tab measures how often the judge agrees with your own verdicts — using the exact judge your runs use — and shows you the disagreements first (see [Write an effective LLM-judge prompt](/how-to/write-llm-judge-prompt)).
    - **Stale scenario** — the agent's response is fine but the scenario's premise no longer applies. Update the scenario's goal or expected output.

    **You should see:** the pattern become legible after reading 5–10 flipped cases — regressions cluster by category far more often than they scatter.
  </Step>

  <Step title="Make the call and keep the trend">
    With the flipped cases read, the decision usually fits one of three shapes:

    - **Ship the candidate.** Regressions are absent or all explainable as calibration edges / stale scenarios.
    - **Block and investigate.** Real regressions in a category that matters. Hand the failing results to the team owning the agent, with links to the two runs as the artifact.
    - **Ship with caveats.** Improvements in one category, tolerable regressions in low-priority surface area. Document the caveats where your team records deploy decisions, and watch the next cycle's runs.

    Trust AI keeps the evidence durable either way: every run stays pinned to its Agent Version, so the Evaluations list doubles as your trend line. Each run's line shows the Agent Version label under the run name alongside a sortable **Overall pass rate** and **Evaluator accuracy**, and the page header keeps the completed-run count and average pass rate in view. Scanning down the version labels and their pass rates over time is how drift shows up between deliberate comparisons.

    Note that the decision itself lives with your team: Trust AI doesn't have a promote-to-production action or a notes field on Agent Versions today, so record the rationale in your deploy process, linking the two runs.
  </Step>
</Steps>

## A worked example

A team evaluates an order-status agent: 120 scenarios, one three-part rubric Evaluator, baseline build at an 87% pass rate. They **Re-run** with only the Agent changed to the candidate, which lands at 84%.

Walking the two results tables against each other:

- **5 real regressions**, all in scenarios where the customer gives a partial order ID. The candidate's prompt tightened its confidence threshold, and it now refuses legitimate partial-match requests. **Action:** block; fix the threshold on the partial-match path.
- **2 calibration edges** — both builds' responses were reasonable; the rubric's "confirms the order before answering" clause is ambiguous for one-message conversations. **Action:** tighten the rubric and check it on the Evaluator's Calibrate tab; not a build problem.
- **2 stale scenarios** — order IDs from a retired test environment. **Action:** update those scenarios' expected output.

The call: **block and investigate**. The regressions are localized and fixable; the next candidate reverts the threshold change on the partial-match path and re-runs the same configuration.

This read takes about 15 minutes once you're fluent. Most of it is the flipped-case drilldown; the headline-level read is usually under a minute.

## Comparing programmatically

There is no run-vs-run diff endpoint today. What ships: each run's results are readable individually, and the evaluations API includes a per-Agent-Version aggregate — run counts and average pass rate grouped by version — that a deploy pipeline can read as a coarse cross-version signal. Full request and response shapes live in the [API Reference](/api-reference).
