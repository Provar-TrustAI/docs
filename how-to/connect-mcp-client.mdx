Trust AI runs an [MCP server](/concepts/mcp-integration) that exposes the Trust Agent's tools. This page covers connecting a **standalone external MCP client** — Claude Desktop, Cursor, or a VS Code MCP extension — to it.

<Warning>
  **Direct external-client connection is not yet available.** Today the MCP server is reached through your authenticated Trust AI session, which powers the in-product Trust Agent and Trust AI's in-process runtime. A desktop client (Claude Desktop, Cursor) can't present that session, so it can't connect over the internet yet. The standalone-client path ships with the OAuth authentication work described below — this page updates with concrete setup steps when it lands.
</Warning>

## What works today

The Trust Agent already drives the MCP tool surface for you **inside the product** — you get its read and write capabilities through the Trust AI UI and the Trust Agent itself, with the same approval gate and observability described on the [MCP Integration](/concepts/mcp-integration) concept page. No client configuration is involved.

## What external connection will require

When the standalone-client path ships, it will follow the **standard MCP authorization flow** rather than a hand-pasted token:

1. You add the Trust AI MCP server to your client (Claude Desktop, Cursor, VS Code).
2. On first use, the client opens a **browser login** — you authenticate with your normal Trust AI **WorkOS SSO**, exactly as you sign in to the app.
3. The client receives a short-lived, org-scoped token automatically and stores it in its own config. **Nothing is copied and pasted.**
4. Read tools run immediately; write tools surface the same approval prompt before anything changes.

This is the authentication pattern the [Model Context Protocol authorization spec](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization) defines, with WorkOS as the identity provider. We'll publish the per-client config snippets, the server URL, and the token policy here once they're real.
