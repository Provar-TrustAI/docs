This how-to walks through the **regression-tracking workflow** Trust AI is built around: comparing two [Agent Versions](/concepts/agent-versions) side-by-side on the same [Dataset](/concepts/datasets) under the same [Evaluators](/concepts/evaluators).

It picks up where [Run your first Evaluation](/tutorials/run-first-evaluation) leaves off — you already have at least one finished [Evaluation](/concepts/evaluations) and an alternate Agent Version to compare against. If you don't, run that tutorial first.

The goal is to turn an Evaluation diff into a **ship / don't-ship / investigate** decision, fast.

## When to compare versions

The three most common triggers:

- **Before deploying a new agent prompt.** Run the candidate against the same Dataset the production version ran against; ship only if the diff is acceptable.
- **After deploying.** A weekly cadence comparing this week's production against last week's surfaces drift that wasn't visible in the candidate review.
- **Ranking parallel candidates.** Multiple prompt variants competing for production; the comparison view ranks them on the same axes.

If you find yourself wanting to compare two Evaluations that **don't share a Dataset or don't share an Evaluator**, stop — the diff isn't apples-to-apples and won't tell you what you think it does. Re-run instead.

## The comparison setup

<Steps>
  <Step title="Pick the two Evaluations to compare">
    Both Evaluations must have run **the same Dataset** and **the same Evaluator(s)**. The only variable allowed to differ is the **Agent Version**.

    From the Project home, navigate to **Evaluations** and either:
    - **Open one Evaluation and click "Compare"** — Trust AI surfaces the eligible comparison runs (same Dataset + Evaluators, different Agent Version) in a dropdown
    - **Select two Evaluations** from the list and click **Compare** in the bulk action bar

    **You should see:** the comparison view loads, with both Agent Versions surfaced as colored [VersionPills](/concepts/agent-versions#the-versionpill-what-users-see-in-the-app) at the top.
  </Step>

  <Step title="Read the aggregate diff">
    The top of the comparison view shows the aggregate-level diff:

    ```
    Dataset: Order-status happy path  (n=120)
    Evaluator: Order status — three-part rubric  (binary)

    [v2026-05-11-prod]  →  [v2026-05-19-candidate]
    Pass rate:   87.5%  →  84.2%       Δ −3.3pp
    Failures:        15  →  19          Δ  +4
    Latency p50:  890ms  →  1240ms      Δ +350ms
    ```

    Three things to check before going row-by-row:

    1. **Pass-rate delta.** Is it within the noise floor for this Dataset size? On a Dataset of 120 binary rows, ±2pp is roughly the noise floor at typical Evaluator agreement rates. ±3.3pp is real signal, worth investigating.
    2. **Failure count.** Pass rate hides direction — 4 new failures vs. 4 new passes both move pass rate by 3.3pp in opposite directions.
    3. **Latency.** If the candidate is significantly slower (or faster), that's a separate signal from the Evaluator scores and often gets ignored — don't.

    **You should see:** the three aggregate stats with their deltas color-coded (red for regression, green for improvement, gray for noise-floor).
  </Step>

  <Step title="Drill into the per-row deltas">
    Below the aggregate, the per-row table shows every Dataset row's score under both Agent Versions, sorted by **delta direction**:

    - **Regressions** (passed → failed) at the top — the rows that matter most
    - **Improvements** (failed → passed) next
    - **Unchanged passes** below
    - **Unchanged failures** at the bottom
    - **Flapping rows** (same Evaluator, different score on re-run) flagged with a yellow icon

    For each regression row, click in to see:

    - The Dataset row input
    - **Both** agent responses (production vs candidate) side by side
    - **Both** Evaluator scores + reasoning
    - The Annotation, if any, that traveled with the row from its source Session

    Reading regressions is the work. Most diffs fall into one of three buckets:

    - **Real regression** — the candidate's response is clearly worse. Agent change is the problem.
    - **Calibration noise** — both responses are reasonable; the Evaluator scored them differently on subjective edge cases. Tighten the rubric, don't blame the agent.
    - **Dataset row is now stale** — the agent's response is fine but the row's premise no longer applies (the order ID is from a closed test environment, etc.). Update the row.

    **You should see:** the per-row drilldown showing both responses + scores + Annotation in one view. The pattern becomes legible after looking at 5-10 regressions.
  </Step>

  <Step title="Handle flapping rows">
    A flapping row is one where **the same Evaluator gives the same Agent Version a different score on different runs**. That's a calibration smell, not an agent regression.

    Trust AI flags these with a yellow icon. Common causes:

    - Evaluator temperature > 0 — fix: set to 0 on the judge model
    - Criteria are ambiguous enough that the judge picks a different interpretation per-call — fix: tighten the rubric (see [Write an effective LLM-judge prompt](/how-to/write-llm-judge-prompt))
    - The row sits on a genuine edge of the criteria — fix: decide whether the row belongs in the Dataset at all, or whether the criteria need refinement to handle this kind of edge case

    Flapping rows shouldn't drive a ship/don't-ship decision; they're a signal to fix the calibration, not the agent.
  </Step>

  <Step title="Make the call">
    With the per-row reading done, the decision usually fits one of three shapes:

    - **Ship the candidate.** Regressions are absent or all explainable as calibration noise / stale rows. Mark the candidate as the new production Agent Version; archive or demote the prior one.
    - **Block and investigate.** Real regressions in a category that matters. Hand the failed rows to the team owning the agent with the comparison link as the artifact.
    - **Ship with caveats.** Mixed picture — improvements in one category, regressions in another, and the regressions are in low-priority surface area you can tolerate. Document the caveats inline on the candidate's Agent Version notes, and watch the next cycle's data.

    Whatever the call, **record it on the candidate Agent Version's notes**. The next person to look at this version (or the next time you do the same comparison) will benefit from the rationale.
  </Step>
</Steps>

## A worked example

```
Dataset: Order-status happy path  (n=120)
Evaluator: Order status — three-part rubric

[v2026-05-11-prod]  →  [v2026-05-19-candidate]
Pass rate:   87.5%  →  84.2%       Δ −3.3pp
```

Per-row drilldown reveals:

- **5 real regressions** — all in a sub-category where the user's message contains a partial order ID. The candidate's prompt was tightened on confidence thresholds; it now refuses to answer when it isn't 100% sure, which causes legitimate partial-match cases to fail. **Action:** block on this; loosen the candidate's confidence threshold for partial-match cases.
- **3 flapping rows** — yellow-flagged. Evaluator temperature was 0.3. **Action:** unrelated to this comparison; ticket the calibration fix separately.
- **2 stale rows** — order IDs from a deprecated test environment. **Action:** update or remove these Dataset rows.

The call: **block and investigate**. The 5 real regressions are localized and fixable; ship a v3 candidate with the confidence-threshold change reverted on the partial-match path.

This kind of read takes ~15 minutes once you're fluent. Most of that is the per-row drilldown; the aggregate-level decision is usually obvious in under a minute.

## Programmatic comparison

The comparison view is also available via API for teams that want to gate deploys on regression-rate thresholds:

<CodeGroup>
  ```bash curl
  curl -X POST https://api.provar.dev/v1/evaluations/$EVAL_A/compare/$EVAL_B \
    -H "Authorization: Bearer $TRUST_AI_TOKEN"
  ```

  ```typescript TypeScript
  const comparison = await client.evaluations.compare(evalA.id, evalB.id);

  if (comparison.regressions.length > comparison.improvements.length) {
    console.error("Net regression detected — blocking deploy");
    process.exit(1);
  }
  ```
</CodeGroup>

Common gating policies:

- **Hard fail on any regression in a specific category** — for surface area where any regression is unacceptable
- **Soft warn on Δ pass rate < −2pp** — for general regression monitoring
- **Block if any new failure has high Annotation priority** — for high-risk customer flows

Full request/response shapes in the [API Reference](/api-reference).
